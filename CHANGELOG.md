## [0.19.0] - 2026-07-22 · [GitHub](https://github.com/petrinhu/glintfx/releases/tag/v0.19.0)

### Added / Adicionado

- **EN:** **Physical input state channel + window lifecycle for `App`** (`HOSTIN-1..5`, Onda 2, consumer-driven GusWorld -- the exactly-one production blocker reported for migrating to App mode: no way to read PHYSICAL keyboard/mouse for a game scene without linking GLFW directly). Three new queries -- `is_key_down(Key)`, `is_mouse_button_down(int)` (`[0, 7]`, GLFW's own honest ceiling, NOT the UI route's 3-button contract), `get_cursor_pos(float&, float&)` (framebuffer physical pixels, same space as `UiEvent`/`get_element_box`) -- read a private input-state table fed by the SAME GLFW callbacks that already drive the synthetic `process_event` route; a new edge-detected `set_key_callback(fn(Key, KeyAction, int modifiers))` rides the same tap (`KeyAction::Press`/`Release`/`Repeat`). `enum class Key` gained ~92 APPEND-ONLY entries (`A`-`Z`, `Digit0`-`Digit9`, `F1`-`F12`, both-side modifiers, lock keys, punctuation, `World1`/`World2`, numpad) ending in a `Count` sentinel -- the pre-existing entries keep their exact numeric values, machine-enforced by `static_assert`s. Window lifecycle: `request_close()` (programmatic quit, bypasses the veto below by design), `set_close_request_callback(fn() -> bool)` (veto for a USER-initiated close only -- X button/Alt+F4), and `set_window_focus_callback`/`set_window_iconify_callback` + `is_window_focused`/`is_window_iconified` (prefixed `window_` so they never collide with the pre-existing DOM `set_focus_callback`). Honest limits, documented not hidden: GLFW itself cannot see a mouse's buttons beyond index 7 (a 12-button mouse's extras are invisible upstream); ABNT2's second `/` key (scancode 97) has no GLFW token at all (`ç` arrives as the physical `Semicolon` position instead -- position semantics, not label semantics); the synthetic `process_event` channel deliberately does NOT write the physical state table (a replay harness cannot forge a "held key"). Manual e2e QA (nested-compositor, `xdotool`/XTest injection) confirmed 3 of 5 contract items live -- physical key -> `is_key_down`, the close-veto DENY path end-to-end (real ICCCM `WM_DELETE_WINDOW`), and focus/iconify -- but the ALLOW/`request_close()`-bypass path and the ABNT2 layout check stayed **inconclusive**: XTest keyboard injection proved flaky on this nested-KWin-on-Wayland stack (intermittent delivery, one observed double-toggle suspected to be X11 auto-repeat), an environment limitation, not a product one. Both items' logic is already covered by a deterministic unit test (`glfw_decide_window_close`, `glfw_event_translate_sanity`) that does not depend on X11 injection; closing the visual e2e gap is tracked as `SEED-UINPUT` (`TODO.md`), recommending raw `uinput` injection instead of XTest. See `docs/embed-integration.md` sections 21-22 for the full contract, including the frame-hook's own-GL-renderer hosting note validated against GusWorld's production numbers (`dt_seconds` error p95 = 20.8 µs over 8633 frames) and the `Gamepads`+`App` coexistence note in `docs/gamepad.md`.
- **PT:** **Canal de estado de input físico + ciclo de vida da janela pro `App`** (`HOSTIN-1..5`, Onda 2, consumer-driven GusWorld -- o único bloqueio de produção reportado pra migrar ao App mode: não havia como ler teclado/mouse FÍSICOS pra uma cena de jogo sem linkar o GLFW direto). Três consultas novas -- `is_key_down(Key)`, `is_mouse_button_down(int)` (`[0, 7]`, o próprio teto honesto do GLFW, NÃO o contrato de 3 botões da rota de UI), `get_cursor_pos(float&, float&)` (pixels físicos de framebuffer, mesmo espaço de `UiEvent`/`get_element_box`) -- leem uma tabela privada de estado de input alimentada pelos MESMOS callbacks GLFW que já dirigem a rota sintética `process_event`; um `set_key_callback(fn(Key, KeyAction, int modifiers))` edge-detectado novo pega o mesmo tap (`KeyAction::Press`/`Release`/`Repeat`). O `enum class Key` ganhou ~92 entradas APPEND-ONLY (`A`-`Z`, `Digit0`-`Digit9`, `F1`-`F12`, modificadores dos dois lados, teclas de trava, pontuação, `World1`/`World2`, numpad) terminando num sentinela `Count` -- as entradas pré-existentes mantêm os valores numéricos exatos, reforçado por máquina via `static_assert`s. Ciclo de vida da janela: `request_close()` (saída programática, bypassa o veto abaixo por design), `set_close_request_callback(fn() -> bool)` (veto só pra close INICIADO PELO USUÁRIO -- botão X/Alt+F4), e `set_window_focus_callback`/`set_window_iconify_callback` + `is_window_focused`/`is_window_iconified` (prefixados `window_` pra nunca colidir com o `set_focus_callback` de DOM pré-existente). Limites honestos, documentados e não escondidos: o próprio GLFW não enxerga botões de mouse além do índice 7 (os extras de um mouse de 12 botões são invisíveis upstream); a segunda tecla `/` do ABNT2 (scancode 97) não tem token GLFW nenhum (`ç` chega como a posição física `Semicolon` em vez disso -- semântica de posição, não de rótulo); o canal sintético `process_event` deliberadamente NÃO escreve a tabela de estado física (um harness de replay não pode forjar uma "tecla segurada"). A lógica dos dois itens já tem cobertura de teste unitário determinístico (`glfw_decide_window_close`, `glfw_event_translate_sanity`) que não depende de injeção X11; fechar o gap visual e2e está rastreado como `SEED-UINPUT` (`TODO.md`), recomendando injeção `uinput` bruta em vez de XTest. Ver `docs/embed-integration.md` seções 21-22 pro contrato completo, incluindo a nota de hospedagem de renderer GL próprio no frame-hook, validada contra os números de produção do GusWorld (erro de `dt_seconds` p95 = 20,8 µs em 8633 frames), e a nota de coexistência `Gamepads`+`App` em `docs/gamepad.md`.
- **EN:** **Dedup/throttle of repeated `RmlUi` warnings** (`LOGTHR-1`, Onda 2): a pure `LogDedup::should_print(type, msg)` helper (bounded, LRU-capped table) collapses the "No font face defined" per-element-per-frame flood at glintfx's own `SystemInterface` layer, in BOTH `App` and embed mode -- first occurrence of an identical message always prints, repeats are suppressed, a summary re-emits at counts 10/100/1000/..., and `LT_ERROR`/`LT_ASSERT` are NEVER suppressed regardless of repetition. ## [0.18.1] - 2026-07-22 · [GitHub](https://github.com/petrinhu/glintfx/releases/tag/v0.18.1)

### Fixed / Corrigido

- **EN:** **Camada 0 `iabs32(INT32_MIN)` defensive guard** (`QW-IABS32`, PROG-1 Wave 1 INBOX sweep): `iabs32` (`src/raster.c`) negated its input without guarding the one value whose negation overflows a signed 32-bit int (`INT32_MIN`) -- unreachable today (the only callers, `dx`/`dy` in `flatten_quad`, already operate on `saturate_fx`-limited differences far from that value), so this is defence-in-depth, not a live-bug fix. A one-line guard (`if (v == INT32_MIN) return INT32_MAX;`) closes it; proven by a new hosted-companion probe (`tests/probe_iabs32.c`, `make probe-iabs32`, `-fsanitize=undefined`) -- reverting the guard makes UBSan abort live with "negation of -2147483648 cannot be represented", confirmed manually before this fix landed.
- **PT:** **Guarda defensiva `iabs32(INT32_MIN)` da Camada 0** (`QW-IABS32`, varredura da INBOX da Onda 1 do `PROG-1`): `iabs32` (`src/raster.c`) negava a entrada sem guardar o único valor cuja negação estoura um `int` de 32 bits com sinal (`INT32_MIN`) -- não-alcançável hoje (os únicos chamadores, `dx`/`dy` em `flatten_quad`, já operam sobre diferenças limitadas por `saturate_fx`, longe desse valor), então é defesa em profundidade, não fix de bug vivo. Uma guarda de 1 linha (`if (v == INT32_MIN) return INT32_MAX;`) fecha o gap; provada por uma sonda hosted-companion nova (`tests/probe_iabs32.c`, `make probe-iabs32`, `-fsanitize=undefined`) -- revertendo a guarda, o UBSan aborta ao vivo com "negation of -2147483648 cannot be represented", confirmado manualmente antes deste fix pousar.
- **EN:** **Suppressed the coredump/DrKonqi noise from the EXPECTED abort in `libcore_consumer`** (`QW-NOCORE`, found by the leader monitoring the journal): `tests/libcore_consumer.cpp` (ADR-0009 gate 3b) intentionally `std::free()`s a `glx_malloc` pointer to prove the glibc rejects the misuse loudly via `abort()` -- the abort is the PASSING outcome, not a real crash, but every `make libcore-test` run was generating a live SIGABRT that triggered a full `drkonqi-coredump-launcher` socket storm. Two layers in the forked child, both applied before `fn()`: `setrlimit(RLIMIT_CORE, {0,0})` (necessary but not sufficient alone -- suppresses the core file, but the kernel's `core_pattern` still pipes to systemd-coredump for any uncaught fatal signal regardless of the rlimit) plus `sigaction(SIGABRT, ...)` (a caught signal is never "uncaught" from the kernel's point of view, so `core_pattern` never fires for it; `_exit(128+sig)` preserves the parent-observed contract `WIFEXITED`/`WIFSIGNALED` unchanged). Confirmed clean via `coredumpctl list --since` and `journalctl` (zero new entries for `libcore_consumer` after the fix, versus `ANOM_ABEND` + the drkonqi service pair before it).
- **PT:** **Suprimido o ruído de coredump/DrKonqi do abort ESPERADO no `libcore_consumer`** (`QW-NOCORE`, achado pelo líder monitorando o journal): `tests/libcore_consumer.cpp` (gate 3b da ADR-0009) faz `std::free()` de propósito num ponteiro do `glx_malloc` pra provar que a glibc rejeita o mau-uso alto via `abort()` -- o abort é o resultado ESPERADO (teste passando), não um crash real, mas cada rodada de `make libcore-test` gerava um SIGABRT ao vivo que disparava uma tempestade de conexão inteira do `drkonqi-coredump-launcher`. Duas camadas no filho forkado, ambas aplicadas antes de `fn()`: `setrlimit(RLIMIT_CORE, {0,0})` (necessário mas não suficiente sozinho -- suprime o arquivo de core, mas o `core_pattern` do kernel ainda envia pro systemd-coredump em qualquer sinal fatal não-capturado, independente do rlimit) mais `sigaction(SIGABRT, ...)` (um sinal CAPTURADO nunca é "não-capturado" do ponto de vista do kernel, então o `core_pattern` nunca dispara pra ele; `_exit(128+sig)` preserva o contrato observado pelo pai, `WIFEXITED`/`WIFSIGNALED` inalterados). Confirmado limpo via `coredumpctl list --since` e `journalctl` (zero entrada nova pro `libcore_consumer` pós-fix, contra `ANOM_ABEND` + o par de serviços drkonqi antes dele).

### Documentation / Documentação

- **EN:** **Own font engine's honest 9px@dp1.0 limit, documented** (`QW-FONT9DOC`, `docs/embed-integration.md` §17): after pen-snap/vsnap closed the sharpness gap against FreeType in 14 of 15 small-body cells, the one residual -- 9px at `dp_ratio=1.0`, x-height fidelity `~0.42` versus FreeType's `~0.85` -- is now documented as what a focused spike proved it to be: curve-top overshoot suppression (not rounding), with no cheap fix. Two ways out recorded for a consumer that hits it: body size `>= 10px` at `dp_ratio=1.0`, or a runtime flip to `FontEngine::FreeType`. **COLR bisection doc-comment refined** (`QW-COLRDOC`, `src/sfnt.c`): distinguishes per-record corruption (reordered/duplicated whole records -- can only cause a false MISS, never a cross-match, since the match reads one 6-byte slot) from per-field corruption (one sub-field altered independently of its siblings -- can yield a memory-safe but semantically bogus hit that no structural parser, bisection or linear, can catch from the bytes alone); the prior wording conflated the two guarantees (A4-EMOJI review S3 MINOR finding).
- **PT:** **Limite honesto de 9px@dp1.0 do motor de fonte próprio, documentado** (`QW-FONT9DOC`, `docs/embed-integration.md` §17): depois que o pen-snap/vsnap fechou o gap de nitidez contra o FreeType em 14 das 15 células de corpo pequeno, o resíduo único -- 9px em `dp_ratio=1.0`, fidelidade de x-height `~0.42` contra `~0.85` do FreeType -- agora está documentado como o que um spike focado provou que é: supressão de overshoot de topo curvo (não arredondamento), sem fix barato. Duas saídas registradas pra um consumidor que bater nisso: corpo `>= 10px` em `dp_ratio=1.0`, ou flip em runtime pra `FontEngine::FreeType`. **Doc-comment da bisecção COLR refinado** (`QW-COLRDOC`, `src/sfnt.c`): distingue corrupção por-registro (registros inteiros reordenados/duplicados -- só pode causar um MISS falso, nunca um cruzamento, já que o match lê um único slot de 6 bytes) de corrupção por-campo (um sub-campo alterado independente dos irmãos -- pode dar um hit memoria-seguro mas semanticamente bogus que nenhum parser estrutural, bisecção ou varredura linear, consegue pegar só pelos bytes); a redação anterior misturava as duas garantias (achado MINOR do review S3 do A4-EMOJI).

## [0.18.0] - 2026-07-22 · [GitHub](https://github.com/petrinhu/glintfx/releases/tag/v0.18.0)

### Added / Adicionado

- **EN:** **Loop, fade, and SFX polyphony** (`AUDIO-F2`, framework 2D, consumer-driven GusWorld): `glintfx::Audio` gains four behaviours a real consumer already had in production before migrating from direct miniaudio to this module -- backwards compatible at the source level (existing `play(id)`/`stop(id)` calls keep compiling with their exact original semantics). **Gapless loop:** `play(id, loop = false, fade_in_s = 0.0f)` and the new live toggle `set_looping(id, loop)`, both acting on the sound's PRIMARY instance only. **Fade in/out:** `play(...)`'s new `fade_in_s` and `stop(id, fade_out_s = 0.0f)`'s new `fade_out_s`, both via miniaudio's own fade support -- a fresh `play()` always resets a stale scheduled fade-stop first (a documented miniaudio footgun); while a `stop(id, fade_out_s)` fade is running, `is_playing(id)` deliberately stays `true` (the sound is still audible). **SFX polyphony:** the new `play_oneshot(id)` creates a fire-and-forget overlapping copy of the sound (`ma_sound_init_copy`, e.g. for rapid-fire gunshots); copies never loop (a looping orphan copy could never be reaped -- deliberate footgun removal), are lazily reaped only from inside `play_oneshot()` itself (no timer, no background thread -- the reap cannot race `shutdown()`), and are capped at 32 live copies per `SoundId` with oldest-first voice-stealing once at the cap. **`is_playing(id)`/`voice_count(id)`** (new, both `const`) observe the primary AND every live copy, needed by 3+ of GusWorld's own headless tests. `set_volume`/`stop_all` now also reach every live copy. The RAII/deterministic-shutdown contract is extended, not touched: `shutdown()` tears down every live copy before every primary, before the engine, for the same refcounted-resource reason already documented for primaries. All new duration parameters (`fade_in_s`/`fade_out_s`) are hardened like every duration in this module: rejected (`false`, state untouched) when non-finite or negative, clamped to `[0, 3600]` s otherwise. See `docs/audio.md` for the full contract, the threading contract (now made explicit: the `Audio` API is single-threaded), and the **ODR migration warning** for consumers switching from a direct miniaudio integration (delete your own `MINIAUDIO_IMPLEMENTATION` translation unit -- glintfx already ships one, and two in one binary is always a link/ODR violation). Streaming (`MA_SOUND_FLAG_STREAM`, for long music without full-RAM decode) remains a known limitation, tracked as `AUDIO-STREAM` (INBOX).

- **EN:** `L1.15-FORMEV` -- 5 observable-event callbacks on both `UiLayer` and `App` (10 methods), closing `AUD-PUB-6(c)`: `set_change_callback(fn(id, value))`, `set_submit_callback(fn(id))`, `set_focus_callback(fn(id))`, `set_blur_callback(fn(id))`, `set_hover_callback(fn(id, entered))` -- forms were inert to the host before this (the only observable element activity was click/scroll). glintfx does NOT play audio itself; these are the hooks a host uses to trigger its own sound/UX reaction (the líder's explicit anchor use case, including hover/focus sound hooks). `change`/`submit` are bubble-phase document-root listeners, a mechanical extension of `set_scroll_callback`'s pattern; `value` on `change` is RmlUi's own `"value"` event parameter, forwarded verbatim (already normalised per control kind -- text/checkbox/radio/select/slider); `submit` reports the `<form>` element's own id (the real dispatch target of RmlUi's `Submit` event). `focus`/`blur` required a **CAPTURE-phase** listener (`in_capture_phase=true`) instead of the bubble-phase idiom every other listener in this family uses -- confirmed against the pinned RmlUi 6.3 source that `Focus`/`Blur` do not bubble, but RmlUi's capture phase always descends root->target regardless of the `bubbles` flag. `hover` is implemented with a SINGLE internal listener on `Mouseover` only (no separate `Mouseout` listener), holding a `current_hover_id_` dedup state that survives across raw events and resets to empty on every fresh `load()` -- this is what prevents re-firing while the cursor crosses id-less children inside the same labeled container. **Two empirical findings not anticipated by the design doc, verified by reading the pinned RmlUi source and proven in the new test (see the doc-comments on `set_focus_callback`/`set_blur_callback` in `bootstrap.hpp` for the full writeup):** (1) `Element::Focus()` on an ALREADY-focused element is a true no-op -- `Context::OnFocusChange`/`SendEvents` dedup by ancestor-chain SET DIFFERENCE, so refocusing the exact current focus target dispatches neither `Blur` nor `Focus`; - **EN:** New decorator `decorator: image-tint( url )` plus three standalone, globally-registered, independently-animatable RCSS properties -- `image-tint-color` (color, accepts `var()`), `image-tint-mode` (`none` | `multiply` | `luminance-multiply` | `screen`, default `none`), `image-tint-threshold` (number, luminance cut, default `0.55`). Consumer-driven by GusWorld: one neutral base texture (white runes + navy stone + gold trim), recolored into N domain colors at runtime via RCSS, without shipping N textures. `image-tint-mode: none` (the default) is a true no-op -- identical cost/behavior to a plain `image()` decorator, zero shader compiled. Full design rationale, including why the decorator needed its own name (RmlUi's decorator model does not let one decorator read/compose over another's render output) and the colour-space handling (un-premultiply before the luminance/saturation math, re-premultiply the output; deliberately no gamma/sRGB correction, consistent with the rest of the renderer): [ADR-0010](docs/adr/0010-image-tint-luminance-key.md). Hardening: an unrecognized `image-tint-mode` keyword falls back to `none` (defense-in-depth, RmlUi's own keyword parser already gates it at parse time); `image-tint-threshold` is clamped to `[0, 0.999]` after an `isfinite` guard -- the upper bound is strictly below `1.0` because the shader's `smoothstep(threshold, 1.0, L)` is GLSL undefined behaviour the instant `edge0 >= edge1`; `image-tint-color` needs no extra clamp; a missing/failed texture load is a no-op, same as the native `image()` decorator. **Honest fidelity note:** "preserves saturated texels" is approximate, not an exact no-op -- the `(1 − saturation)` weighting is linear, so a highly-but-not-fully-saturated texel (e.g. `sat≈0.69`) still receives a non-trivial fraction of the tint at the default threshold (`0.55`). Near-total preservation of a specific saturated asset requires tuning `image-tint-threshold` above that asset's own luminance, trading off against how strongly the neutral zone gets tinted -- see the ADR-0010 addendum and the new "How-to: luminance-key tint" section in `docs/effects.md` for the tuning guidance. Consumer-driven pelo GusWorld: uma textura base neutra (runas brancas + pedra navy + trim ouro), recolorida em N cores de domínio em runtime via RCSS, sem shippar N texturas. `image-tint-mode: none` (o default) é um no-op de verdade -- custo/comportamento idêntico a um decorator `image()` simples, zero shader compilado. Implementado como um fragment shader genuinamente novo (`"glintfx-tint"`) injetado no `Gl3RenderInterface` via overrides de wrapper-marcado de `CompileShader`/`RenderShader`/`ReleaseShader` -- alcançado exclusivamente pela superfície pública de API do RmlUi, **zero patch no core vendorizado do RmlUi**. Racional de design completa, incluindo por que o decorator precisou do próprio nome (o modelo de decorator do RmlUi não deixa um decorator ler/compor sobre a saída de render de outro) e o tratamento de espaço de cor (despremultiplicar antes da matemática de luminância/saturação, repremultiplicar a saída; deliberadamente sem correção gama/sRGB, consistente com o resto do renderer): [ADR-0010](docs/adr/0010-image-tint-luminance-key.md). Hardening: uma keyword `image-tint-mode` não reconhecida cai pra `none` (defesa-em-profundidade, o próprio parser keyword do RmlUi já protege no momento do parse); `image-tint-threshold` é clampado a `[0, 0.999]` após uma guarda `isfinite` -- o limite superior é estritamente abaixo de `1.0` porque o `smoothstep(threshold, 1.0, L)` do shader é comportamento indefinido do GLSL no instante em que `edge0 >= edge1`; `image-tint-color` não precisa de clamp extra; falha/ausência de textura é no-op, igual ao decorator `image()` nativo. **Nota honesta de fidelidade:** "preserva texels saturados" é aproximado, não um no-op exato -- a ponderação `(1 − saturação)` é linear, então um texel altamente-mas-não-totalmente-saturado (ex.: `sat≈0.69`) ainda recebe uma fração não-trivial do tingimento no threshold default (`0.55`). Preservação quase-total de um asset saturado específico exige tunar `image-tint-threshold` acima da luminância própria daquele asset, em troca de o quanto a zona neutra é tingida -- ver o adendo do ADR-0010 e a nova seção "How-to: luminance-key tint" em `docs/effects.md` pra orientação de tuning. (`GLINTFX-TINT-1/2/3`)

- **EN:** `load(rml_path)` [Important] -- each call used to overwrite the internal document pointer (`Bootstrap::Impl::doc`) with the newly loaded document without ever closing the previous one, leaking it and leaving it still `Show()`n underneath the new one (a visible "ghost" document, especially noticeable with transparency/glow). Reported by GusWorld: reproduces on every `load()` call on the same `UiLayer`/`App` while navigating between screens of a multi-screen menu (reload-on-change pattern) -- each navigation leaked one more document. Fixed by closing the previous document (`ElementDocument::Close()`, destruction deferred to the next `Context::Update()`, already called once per frame by `Engine::update()`) **only after** the new `LoadDocument()` succeeds -- if the new load fails, the host keeps whatever was already on screen instead of being left with zero documents. Verified by new test `document_reload_leak` (A→B→A navigation and 5x repeated reload on the same `Engine`, both build configs): `Rml::Context::GetNumDocuments()` stays at 1 instead of growing; confirmed the test has teeth by temporarily removing the guard (document count grew to 2, test failed) before restoring it. - **EN:** `DataBinder::bind_number/bind_string/bind_bool/bind_list` [Important, heap-use-after-free] -- found while auditing for siblings of the `load()` bug above (same "reassign an owning handle without releasing/checking the previous one" class, one level worse: memory corruption, not just a leak).   **PT:** `DataBinder::bind_number/bind_string/bind_bool/bind_list` [Important, heap-use-after-free] -- achado ao auditar por gêmeos do bug do `load()` acima (mesma classe "reatribuir um handle possuído sem liberar/checar o anterior", um nível pior: corrupção de memória, não só vazamento). Chamar `bind_*` duas vezes com a **mesma chave** antes do `load()` sobrescrevia o slot do mapa com um novo `std::unique_ptr`, que destrói (libera) a célula anterior **antes** do segundo `Rml::DataModelConstructor::Bind()` rodar; esse segundo `Bind()` então falha silenciosamente (o `DataModel::BindVariable` do RmlUi usa `map::emplace`, que nunca sobrescreve uma chave existente), então o próprio mapa de variáveis do RmlUi fica apontando pra primeira célula já liberada -- qualquer leitura subsequente dessa variável (ex.: uma view `{{chave}}` reavaliando) é um heap-use-after-free. Corrigido com o mesmo guard check-before-mutate que `create_data_model` já usava: uma chave duplicada agora é rejeitada (`return false`) **antes** de tocar o mapa, igualando o resto do hardening de entrada desta API.
