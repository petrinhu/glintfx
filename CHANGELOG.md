## [0.3.1] - 2026-07-04 · [GitHub](https://github.com/petrinhu/glintfx/releases/tag/v0.3.1)

### Added / Adicionado

- **EN:** `decorator: polygon(<sides>, <fill>[, <rotation>])`'s 2nd argument now also accepts `radial-gradient(...)` / `linear-gradient(...)` in place of a solid `<color>` -- consumer-driven by GusWorld's brass hexagon corner nodes needing centre-clear -> edge-dark "screw head" volume instead of a flat fill. **Technique chosen: reuse RmlUi's own real gradient fragment shader** (`Rml::RenderManager::CompileShader("radial-gradient"/"linear-gradient", ...)`, the exact public API and Dictionary key contract `Rml::DecoratorRadialGradient`/`Rml::DecoratorLinearGradient` themselves use), not hand-rolled vertex-colour rings: because the shader colours each fragment from an interpolated tex_coord (an affine function of vertex position), the polygon gets a perfectly circular/linear blend under its triangle-fan mesh, not a faceted approximation, at less code than a correct multi-ring implementation would need. No shader code was written -- the fragment shader lives in RmlUi's already-vendored GL3 backend and is reached exclusively through public headers; `render_gl3.cpp` and RmlUi core are untouched. **Syntax:** the shorthand's 2nd positional item is now parsed with the built-in passthrough `"string"` RCSS parser (not `"color"`), deferring the colour-vs-gradient decision to a new `ParseFill()` in `decorator_polygon.cpp`, itself built entirely on RmlUi's own public parsers looked up by name (`StyleSheetSpecification::GetParser("color"/"color_stop_list"/"angle"/"number_percent")`) -- zero hand-rolled colour/number lexing. Radial accepts an optional `circle [at <x%> <y%>]` centre-position prefix (only `circle`, no `ellipse` -- the polygon's fill shape is always its own inscribed circle); linear requires the angle as a mandatory first argument (no CSS `to <side>` keyword form). In both, the **last** stop lands exactly on the polygon's own inscribed-circle radius (not CSS's farthest-corner default), so the outer ring always lines up with the shape's own edge. Solid-color fill (`polygon(6, #C9A24B)`) is fully unchanged, and the existing `sides` input hardening (`[3, 1024]`, float-cast-overflow-safe) is untouched and still runs first. **Fill hardening:** a malformed fill (invalid colour, invalid/unsupported stop unit, missing linear angle, empty gradient) makes the whole decorator ignored, fail-high, exactly like an out-of-range `sides` -- never a partial or garbage render. Verified by new test `polygon_gradient_sanity` (runs in both build configs, UBSan-clean): radial ramp direction, an explicit stop-percentage landing exactly on its own colour, the linear angle-to-direction mapping, the solid fill byte-identical at two sample points, and all four fill/`sides`-hardening rejection cases. Documented in `docs/effects.md`.
  **PT:** o 2º argumento de `decorator: polygon(<lados>, <preenchimento>[, <rotação>])` agora também aceita `radial-gradient(...)` / `linear-gradient(...)` no lugar de uma `<cor>` sólida -- consumer-driven pelos nós hexagonais de latão dos cantos do GusWorld, que precisavam do volume "cabeça de parafuso" centro-claro -> borda-escura em vez de um preenchimento chapado. **Técnica escolhida: reusar o próprio shader de fragmento de gradiente real do RmlUi** (`Rml::RenderManager::CompileShader("radial-gradient"/"linear-gradient", ...)`, a mesma API pública e o mesmo contrato de chaves do Dictionary que `Rml::DecoratorRadialGradient`/`Rml::DecoratorLinearGradient` usam), não anéis de vertex-color feitos à mão: como o shader colore cada fragmento a partir de um tex_coord interpolado (uma função afim da posição do vértice), o polígono recebe um blend perfeitamente circular/linear sob sua malha triangle-fan, não uma aproximação facetada, com menos código do que uma implementação correta multi-anel exigiria. Nenhum código de shader foi escrito -- o shader de fragmento mora no backend GL3 já vendorizado do RmlUi e é alcançado exclusivamente por headers públicos; `render_gl3.cpp` e o core do RmlUi ficam intocados. **Sintaxe:** o 2º item posicional do shorthand agora é parseado com o parser RCSS passthrough embutido `"string"` (não `"color"`), adiando a decisão cor-vs-gradiente para um novo `ParseFill()` em `decorator_polygon.cpp`, ele mesmo construído inteiramente sobre os parsers públicos do próprio RmlUi buscados por nome (`StyleSheetSpecification::GetParser("color"/"color_stop_list"/"angle"/"number_percent")`) -- zero lexing de cor/número feito à mão. O radial aceita um prefixo opcional de posição-de-centro `circle [at <x%> <y%>]` (só `circle`, sem `ellipse` -- a forma de preenchimento do polígono é sempre o próprio círculo inscrito); o linear exige o ângulo como 1º argumento obrigatório (sem a forma de palavra-chave `to <side>` do CSS). Em ambos, o **último** stop cai exatamente no raio do círculo inscrito do próprio polígono (não no default farthest-corner do CSS), então o anel externo sempre alinha com a borda da própria forma. O preenchimento de cor sólida (`polygon(6, #C9A24B)`) fica totalmente inalterado, e o hardening de entrada já existente de `sides` (`[3, 1024]`, seguro contra float-cast-overflow) fica intocado e continua rodando primeiro. **Hardening de preenchimento:** um preenchimento malformado (cor inválida, unidade de stop inválida/não-suportada, ângulo do linear ausente, gradiente vazio) faz o decorator inteiro ser ignorado, fail-high, exatamente como um `lados` fora do range -- nunca um render parcial ou lixo. Verificado pelo novo teste `polygon_gradient_sanity` (roda nos 2 configs de build, UBSan-limpo): direção da rampa radial, um stop de porcentagem explícito caindo exatamente na própria cor, o mapeamento ângulo-para-direção do linear, o preenchimento sólido byte-idêntico em dois pontos de amostra, e os quatro casos de rejeição de hardening de preenchimento/`sides`. Documentado em `docs/effects.md`.

### Fixed / Corrigido

- **EN:** `load(rml_path)` [Important] -- each call used to overwrite the internal document pointer (`Bootstrap::Impl::doc`) with the newly loaded document without ever closing the previous one, leaking it and leaving it still `Show()`n underneath the new one (a visible "ghost" document, especially noticeable with transparency/glow). Reported by GusWorld: reproduces on every `load()` call on the same `UiLayer`/`App` while navigating between screens of a multi-screen menu (reload-on-change pattern) -- each navigation leaked one more document. Fixed by closing the previous document (`ElementDocument::Close()`, destruction deferred to the next `Context::Update()`, already called once per frame by `Engine::update()`) **only after** the new `LoadDocument()` succeeds -- if the new load fails, the host keeps whatever was already on screen instead of being left with zero documents. Verified by new test `document_reload_leak` (A→B→A navigation and 5x repeated reload on the same `Engine`, both build configs): `Rml::Context::GetNumDocuments()` stays at 1 instead of growing; confirmed the test has teeth by temporarily removing the guard (document count grew to 2, test failed) before restoring it. Clean under `GLINTFX_SANITIZE=ON`.
  **PT:** `load(rml_path)` [Important] -- cada chamada sobrescrevia o ponteiro interno do documento (`Bootstrap::Impl::doc`) com o documento recém-carregado sem nunca fechar o anterior, vazando-o e deixando-o ainda `Show()`ado por baixo do novo (um documento "fantasma" visível, especialmente perceptível com transparência/glow). Reportado pelo GusWorld: reproduz em toda chamada de `load()` na mesma `UiLayer`/`App` ao navegar entre telas de um menu com várias telas (padrão reload-on-change) -- cada navegação vazava mais um documento. Corrigido fechando o documento anterior (`ElementDocument::Close()`, destruição adiada até o próximo `Context::Update()`, já chamado uma vez por frame por `Engine::update()`) **só depois** do novo `LoadDocument()` ter sucesso -- se o novo load falhar, o host mantém o que já estava na tela em vez de ficar com zero documentos. Verificado pelo novo teste `document_reload_leak` (navegação A→B→A e 5x reload repetido no mesmo `Engine`, nos 2 configs de build): `Rml::Context::GetNumDocuments()` fica em 1 em vez de crescer; confirmado que o teste tem dente removendo o guard temporariamente (a contagem de documentos cresceu pra 2, o teste falhou) antes de restaurá-lo. Limpo sob `GLINTFX_SANITIZE=ON`.
- **EN:** `DataBinder::bind_number/bind_string/bind_bool/bind_list` [Important, heap-use-after-free] -- found while auditing for siblings of the `load()` bug above (same "reassign an owning handle without releasing/checking the previous one" class, one level worse: memory corruption, not just a leak). Calling `bind_*` twice with the **same key** before `load()` used to overwrite the map slot with a new `std::unique_ptr`, which destroys (frees) the previous cell **before** the second `Rml::DataModelConstructor::Bind()` call runs; that second `Bind()` then fails silently (RmlUi's `DataModel::BindVariable` uses `map::emplace`, which never overwrites an existing key), so RmlUi's own variable map is left pointing at the now-freed first cell -- any subsequent read of that variable (e.g. a `{{key}}` view re-evaluating) is a heap-use-after-free. Reachable only via a caller bug (binding the same key twice), not the ordinary happy path, hence Important rather than Critical -- but a real memory-safety issue, not merely a leak. Fixed with the same check-before-mutate guard `create_data_model` already used: a duplicate key is now rejected (`return false`) **before** touching the map, matching every other entry-point hardening in this API. Verified by new test `data_model_rebind_uaf`: confirmed the fix has teeth by temporarily removing the guard and rebuilding under `GLINTFX_SANITIZE=ON` -- ASan caught the exact heap-use-after-free live (`Rml::Variant::Set` <- `DataViewText::Update` <- `DataModel::Update` <- `Context::LoadDocument`) before the guard was restored and ASan/UBSan confirmed clean again. Non-duplicate keys are unaffected.
  **PT:** `DataBinder::bind_number/bind_string/bind_bool/bind_list` [Important, heap-use-after-free] -- achado ao auditar por gêmeos do bug do `load()` acima (mesma classe "reatribuir um handle possuído sem liberar/checar o anterior", um nível pior: corrupção de memória, não só vazamento). Chamar `bind_*` duas vezes com a **mesma chave** antes do `load()` sobrescrevia o slot do mapa com um novo `std::unique_ptr`, que destrói (libera) a célula anterior **antes** do segundo `Rml::DataModelConstructor::Bind()` rodar; esse segundo `Bind()` então falha silenciosamente (o `DataModel::BindVariable` do RmlUi usa `map::emplace`, que nunca sobrescreve uma chave existente), então o próprio mapa de variáveis do RmlUi fica apontando pra primeira célula já liberada -- qualquer leitura subsequente dessa variável (ex.: uma view `{{chave}}` reavaliando) é um heap-use-after-free. Alcançável só via erro do chamador (ligar a mesma chave duas vezes), não o caminho feliz comum, por isso Important em vez de Critical -- mas um problema real de segurança de memória, não apenas um vazamento. Corrigido com o mesmo guard check-before-mutate que `create_data_model` já usava: uma chave duplicada agora é rejeitada (`return false`) **antes** de tocar o mapa, igualando o resto do hardening de entrada desta API. Verificado pelo novo teste `data_model_rebind_uaf`: confirmado que o fix tem dente removendo o guard temporariamente e reconstruindo sob `GLINTFX_SANITIZE=ON` -- o ASan capturou o heap-use-after-free exato ao vivo (`Rml::Variant::Set` <- `DataViewText::Update` <- `DataModel::Update` <- `Context::LoadDocument`) antes do guard ser restaurado e o ASan/UBSan confirmarem limpo de novo. Chaves não-duplicadas não são afetadas.

## [0.3.0] - 2026-07-04 · [GitHub](https://github.com/petrinhu/glintfx/releases/tag/v0.3.0)

### Added / Adicionado

- **EN:** `decorator: polygon(<sides>, <color>[, <rotation>])` -- a new custom RCSS decorator that fills an element's box with a solid-color regular N-gon (triangle-fan inscribed in a circle of radius = half the shorter box dimension; `rotation` is optional, in degrees, default orientation points the first vertex straight up, negative/any finite angle allowed). Consumer-driven by GusWorld's hexagonal slider node. No shader, no change to `render_gl3.cpp` or RmlUi core -- pure extension via the public `Rml::Decorator`/`Rml::DecoratorInstancer` interface, modelled directly on `Rml::DecoratorStraightGradient` (new internal files `glintfx/src/decorator_polygon.{hpp,cpp}`, registered by `Bootstrap::init()`). Glow and clip-path use cases need zero new code: `filter: drop-shadow(...)` already follows the polygon's alpha shape, and `mask-image: polygon(...)` already works because any decorator can be a mask source. **Input hardening:** `sides` is validated defensively against the range **`[3, 1024]`** on the raw parsed float *before* any int conversion (RCSS may come from a less-trusted theme/mod/dynamic source). An empty `polygon()`, a below-floor value (`polygon(2)`, negatives), an above-ceiling value (`polygon(100000)` -- a DoS-by-typo guard: ~a billion triangles otherwise), or a non-finite/astronomical value (`polygon(1e30)`, avoiding float-cast-overflow UB) all cause the decorator to be **ignored** (fail-high) with a warning naming the received value and valid range -- never a silent white hexagon. Documented in `docs/effects.md`.
  **PT:** `decorator: polygon(<lados>, <cor>[, <rotação>])` -- um novo decorator RCSS customizado que preenche a caixa de um elemento com um N-ágono regular de cor sólida (triangle-fan inscrito num círculo de raio = metade da dimensão menor da caixa; `rotação` é opcional, em graus, orientação padrão aponta o primeiro vértice reto pra cima, ângulo negativo/qualquer finito permitido). Consumer-driven pelo nó slider hexagonal do GusWorld. Sem shader, sem mudança em `render_gl3.cpp` ou no core do RmlUi -- pura extensão via a interface pública `Rml::Decorator`/`Rml::DecoratorInstancer`, moldada diretamente em `Rml::DecoratorStraightGradient` (novos arquivos internos `glintfx/src/decorator_polygon.{hpp,cpp}`, registrados por `Bootstrap::init()`). Casos de uso de glow e clip-path não precisam de código novo: `filter: drop-shadow(...)` já segue a forma do alpha do polígono, e `mask-image: polygon(...)` já funciona porque qualquer decorator pode ser fonte de máscara. **Hardening de entrada:** `lados` é validado defensivamente contra o range **`[3, 1024]`** no float bruto parseado *antes* de qualquer conversão pra int (RCSS pode vir de tema/mod/fonte dinâmica menos confiável). Um `polygon()` vazio, um valor abaixo do piso (`polygon(2)`, negativos), um valor acima do teto (`polygon(100000)` -- guard de DoS-por-typo: senão ~um bilhão de triângulos), ou um valor não-finito/astronômico (`polygon(1e30)`, evitando UB de float-cast-overflow) fazem o decorator ser **ignorado** (fail-high) com um aviso nomeando o valor recebido e o range válido -- nunca um hexágono branco silencioso. Documentado em `docs/effects.md`.

### Fixed / Corrigido

- **EN:** Input hardening across the public API surface (security-audit findings) -- treat RCSS/parameters as coming from a less-trusted source, not just from a well-behaved caller:
  - **`load(nullptr)` [Critical]** -- previously constructed a `std::string(nullptr)` internally, which is undefined behaviour and crashed the host process via `std::terminate()`. Now returns `false` like an ordinary load failure. The `if (!rml_path) return false;` guard already existed in the sibling entry points (`DataBinder`, `get_element_box`, `set_base_url`) and was simply missing from `load()` itself.
  - **`set_dp_ratio(ratio)` [Important]** -- now rejects `!std::isfinite(ratio) || ratio <= 0` and keeps the previous value instead of applying it. A `NaN` or `0` ratio previously propagated silently into `Rml::Context::SetDensityIndependentPixelRatio`, poisoning every subsequent `dp`-unit layout computation.
  - **`set_viewport(w, h, ...)` [Important]**, both the 2-arg and 5-arg (`UiLayer` letterbox) overloads -- now rejects `w <= 0 || h <= 0` as a no-op, mirroring the guard already present on the `Resize` event path in `process_event()`.
  All three are fail-safe (reject-and-keep-previous-state), never fail-open. Verified by `input_hardening_sanity` (embed, both build configs) and `app_input_hardening_smoke` (GLFW-only), both UBSan-clean under `GLINTFX_SANITIZE=ON`.
  **PT:** Hardening de entrada em toda a superfície pública da API (achados de auditoria de segurança) -- tratando RCSS/parâmetros como vindos de uma fonte menos confiável, não apenas de um chamador bem-comportado:
  - **`load(nullptr)` [Critical]** -- antes construía um `std::string(nullptr)` internamente, comportamento indefinido que derrubava o processo host via `std::terminate()`. Agora retorna `false`, como uma falha de load comum. O guard `if (!rml_path) return false;` já existia nos pontos de entrada irmãos (`DataBinder`, `get_element_box`, `set_base_url`) e faltava só no próprio `load()`.
  - **`set_dp_ratio(ratio)` [Important]** -- agora rejeita `!std::isfinite(ratio) || ratio <= 0` e mantém o valor anterior em vez de aplicá-lo. Um ratio `NaN` ou `0` antes se propagava silenciosamente para `Rml::Context::SetDensityIndependentPixelRatio`, envenenando todo cálculo de layout subsequente com unidade `dp`.
  - **`set_viewport(w, h, ...)` [Important]**, tanto a sobrecarga de 2 args quanto a de 5 args (letterbox do `UiLayer`) -- agora rejeita `w <= 0 || h <= 0` como no-op, replicando o guard já presente no caminho do evento `Resize` em `process_event()`.
  As três são fail-safe (rejeita-e-mantém-estado-anterior), nunca fail-open. Verificado por `input_hardening_sanity` (embed, nos 2 configs de build) e `app_input_hardening_smoke` (GLFW-only), ambos UBSan-limpos sob `GLINTFX_SANITIZE=ON`.
- **EN:** `glintfx::version()` now correctly reports **"0.3.0"**. It had been silently stuck reporting `"0.2.4"` since the v0.2.5 release -- the `project(glintfx ... VERSION ...)` bump in `CMakeLists.txt` was forgotten in that release window, and since `version()` reads the CMake-generated `GLINTFX_VERSION` (fixed to read from CMake, rather than a hardcoded literal, back in v0.2.4 -- see below), the stale build number rode along unnoticed through v0.2.5. Single source of truth (`project(VERSION)`) confirmed via repo-wide grep before this release.
  **PT:** `glintfx::version()` agora reporta corretamente **"0.3.0"**. Estava silenciosamente travado em `"0.2.4"` desde a release v0.2.5 -- o bump de `project(glintfx ... VERSION ...)` no `CMakeLists.txt` foi esquecido naquela janela de release, e como `version()` lê o `GLINTFX_VERSION` gerado pelo CMake (corrigido para ler do CMake, em vez de literal hardcoded, lá na v0.2.4 -- ver abaixo), o número de build defasado seguiu despercebido durante toda a v0.2.5. Fonte única (`project(VERSION)`) confirmada via grep no repo inteiro antes desta release.

## [0.2.5] - 2026-07-02 · [GitHub](https://github.com/petrinhu/glintfx/releases/tag/v0.2.5)

### Added / Adicionado

- **EN:** `set_click_callback(std::function<void(const char* element_id)>)` on both `UiLayer` and `App` -- fires on "click" (bubble phase, listener attached to the document root by `load()`), reporting the id of the clicked element, or the nearest ancestor id if the target itself has none, or `""` if no element in the ancestor chain has an id. No ordering constraint versus `load()` -- can be set before or after. Verified by `click_callback_sanity` and `app_click_callback_smoke`. Resolves `L1.11-CLICKCB`.
  **PT:** `set_click_callback(std::function<void(const char* element_id)>)` em `UiLayer` e `App` -- dispara em "click" (fase bubble, listener anexado à raiz do documento por `load()`), reportando o id do elemento clicado, ou o id do ancestral mais próximo se o próprio alvo não tiver, ou `""` se nenhum elemento na cadeia de ancestrais tiver id. Sem restrição de ordem vs. `load()` -- pode ser setado antes ou depois. Verificado por `click_callback_sanity` e `app_click_callback_smoke`. Resolve `L1.11-CLICKCB`.
- **EN:** `get_element_box(const char* id) -> ElementBox { bool found; float x, y, w, h; }` on both `UiLayer` and `App` -- returns the border-box geometry (`Rml::BoxArea::Border`) of an element by id, in window-space physical pixels, or `found=false` if the id does not exist or no document is loaded yet. New public header `glintfx/element_box.hpp`. Verified by `element_box_sanity` and `app_element_box_smoke`. Resolves `L1.12-ELEMBOX`.
  **PT:** `get_element_box(const char* id) -> ElementBox { bool found; float x, y, w, h; }` em `UiLayer` e `App` -- retorna a geometria border-box (`Rml::BoxArea::Border`) de um elemento por id, em pixels físicos espaço-janela, ou `found=false` se o id não existir ou nenhum documento estiver carregado ainda. Novo header público `glintfx/element_box.hpp`. Verificado por `element_box_sanity` e `app_element_box_smoke`. Resolve `L1.12-ELEMBOX`.
- **EN:** `UiLayer::set_viewport(int x, int y, int w, int h, int target_h)` -- places the composited UI within a sub-region of the host's window (letterbox), resolving the hardcoded-`(0,0)`-origin gap tracked as `GAP-2` since ADR-0008. `(x, y)` are top-down window-space (the same convention as `UiEvent`'s mouse coordinates and `get_element_box()`'s return value); `target_h` is the host's full render-target height, used only to convert internally to OpenGL's bottom-up viewport offset. The legacy `set_viewport(w, h)` (2-arg) is unchanged and equivalent to `(0, 0, w, h, h)`. `UiLayer`-only by design -- `App` always owns its whole window, so a sub-viewport offset is meaningless there (see ADR-0008 "F3 implementation update"). ### Fixed / Corrigido
