# ADR-0009 — Internalization boundary / Fronteira de internalização

- **Status:** Accepted (2026-07-09)
- **Deciders:** petrus (líder), caetano-cto, software-architect
- **Tags:** architecture, internalization, one-way-door
- **Supersedes in part:** [[0006-layered-hybrid-architecture]] (the "the layers don't link" clause — see Addendum 2026-07-09 on that ADR).

## Context / Contexto

**EN:** ADR-0006 set Layer 0 (sovereign C+ASM core) and Layer 1 (glintfx, the C++23 deliverable) as **two separate processes that never link** — "the boundary is the process, not the linker" — because at the time Layer 0 had no allocator with real `free`, no libc-equivalent surface worth consuming, and internalization was a distant, unscoped "trajectory". That has changed: Layer 0's core pipeline (`core-v0.1.0`) is now **complete** — I/O bootstrap, its own TDD harness, a libc-core (`memcpy`/`memset`/`memmove`/`memcmp`, `strlen`/`strcmp`/`strncmp`/`strcpy`/`strncpy`/`strcat`/`strchr`, int↔string conversion), mini-printf, and a bump allocator over `mmap` — and internalization work (`SOV-*`/`TRANS-*`) is now a real, near-term track, not a someday-maybe. We must decide **how a piece of Layer 0 code actually reaches a Layer 1 binary** once reimplemented: as a separate process (ADR-0006's original framing), as a statically linked archive, or as something else — and, orthogonally, **how much of an internalized RmlUi interface (e.g. `FontEngineInterface`, `RenderInterface`) must itself be sovereign C**, given that these interfaces are C++ virtual classes by contract with RmlUi and cannot be written in freestanding C without a C++ ABI shim.

**PT:** O ADR-0006 fixou a Camada 0 (núcleo soberano C+ASM) e a Camada 1 (glintfx, a entrega C++23) como **dois processos separados que nunca se linkam** — "a fronteira é o processo, não o linker" — porque, na época, a Camada 0 não tinha alocador com `free` real, não tinha superfície equivalente-a-libc que valesse a pena consumir, e a internalização era uma "trajetória" distante e sem escopo. Isso mudou: o pipeline central da Camada 0 (`core-v0.1.0`) está agora **completo** — bootstrap de I/O, harness próprio de TDD, uma libc-núcleo (`memcpy`/`memset`/`memmove`/`memcmp`, `strlen`/`strcmp`/`strncmp`/`strcpy`/`strncpy`/`strcat`/`strchr`, conversão int↔string), mini-printf, e um alocador bump sobre `mmap` — e o trabalho de internalização (`SOV-*`/`TRANS-*`) é agora uma trilha real e de curto prazo, não mais um "algum dia". Precisamos decidir **como uma peça de código da Camada 0 de fato chega a um binário da Camada 1** uma vez reimplementada: como processo separado (o enquadramento original do ADR-0006), como archive linkado estaticamente, ou algo diferente — e, ortogonalmente, **quanto de uma interface internalizada do RmlUi (ex.: `FontEngineInterface`, `RenderInterface`) precisa ela mesma ser C soberano**, dado que essas interfaces são classes virtuais C++ por contrato com o RmlUi e não podem ser escritas em C freestanding sem uma camada de ABI C++.

## Decision / Decisão

**EN:** Adopt **Option A — phased ("A-faseada")**: static incremental linking, with a mandatory split pattern between the C++ glue and the C algorithmic core.

- **Static incremental link.** Layer 0 gains a build target, `libcore.a`, whose public symbols carry a distinct namespace prefix (`glx_*`). This is the **destination** of every internalized piece: as each Layer-0 clean-room reimplementation lands, it is exposed via `libcore.a` and consumed **explicitly** by Layer 1 (`#include <core/...>`), linked into the glintfx binary. This is a deliberate, targeted narrowing of ADR-0006's "never link" clause — it does not undo the two-layer split (Layer 0's build system, freestanding constraints and zero-libc discipline are untouched); it only changes how a *finished* Layer-0 artifact reaches a Layer-1 binary.
- **Mandatory pattern: "hosted shell + C meat" ("casca-hosted + carne-C").** Any C++ class that implements an RmlUi interface (`FontEngineInterface`, `RenderInterface`, …) stays **ordinary C++, hosted** (normal `libstdc++`, normal exceptions, normal `operator new`/`delete` — it is glue code, not sovereign territory). Only the **algorithmic meat** inside it — SFNT parsing, outline rasterization, geometry math — is written as **plain C in Layer 0**, exposed via `extern "C"`, and statically linked in. This makes **`SOV-CXXRT`** (a freestanding C++ runtime: `operator new`/`delete`, `__cxa_guard_acquire/release`, `__cxa_atexit`, an exception policy) **optional, not a blocker** for internalization to proceed — sovereignty is spent where it has value (the algorithm), not on reimplementing C++ glue code that RmlUi's contract already forces to be C++.
- **Gates** — nothing here executes ahead of these:
  1. **`REL-TAG core-v0.1.0` is closed.** Already done; this ADR would not have been written otherwise.
  2. **`SOV-ALLOC` (an allocator with a real `free`) is the first prerequisite — a hard blocker.** Layer 0 is bump-only by ADR-0004; UI code allocates and frees per layout pass. No internalized piece that allocates may proceed before `SOV-ALLOC` lands.
  3. **Heap ownership rule, enforced and tested, not just documented:** a pointer allocated by `glx_*` **must never** be passed to glibc's `free()`, and vice-versa. This is the *real* risk of Option A — not symbol collision (the `glx_` prefix handles that trivially), but **silent heap corruption between two allocators that ASan will not catch across the boundary**. Mitigation: opaque handle types or a naming/suffix convention at every ownership-crossing API, plus a dedicated boundary test that exercises cross-allocator misuse deliberately (expected to fail loudly, not corrupt silently).
  4. **Pilot: gl3w.** Smallest piece, a mechanical contract (function-pointer loading against a fixed header list), almost no heap involvement. It proves the clean-room + `libcore.a` + `glx_` process end-to-end at minimum risk before a heap-heavy target (e.g. FreeType glyph rasterization) is attempted.

**Option B (IPC) is formally discarded for the hot path.** GL function pointers and the GL context are **per-process** state — gl3w (which *is* the function-pointer table) and the effects `RenderInterface` (which issues GL calls every frame) are **physically impossible** to run across a process boundary; there is no IPC design that fixes this, it is a property of how OpenGL contexts work. Option B remains *theoretically* worth revisiting only for a narrow, CPU-bound, serializable slice (e.g. font → bitmap, a pure data-in/data-out transform with no live GL state), but it is **not the path taken here** and is not scheduled.

**Accepted cost.** Once a piece uses the hosted-shell/C-meat pattern, the *glintfx binary as a whole* is no longer zero-libc-observable at the binary level — `libstdc++`'s own `memcpy` etc. is unavoidably present on the C++-hosted side. Purity stops being a property of the shipped binary and becomes a property of **the source, per module** (verifiable via `nm` + the `glx_` prefix convention, module by module). The standalone `core-v0.1.0` binary is unaffected and remains 100% pure and independently auditable — this cost is scoped to the mixed glintfx binary only, which was never claiming zero-libc purity in the first place (ADR-0006).

**PT:** Adotar a **Opção A — faseada**: link estático incremental, com um padrão obrigatório de separação entre o glue C++ e o núcleo algorítmico em C.

- **Link estático incremental.** A Camada 0 ganha um alvo de build, `libcore.a`, cujos símbolos públicos carregam prefixo de namespace próprio (`glx_*`). Este é o **destino** de toda peça internalizada: à medida que cada reimplementação clean-room da Camada 0 é entregue, ela é exposta via `libcore.a` e consumida **explicitamente** pela Camada 1 (`#include <core/...>`), linkada no binário do glintfx. É um estreitamento deliberado e pontual da cláusula "nunca linkam" do ADR-0006 — não desfaz a separação em duas camadas (o build system da Camada 0, suas restrições freestanding e sua disciplina zero-libc permanecem intactos); só muda como um artefato *pronto* da Camada 0 chega a um binário da Camada 1.
- **Padrão obrigatório: "casca-hosted + carne-C".** Qualquer classe C++ que implemente uma interface do RmlUi (`FontEngineInterface`, `RenderInterface`, …) permanece **C++ comum, hosted** (`libstdc++` normal, exceções normais, `operator new`/`delete` normal — é código de cola, não território soberano). Só a **carne algorítmica** dentro dela — parser SFNT, rasterizador de outline, geometria — é escrita como **C puro na Camada 0**, exposta via `extern "C"`, e linkada estaticamente. Isso torna o **`SOV-CXXRT`** (um runtime C++ freestanding: `operator new`/`delete`, `__cxa_guard_acquire/release`, `__cxa_atexit`, política de exceção) **opcional, não um bloqueador** para a internalização avançar — a soberania é gasta onde tem valor (o algoritmo), não em reimplementar código de cola C++ que o próprio contrato do RmlUi já obriga a ser C++.
- **Gates** — nada aqui executa antes destes:
  1. **`REL-TAG core-v0.1.0` está fechado.** Já feito; este ADR não teria sido escrito de outra forma.
  2. **`SOV-ALLOC` (um alocador com `free` real) é o 1º pré-requisito — hard-blocker.** A Camada 0 é bump-only por ADR-0004; código de UI aloca e libera a cada passe de layout. Nenhuma peça internalizada que aloque memória pode avançar antes do `SOV-ALLOC` chegar.
  3. **Regra de ownership de heap, aplicada e testada, não só documentada:** um ponteiro alocado por `glx_*` **nunca** pode ser passado a `free()` da glibc, nem vice-versa. Este é o risco *real* da Opção A — não colisão de símbolo (o prefixo `glx_` resolve isso trivialmente), mas **corrupção de heap silenciosa entre dois alocadores, que o ASan não pega através da fronteira**. Mitigação: tipos de handle opacos ou convenção de nome/sufixo em toda API que cruza ownership, mais um teste dedicado de fronteira que exercita mau uso cross-allocator de propósito (esperado falhar alto, não corromper em silêncio).
  4. **Piloto: gl3w.** Menor peça, contrato mecânico (carregamento de ponteiros de função contra uma lista de header fixa), quase nenhum envolvimento de heap. Prova o processo clean-room + `libcore.a` + `glx_` de ponta a ponta com risco mínimo, antes de se tentar um alvo pesado em heap (ex.: rasterização de glifo do FreeType).

**A Opção B (IPC) é formalmente descartada pro hot path.** Ponteiros de função GL e o contexto GL são estado **per-process** — o gl3w (que *é* a tabela de ponteiros de função) e o `RenderInterface` de efeitos (que emite chamadas GL a cada frame) são **fisicamente impossíveis** de rodar através de uma fronteira de processo; não existe design de IPC que resolva isso, é uma propriedade de como contextos OpenGL funcionam. A Opção B segue *teoricamente* cogitável só pra uma fatia estreita, CPU-bound e serializável (ex.: fonte → bitmap, uma transformação pura dado-entra/dado-sai sem estado GL vivo), mas **não é o caminho tomado aqui** e não está agendada.

**Custo aceito.** Uma vez que uma peça usa o padrão casca-hosted/carne-C, o *binário do glintfx como um todo* deixa de ser zero-libc-observável no nível binário — o próprio `memcpy` da `libstdc++`, entre outros, é inevitavelmente presente no lado C++ hosted. A pureza deixa de ser propriedade do binário entregue e passa a ser propriedade **da fonte, por-módulo** (verificável via `nm` + a convenção de prefixo `glx_`, módulo a módulo). O binário standalone `core-v0.1.0` não é afetado e continua 100% puro e auditável independentemente — esse custo tem escopo só no binário misto do glintfx, que nunca reivindicou pureza zero-libc pra começo de conversa (ADR-0006).

## Consequences / Consequências

**EN:** Internalization work (`SOV-*`/`TRANS-*`) has a concrete, executable target from now on: land a piece in `libcore.a` behind `glx_*`, wire it into a hosted-shell class, ship it in glintfx. `SOV-CXXRT` moves from "prerequisite for any internalization" to "its own independent track, pursued only where hosted glue genuinely becomes a bottleneck" — this materially de-risks and de-scopes the first internalization milestones. The `libcore.a` boundary is now a **one-way door for the linking model**: once glintfx statically links Layer-0 code, walking back to "the layers never link" (ADR-0006's original clause) would require unwinding every internalized module — hence this ADR's `one-way-door` tag, and why it required the líder's sign-off rather than being a two-way-door call left to the architect. The heap-ownership rule (gate 3) becomes a **standing test obligation**: every future internalized module that touches allocation must ship a boundary-misuse test, not just unit tests of its own correctness. `SOV-ALLOC` becomes a **release-blocking dependency** for any allocation-touching internalization item in `TODO.md` — schedule it first.

**PT:** O trabalho de internalização (`SOV-*`/`TRANS-*`) tem, a partir de agora, um alvo concreto e executável: entregar uma peça na `libcore.a` atrás de `glx_*`, ligá-la numa classe-casca hosted, embarcar no glintfx. O `SOV-CXXRT` passa de "pré-requisito pra qualquer internalização" para "trilha própria e independente, perseguida só onde o glue hosted virar gargalo de verdade" — isso desarrisca e reduz de escopo materialmente os primeiros marcos de internalização. A fronteira `libcore.a` é agora uma **porta de mão única pro modelo de linkagem**: uma vez que o glintfx linka estaticamente código da Camada 0, voltar atrás pra "as camadas nunca se linkam" (cláusula original do ADR-0006) exigiria desfazer cada módulo internalizado — daí a tag `one-way-door` deste ADR, e por que exigiu aval do líder em vez de ser uma chamada de porta-de-mão-dupla deixada ao arquiteto. A regra de ownership de heap (gate 3) vira uma **obrigação de teste permanente**: todo módulo internalizado futuro que toque alocação deve embarcar um teste de mau-uso de fronteira, não só testes unitários da própria correção. O `SOV-ALLOC` vira uma **dependência bloqueadora de release** pra qualquer item de internalização em `TODO.md` que toque alocação — agendá-lo primeiro.

## Options considered / Opções consideradas

- **Option A — phased static incremental link, "hosted shell + C meat" (chosen / escolhida).** `libcore.a` + `glx_*` prefix; RmlUi-interface classes stay C++ hosted, only the algorithmic core is sovereign C, statically linked. **Pros:** unblocks internalization now, without first building a full freestanding C++ runtime; low integration friction (one build artifact, ordinary linking); risk is concentrated and testable (heap ownership at explicit crossing points) rather than diffuse. **Cons:** the shipped glintfx binary is no longer zero-libc-observable as a whole; purity becomes a source-level, per-module property that must be actively verified (`nm` + convention), not a binary-level guarantee; a real, non-hypothetical new failure class (cross-allocator heap corruption) is introduced and must be tested for, not just assumed away.
- **Option B — separate process + IPC / processo separado + IPC.** Keeps ADR-0006's original framing literally: Layer 0 stays a fully separate, zero-libc process; Layer 1 talks to it over a socket/pipe/shared-memory protocol. **Pros:** preserves whole-binary zero-libc purity for any internalized piece running as its own process; strongest possible isolation (a crash in Layer 0 cannot corrupt Layer 1's heap). **Cons:** physically incompatible with the two actual internalization targets that matter most — gl3w (a per-process function-pointer table) and the effects `RenderInterface` (per-process GL context, issuing draw calls every frame at 60+ fps) cannot cross a process boundary at all; even where technically possible (e.g. font rasterization) it adds serialization overhead and IPC failure modes (partial writes, dead peer, backpressure) to every frame-critical call. Rejected for the hot path; left open only for a narrow future CPU-bound/serializable case.
- **Option C — full freestanding C++ (`SOV-CXXRT`) first, then internalize / C++ freestanding completo primeiro.** Build the freestanding C++ runtime (`operator new`/`delete`, `__cxa_guard`, `__cxa_atexit`, exception policy) before internalizing anything RmlUi-interface-shaped, so the *entire* internalized surface — glue and algorithm alike — can be sovereign C++ in Layer 0, no hosted shell needed. **Pros:** conceptually the cleanest end state — no permanent split between "sovereign algorithm" and "hosted glue"; whole-module purity, not just algorithmic-core purity. **Cons:** turns `SOV-CXXRT` into a hard blocker in front of every internalization milestone, including the gl3w pilot, which does not need it; is itself a substantial, open-ended piece of work (freestanding C++ runtimes are notoriously fiddly — exception unwinding tables, static-init guards, ABI edge cases) with no schedule; defers all internalization progress behind an unbounded prerequisite. Rejected **for now** — not discarded forever; Option A's hosted-shell pattern makes `SOV-CXXRT` optional rather than closing the door on it, so Option C's end state remains reachable later if a concrete case demands it (see Consequences).

## Addendum 2026-07-19 — PIE/ASLR cost of the -fno-pic archive / Custo PIE/ASLR do archive -fno-pic

**EN:** Audit `AUD-ABI-Δ` (orchestrator-verified via `readelf`) found a real, previously undocumented cost of Option A's static-link boundary: `libcore.a` is compiled `-fno-pic` — a direct, non-negotiable consequence of Layer 0's freestanding contract (ADR-0005/ELF layout: `ld -nostdlib -static -no-pie`) — and linking it into a **hosted** C++ consumer silently forces the *entire consumer binary* out of PIE. `readelf -h build/bin/libcore_consumer` reports `Type: EXEC` (ET_EXEC), not `DYN`: `ld` falls back to `ET_EXEC` without warning or error, because it cannot place non-PIC code inside a PIE image. The consumer **loses ASLR** for its whole address space, not just the linked-in `libcore.a` object files.

This is **not memory corruption** and **not a new vulnerability** — all 16 correctness checks in `libcore_consumer` pass; the C↔C++ boundary itself is functionally sound. It is scored 🟠 (important, not 🔴 critical) for that reason. It matters because Option A's stated destination for `libcore.a` is to host **hostile-input parsers** (SFNT, raster, hint — already archive members per the gl3w-then-FreeType pilot order in gate 4 above): once one of those lands in a real hosted production consumer, that consumer's entire process runs with ASLR disabled, which is a real hardening regression for anything parsing attacker-controlled input, even though the parser itself is memory-safe by construction.

**Scope, precisely:** this cost is confined to *hosted consumer binaries* that link `libcore.a` (e.g. `libcore_consumer`, and — later — glintfx itself once a hosted-shell/C-meat module is wired in). It does **not** touch the standalone Layer-0 `core-v0.1.0` binary, which is `-no-pie` by its own ADR-0005 design already and was never claiming ASLR in the first place. It is an **accepted cost**, cumulative with the zero-libc-observability cost already named in this ADR's "Accepted cost" section above — both are the price of Option A's static-link boundary, not new risks introduced by this addendum, just risks that were unnamed until this audit.

**Deferred alternative — `INBOX AUD-C0-PICVAR`.** A `-fpic` variant of the archive (`libcore_pic.a`) would let a PIE-requiring consumer link Layer-0 code without losing ASLR. This is **not built now**: no real consumer currently requires PIE (`libcore_consumer` is a test harness; glintfx has not yet linked a hosted-shell/C-meat module in production), and building/maintaining a second archive variant against a hypothetical consumer would be scope creep ahead of demand (cf. [[feedback_cadencia_releases_consumidor]] / [[feedback_escopo_distribuicao_geral]] — Layer-0 quality decisions target the general audience, but a `-fpic` build is added when a real consumer requires it, not speculatively). Seeded here as `AUD-C0-PICVAR` for `TODO.md`'s INBOX, to be pulled the day a PIE-requiring hosted consumer of `libcore.a` is real.

**PT:** A auditoria `AUD-ABI-Δ` (verificada pelo orquestrador via `readelf`) achou um custo real, até então não documentado, da fronteira de link estático da Opção A: a `libcore.a` é compilada `-fno-pic` — consequência direta e inegociável do contrato freestanding da Camada 0 (ADR-0005/layout ELF: `ld -nostdlib -static -no-pie`) — e linkar esse archive num consumidor C++ **hosted** força em silêncio o *binário-consumidor inteiro* pra fora de PIE. `readelf -h build/bin/libcore_consumer` reporta `Type: EXEC` (ET_EXEC), não `DYN`: o `ld` cai pra `ET_EXEC` sem aviso nem erro, porque não pode colocar código não-PIC dentro de uma imagem PIE. O consumidor **perde ASLR** pro seu espaço de endereço inteiro, não só pros object files da `libcore.a` linkados.

Isso **não é corrupção de memória** e **não é vulnerabilidade nova** — todas as 16 checagens de correção do `libcore_consumer` passam; a fronteira C↔C++ em si é funcionalmente sólida. É classificado 🟠 (importante, não 🔴 crítico) por isso. Importa porque o destino declarado da `libcore.a` na Opção A é hospedar **parsers de input hostil** (SFNT, raster, hint — já membros do archive pela ordem de piloto gl3w-depois-FreeType do gate 4 acima): no dia em que um desses chegar a um consumidor hosted de produção real, o processo inteiro desse consumidor roda com ASLR desligado, o que é uma regressão real de hardening pra qualquer coisa que parseie input controlado por atacante, mesmo que o parser em si seja memory-safe por construção.

**Escopo, com precisão:** esse custo fica confinado a *binários-consumidores hosted* que linkam a `libcore.a` (ex.: `libcore_consumer`, e — depois — o próprio glintfx quando um módulo casca-hosted/carne-C for ligado em produção). **Não** toca o binário standalone `core-v0.1.0` da Camada 0, que já é `-no-pie` pelo próprio design do ADR-0005 e nunca reivindicou ASLR pra começo de conversa. É um **custo aceito**, cumulativo com o custo de não-zero-libc-observável já nomeado na seção "Custo aceito" deste ADR acima — os dois são o preço da fronteira de link estático da Opção A, não riscos novos introduzidos por este adendo, só riscos que estavam sem nome até esta auditoria.

**Alternativa deferida — `INBOX AUD-C0-PICVAR`.** Uma variante `-fpic` do archive (`libcore_pic.a`) deixaria um consumidor que exige PIE linkar código da Camada 0 sem perder ASLR. Isso **não é construído agora**: nenhum consumidor real hoje exige PIE (`libcore_consumer` é harness de teste; o glintfx ainda não linkou nenhum módulo casca-hosted/carne-C em produção), e construir/manter uma segunda variante de archive contra um consumidor hipotético seria scope creep à frente da demanda (cf. [[feedback_cadencia_releases_consumidor]] / [[feedback_escopo_distribuicao_geral]] — decisões de qualidade da Camada 0 miram o público amplo, mas um build `-fpic` se soma quando um consumidor real exigir, não especulativamente). Semeado aqui como `AUD-C0-PICVAR` pra INBOX do `TODO.md`, pra ser puxado no dia em que um consumidor hosted real de `libcore.a` exigir PIE.
