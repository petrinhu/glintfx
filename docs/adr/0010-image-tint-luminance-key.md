# ADR-0010 — Decorator `image-tint()` — luminance-key texture tinting / Tingimento de textura por luminance-key

- **Status:** Proposed (2026-07-09) — the decorator-name deviation from the líder's original RCSS sketch (§ Decision, item b) needs explicit confirmation before implementation starts.
- **Deciders:** petrus (líder, pending confirmation on item b), software-architect
- **Tags:** architecture, api, additive, two-way-door, rendering
- **Relates to:** [[0006-layered-hybrid-architecture]] (Layer 1 boundary), [[0008-embed-guest-mode]] (`Gl3RenderInterface` is the same subclass extended here). Not to be confused with **ADR-0009**, reserved (per `TODO.md`, `AUD-TRANS-5`) for the Layer 0/Layer 1 boundary decision — this is a **different, unrelated** ADR, numbered 0010 to avoid the collision.

## Context / Contexto

**EN:** The consumer (GusWorld) has one neutral base texture (white runes + navy stone + gold trim) and wants **N domain colours from that one asset**, swapped at runtime via RCSS, animatable. The native RmlUi `image-color` property (already shipped, documented in `docs/effects.md`) does a uniform multiply — it tints the gold trim along with the runes, which is visually wrong for this asset. A **luminance-key** tint — recolour only light+neutral texels, weighted by `smoothstep(threshold,1,luminance) × (1-saturation)`, leaving saturated (gold) and dark (stone) texels alone — is not a capability RmlUi's shipped GL3 backend has; no combination of its existing `CompileShader` kinds (`linear-gradient`/`radial-gradient`/`conic-gradient`/`shader:creation`) samples an arbitrary bound texture, so no amount of uniform trickery can approximate this. A genuinely new fragment shader is required.

The project's non-negotiable constraint (`CLAUDE.md`, "Camada 1 (glintfx): pode linkar as libs listadas em `NOTICE`... Novas deps externas ainda exigem decisão do líder") extends by established precedent (`decorator_polygon.hpp`'s own file header) to: **no patch to the vendored RmlUi source** (`examples/RmlUi`, pinned via `FetchContent` at `2cd28864ae25ed345b70598751703a5433b12356`). Any solution must reach the new shader exclusively through RmlUi's **public** API surface (`Include/RmlUi/Core/*.h`), the same rule `polygon()`'s gradient-fill reuse already follows.

**PT:** O consumidor (GusWorld) tem uma textura base neutra (runas brancas + pedra navy + trim ouro) e quer **N cores de domínio a partir desse único asset**, trocadas em runtime via RCSS, animáveis. A propriedade nativa `image-color` do RmlUi (já entregue, documentada em `docs/effects.md`) faz um multiply uniforme — tinge o trim ouro junto com as runas, visualmente errado pra este asset. Um tingimento **luminance-key** — recolorir só texels claros+neutros, ponderado por `smoothstep(threshold,1,luminância) × (1-saturação)`, preservando texels saturados (ouro) e escuros (pedra) — não é uma capacidade que o backend GL3 embarcado do RmlUi tem; nenhuma combinação dos tipos existentes de `CompileShader` (`linear-gradient`/`radial-gradient`/`conic-gradient`/`shader:creation`) amostra uma textura arbitrária vinculada, então nenhum truque de uniform consegue aproximar isso. Um fragment shader genuinamente novo é necessário.

A restrição inegociável do projeto (`CLAUDE.md`, "Camada 1 (glintfx): pode linkar as libs listadas em `NOTICE`... Novas deps externas ainda exigem decisão do líder") se estende, por precedente já estabelecido (o próprio cabeçalho de `decorator_polygon.hpp`), a: **nenhum patch no source vendorizado do RmlUi** (`examples/RmlUi`, pinado via `FetchContent` em `2cd28864ae25ed345b70598751703a5433b12356`). Qualquer solução precisa alcançar o shader novo exclusivamente pela superfície **pública** do RmlUi (`Include/RmlUi/Core/*.h`), a mesma regra que o reuso de gradiente do `polygon()` já segue.

## Investigation summary — why the obvious approach fails / Resumo da investigação — por que a abordagem óbvia falha

**EN:** The brief's starting hypothesis was "override `CompileShader`/`RenderShader` on `Gl3RenderInterface` (`glintfx/src/render_gl3.cpp`) to recognise a new shader name and dispatch to a custom GL program." Both methods **are** virtual and non-`final` on the base (`RenderInterface_GL3` — `examples/RmlUi/Backends/RmlUi_Renderer_GL3.h:64-67`), so the override itself compiles. The wall is one level deeper: `RenderShader`'s 2nd parameter, `Rml::CompiledGeometryHandle`, is a raw `uintptr_t` (`Include/RmlUi/Core/Types.h:79-80`) that — for the concrete GL3 backend — actually points to `Gfx::CompiledGeometryData` (holds the VAO), a struct **defined only inside `RmlUi_Renderer_GL3.cpp`** (`:445`), **not forward-declared in any installed header** (`RmlUi_Renderer_GL3.h` forward-declares only `Gfx::ProgramData`/`Gfx::FramebufferData`, not `CompiledGeometryData`). A glintfx translation unit cannot legally dereference that handle — the type is incomplete outside RmlUi's own `.cpp`, and reconstructing its layout by guesswork would be exactly the "reaching into an implementation-detail header RmlUi is free to change without notice" anti-pattern `decorator_polygon.cpp` already explicitly rejected for `ComputeAngle`. **Verdict: reusing RmlUi's own compiled-geometry buffer for a custom shader draw is not viable without a core patch.**

Three follow-on findings make a **different**, still-non-invasive design viable:

1. **`Rml::TextureHandle` is `uintptr_t` too, but for GL3 it genuinely *is* the raw `GLuint`** (`RenderInterface_GL3::GenerateTexture` — `Backends/RmlUi_Renderer_GL3.cpp:1280` — `return (Rml::TextureHandle)texture_id;`). `Geometry::Render(translation, texture, shader)` resolves an `Rml::Texture` to this raw handle **internally** and hands it to `RenderShader`'s 4th parameter — for free, via 100% public API. No custom texture loading/caching is needed; `RenderShader`'s `texture` argument, once cast to `GLuint`, is directly bindable.
2. **`RenderManager::GetState().transform`** (`Include/RmlUi/Core/RenderManager.h:64`, `RenderState::transform` field, `:33-37`) is the backend-agnostic, **public** equivalent of the private `RenderInterface_GL3::GetTransform()` — readable from *any* decorator code, no bridge into the concrete backend needed for CSS `transform:` support, *if* the draw happens outside `Gl3RenderInterface`. But once the draw happens *inside* an override method of `Gl3RenderInterface` itself (see Decision), `this->GetTransform()` (public, `RmlUi_Renderer_GL3.h:76`) is directly reachable — and, critically, it already returns `projection × transform` pre-combined (`SetTransform`, `Backends/RmlUi_Renderer_GL3.cpp:1471`: `transform = projection * (*new_transform)`), matching `shader_vert_main`'s `gl_Position = _transform * vec4(pos + _translate, 0, 1)` (`:50-69`) exactly — no independent reconstruction of the ortho projection is needed.
3. **`RenderInterface_GL3::ResetProgram()`** (public, `RmlUi_Renderer_GL3.h:77` — `void ResetProgram() { UseProgram(ProgramId::None); }`, impl `Backends/RmlUi_Renderer_GL3.cpp:2137`) exists precisely to invalidate the backend's private `active_program` cache after external GL calls change the bound program — but it, too, is a method of the *concrete* class, unreachable from decorator code (`RenderGl3::iface()` in glintfx only exposes the *abstract* `Rml::RenderInterface*`, and no bridge to the concrete `Gl3RenderInterface` exists anywhere in glintfx today — confirmed by grep, `ui_layer.cpp`'s `GlStateGuard` does not reach it either). Any raw `glUseProgram()` call made **outside** `Gl3RenderInterface` would silently desync this cache and corrupt the *next* native draw call (RmlUi would believe its own program is still bound and skip re-`glUseProgram`-ing it).

**Conclusion:** points 2 and 3 together mean the custom draw **must** execute from *inside* an override on `Gl3RenderInterface` itself (where `this->GetTransform()`/`this->ResetProgram()` are free, inherited, public members) — not from a separate free function or from decorator code directly. That constrains the shape of the solution but does not block it: `CompileShader`/`RenderShader` (both overridable) remain the right entry point, **provided the geometry draw call itself never touches the RmlUi-opaque `geometry_handle`** — glintfx must own an entirely independent, tiny VAO/VBO/EBO (2 attributes: position, texcoord — no need to match RmlUi's own `inColor0`-carrying vertex layout, since this is a wholly separate GL program) for the quad it draws, built once per decorated element and threaded into `CompileShader`'s `Dictionary` parameters (which **do** support `UINT`/`FLOAT`/`COLOURF`/`INT` — `Include/RmlUi/Core/Variant.h:20-47` — confirmed against `decorator_polygon.cpp`'s own established `Rml::Variant(Vector2f)`/`Rml::Variant(bool)` usage).

**PT:** A hipótese inicial do brief era "sobrescrever `CompileShader`/`RenderShader` no `Gl3RenderInterface` (`glintfx/src/render_gl3.cpp`) pra reconhecer um nome de shader novo e despachar pra um programa GL customizado." Os dois métodos **são** virtuais e não-`final` na base (`RenderInterface_GL3` — `examples/RmlUi/Backends/RmlUi_Renderer_GL3.h:64-67`), então o override em si compila. O muro está um nível mais fundo: o 2º parâmetro de `RenderShader`, `Rml::CompiledGeometryHandle`, é um `uintptr_t` cru (`Include/RmlUi/Core/Types.h:79-80`) que — para o backend GL3 concreto — de fato aponta pra `Gfx::CompiledGeometryData` (guarda a VAO), uma struct **definida só dentro de `RmlUi_Renderer_GL3.cpp`** (`:445`), **não forward-declarada em nenhum header instalado** (`RmlUi_Renderer_GL3.h` forward-declara só `Gfx::ProgramData`/`Gfx::FramebufferData`, não `CompiledGeometryData`). Uma translation unit da glintfx não pode legalmente desreferenciar esse handle — o tipo é incompleto fora do `.cpp` do próprio RmlUi, e reconstruir seu layout por suposição seria exatamente o anti-padrão de "alcançar um header de detalhe-de-implementação que o RmlUi é livre pra mudar sem aviso" que o próprio `decorator_polygon.cpp` já rejeitou explicitamente pro `ComputeAngle`. **Veredito: reusar o buffer de geometria compilada do próprio RmlUi pra um draw de shader customizado não é viável sem patch de core.**

Três achados subsequentes tornam um design **diferente**, ainda não-invasivo, viável:

1. **`Rml::TextureHandle` também é `uintptr_t`, mas pro GL3 de fato *é* o `GLuint` cru** (`RenderInterface_GL3::GenerateTexture` — `Backends/RmlUi_Renderer_GL3.cpp:1280` — `return (Rml::TextureHandle)texture_id;`). `Geometry::Render(translation, texture, shader)` resolve uma `Rml::Texture` pra esse handle cru **internamente** e entrega ao 4º parâmetro de `RenderShader` — de graça, via API 100% pública. Nenhum carregamento/cache de textura customizado é necessário; o argumento `texture` de `RenderShader`, uma vez castado pra `GLuint`, é diretamente vinculável.
2. **`RenderManager::GetState().transform`** (`Include/RmlUi/Core/RenderManager.h:64`, campo `RenderState::transform`, `:33-37`) é o equivalente agnóstico-de-backend, **público**, do `RenderInterface_GL3::GetTransform()` privado — legível de *qualquer* código de decorator, sem precisar de ponte pro backend concreto pra suportar `transform:` do CSS, *se* o draw acontecer fora do `Gl3RenderInterface`. Mas uma vez que o draw acontece *dentro* de um método override do próprio `Gl3RenderInterface` (ver Decisão), `this->GetTransform()` (público, `RmlUi_Renderer_GL3.h:76`) é diretamente alcançável — e, criticamente, já retorna `projeção × transform` pré-combinados (`SetTransform`, `Backends/RmlUi_Renderer_GL3.cpp:1471`: `transform = projection * (*new_transform)`), casando exatamente com `gl_Position = _transform * vec4(pos + _translate, 0, 1)` do `shader_vert_main` (`:50-69`) — nenhuma reconstrução independente da projeção ortográfica é necessária.
3. **`RenderInterface_GL3::ResetProgram()`** (público, `RmlUi_Renderer_GL3.h:77` — `void ResetProgram() { UseProgram(ProgramId::None); }`, impl `Backends/RmlUi_Renderer_GL3.cpp:2137`) existe precisamente pra invalidar o cache privado `active_program` do backend após chamadas GL externas mudarem o programa vinculado — mas ele também é método da classe *concreta*, inalcançável do código de decorator (`RenderGl3::iface()` na glintfx hoje só expõe o `Rml::RenderInterface*` *abstrato*, e nenhuma ponte pro `Gl3RenderInterface` concreto existe em lugar nenhum da glintfx hoje — confirmado por grep, nem o `GlStateGuard` de `ui_layer.cpp` o alcança). Qualquer `glUseProgram()` cru feito **fora** do `Gl3RenderInterface` dessincronizaria silenciosamente esse cache e corromperia a *próxima* chamada de draw nativa (o RmlUi acreditaria que seu próprio programa continua vinculado e pularia o re-`glUseProgram`).

**Conclusão:** os pontos 2 e 3 juntos significam que o draw customizado **precisa** executar de *dentro* de um override no próprio `Gl3RenderInterface` (onde `this->GetTransform()`/`this->ResetProgram()` são membros públicos, herdados, de graça) — não de uma função livre separada nem do código do decorator diretamente. Isso restringe a forma da solução mas não a bloqueia: `CompileShader`/`RenderShader` (ambos sobrescrevíveis) continuam o ponto de entrada certo, **desde que a chamada de draw da geometria em si nunca toque o `geometry_handle` opaco do RmlUi** — a glintfx precisa possuir uma VAO/VBO/EBO própria, inteiramente independente e pequena (2 atributos: posição, texcoord — sem necessidade de casar com o layout de vértice `inColor0`-carregando do próprio RmlUi, já que este é um programa GL totalmente separado) para o quad que desenha, construída uma vez por elemento decorado e passada aos parâmetros `Dictionary` do `CompileShader` (que **suportam** `UINT`/`FLOAT`/`COLOURF`/`INT` — `Include/RmlUi/Core/Variant.h:20-47` — confirmado contra o uso já estabelecido de `Rml::Variant(Vector2f)`/`Rml::Variant(bool)` do próprio `decorator_polygon.cpp`).

## Decision / Decisão

**EN:**

**(a) Shader injection mechanism — override `CompileShader`+`RenderShader`+`ReleaseShader` on `Gl3RenderInterface`, tagged-wrapper dispatch.** `Gl3RenderInterface::CompileShader(name, params)`: if `name == "glintfx-tint"`, build glintfx's own small tag struct (`vao`, `index_count`, tint colour/mode/threshold, all read from `params`) and return it wrapped in a tiny discriminated union `GlintfxShaderHandle{is_tint, ...}`; for every other name, delegate unchanged to `RenderInterface_GL3::CompileShader(name, params)` and wrap *that* result too (same wrapper, `is_tint=false`) — **every** `CompileShader` call (including `polygon()`'s own gradient calls) funnels through this override once virtual dispatch resolves to the concrete `Gl3RenderInterface`, so wrapping is mandatory for `RenderShader`/`ReleaseShader` to tell the two cases apart safely. `RenderShader`: unwrap; if `is_tint`, bind glintfx's own lazily-compiled GLSL program + the caller-supplied VAO (glintfx-owned, **never** RmlUi's `geometry_handle`, which is left untouched), set uniforms from `this->GetTransform()` (public, pre-combined projection×transform) + the `translation`/`texture` parameters RmlUi already resolved for us, draw, then call `this->ResetProgram()` before returning; else delegate unchanged to `RenderInterface_GL3::RenderShader(unwrapped_handle, geometry_handle, translation, texture)`. `ReleaseShader` mirrors the same unwrap-and-delegate-or-free-tag-only split (the VAO itself is **not** owned by the shader handle — see (c)).

**(b) RCSS surface — REQUIRES a naming adjustment from the líder's original sketch.** The literal syntax proposed (`decorator: image(base.png); image-tint-color: ...;`) cannot be wired to a native RmlUi decorator without a core patch — the built-in `image` decorator is RmlUi's own C++ class, compiled without any knowledge of `image-tint-*`, and reading arbitrary sibling properties from inside a *foreign* decorator's `RenderElement` is not something RmlUi's decorator model supports. The decorator **function name itself must be glintfx's own**:
```rcss
.card {
  decorator: image-tint( base.png );     /* NEW glintfx decorator, not the native "image" */
  image-tint-color: var(--domain);       /* unchanged from the brief -- global, interpolable */
  image-tint-mode: luminance-multiply;   /* unchanged -- none | multiply | luminance-multiply | screen */
  image-tint-threshold: 0.55;            /* unchanged */
}
```
`image-tint-color`/`-mode`/`-threshold` stay exactly as specified: **standalone, globally-registered element properties** (`Rml::StyleSheetSpecification::RegisterProperty`, *not* decorator-shorthand arguments), read directly off the element (`element->GetProperty(...)`) inside `ImageTintDecorator::RenderElement` — orthogonal to the decorator function, animatable independently via ordinary `transition`/`@keyframes` (RmlUi interpolates any `Unit::COLOUR`/`Unit::NUMBER` property regardless of which C++ decorator consumes it). `image-tint-mode: none` (default) renders through the **vanilla** `Geometry::Render(translation, texture)` path (no custom shader compiled, `ProgramId::Texture`, identical cost/behaviour to a plain `image()` decorator) — the new machinery in (a) is only exercised when an author opts into a non-`none` mode.

**(c) Per-element data — one VAO, shader recompiled every `RenderElement()` call (not cached).** `ImageTintDecorator::GenerateElementData` builds, once per element (rebuilt only when the decorator's own `src` argument changes — the normal RmlUi decorator-data lifecycle), a 4-vertex/6-index quad VAO/VBO/EBO sized to the element's fill box, via raw gl3w calls — glintfx's first hand-rolled VAO outside of what RmlUi itself manages. `image-tint-color`/`-mode`/`-threshold` (and CSS `opacity`) are **not** baked at `GenerateElementData` time — because they are ordinary, independently-transitionable properties (not decorator arguments), RmlUi does **not** re-run `GenerateElementData` when only they change, so baking them once would go visually stale under `transition:`/`@keyframes`. Instead, `RenderElement` re-reads them **fresh every call** (every frame, for every visible instance) and calls `render_manager->CompileShader("glintfx-tint", {...fresh values..., "vao": <cached GLuint>})` immediately followed by `Geometry::Render(translation, texture, shader)` — the local `Rml::CompiledShader` is a `UniqueRenderResource` (RAII, same idiom `decorator_polygon.cpp` already documents) and self-releases at end of scope. Only the VAO/VBO/EBO (geometry-independent of tint state) is long-lived, released in `ReleaseElementData`.

**(d) Colour space — no gamma/sRGB correction; premultiply un/re-applied around the luminance math.** `RenderInterface_GL3::GenerateTexture` uploads `GL_RGBA8` (not `GL_SRGB8_ALPHA8`) and every existing blend/gradient/multiply operation in this renderer treats channel values as plain [0,1] numbers with **no** linearisation anywhere in the pipeline (confirmed by grep — `GL_SRGB8_ALPHA8` appears only in a *disabled alternative* comment, never enabled). Applying gamma correction in the new tint shader alone would make tinted elements look inconsistently brighter/darker than every neighbouring untinted/gradient/`image-color`'d element on screen — **recommendation: do not add gamma correction**, unless the consumer has a specific, tested reason tied to how `base.png` was authored (flagged as an open question for the líder/consumer, not decided here). The one colour-space correction that **is** mandatory (independent of gamma) is **un-premultiplying before the luminance/saturation math and re-premultiplying the output**: `Gl3RenderInterface::LoadTexture` already premultiplies every texture's RGB by its own alpha on the CPU (`render_gl3.cpp:116-138`), so a texel's `rgb` at partial alpha is *already* scaled down — computing luminance/saturation directly on that scaled-down `rgb` (as the consumer's GLSL sketch does) systematically **under-weights** the tint on semi-transparent/anti-aliased pixels (edges would tint less than solid interior, a visible artifact). The shader must divide by `texel.a` (guarded against 0) before the luminance-key math, then multiply back by `texel.a × image-tint-color.a × element.opacity` on the way out. See the plan for the corrected GLSL.

**(e) Texture loading/caching.** `DecoratorInstancerInterface::GetTexture(src)` (`Include/RmlUi/Core/Decorator.h:93` — resolves relative paths against the document that declared the `decorator:` rule, honours `set_asset_base_url`) is called **once**, at `InstanceDecorator` time (mirrors `PolygonDecorator::Initialise` baking resolved args into the decorator instance) — **not** per-element. This means glintfx neither duplicates `Gl3RenderInterface::LoadTexture`'s stb_image/premultiply logic nor bypasses RmlUi's own texture cache/dedup: every element sharing the same `decorator: image-tint(base.png)` rule shares one `Rml::Texture`, exactly like the native `image` decorator does.

**PT:**

**(a) Mecanismo de injeção de shader — sobrescrever `CompileShader`+`RenderShader`+`ReleaseShader` no `Gl3RenderInterface`, despacho por wrapper marcado.** `Gl3RenderInterface::CompileShader(name, params)`: se `name == "glintfx-tint"`, constrói uma pequena struct de tag própria da glintfx (`vao`, `index_count`, cor/modo/threshold de tingimento, todos lidos de `params`) e retorna embrulhada numa pequena união discriminada `GlintfxShaderHandle{is_tint, ...}`; para todo outro nome, delega inalterado a `RenderInterface_GL3::CompileShader(name, params)` e embrulha *esse* resultado também (mesmo wrapper, `is_tint=false`) — **toda** chamada de `CompileShader` (inclusive as chamadas de gradiente do próprio `polygon()`) passa por este override uma vez que o despacho virtual resolve pro `Gl3RenderInterface` concreto, então embrulhar é obrigatório pra `RenderShader`/`ReleaseShader` distinguirem os dois casos com segurança. `RenderShader`: desembrulha; se `is_tint`, vincula o próprio programa GLSL compilado sob demanda + a VAO fornecida pelo chamador (própria da glintfx, **nunca** o `geometry_handle` do RmlUi, que fica intocado), seta uniforms a partir de `this->GetTransform()` (público, projeção×transform já combinados) + os parâmetros `translation`/`texture` que o RmlUi já resolveu pra gente, desenha, então chama `this->ResetProgram()` antes de retornar; senão delega inalterado a `RenderInterface_GL3::RenderShader(handle_desembrulhado, geometry_handle, translation, texture)`. `ReleaseShader` espelha a mesma divisão desembrulha-e-delega-ou-libera-só-a-tag (a VAO em si **não** é dona do shader handle — ver (c)).

**(b) Superfície RCSS — EXIGE um ajuste de nomenclatura em relação ao esboço original do líder.** A sintaxe literal proposta (`decorator: image(base.png); image-tint-color: ...;`) não pode ser conectada a um decorator nativo do RmlUi sem patch de core — o decorator embutido `image` é uma classe C++ própria do RmlUi, compilada sem nenhum conhecimento de `image-tint-*`, e ler propriedades irmãs arbitrárias de dentro do `RenderElement` de um decorator *alheio* não é algo que o modelo de decorator do RmlUi suporta. O **nome da função do decorator em si precisa ser próprio da glintfx**:
```rcss
.card {
  decorator: image-tint( base.png );     /* decorator NOVO da glintfx, não o "image" nativo */
  image-tint-color: var(--domain);       /* inalterado em relação ao brief -- global, interpolável */
  image-tint-mode: luminance-multiply;   /* inalterado -- none | multiply | luminance-multiply | screen */
  image-tint-threshold: 0.55;            /* inalterado */
}
```
`image-tint-color`/`-mode`/`-threshold` ficam exatamente como especificado: **propriedades de elemento standalone, registradas globalmente** (`Rml::StyleSheetSpecification::RegisterProperty`, *não* argumentos de shorthand de decorator), lidas diretamente do elemento (`element->GetProperty(...)`) dentro de `ImageTintDecorator::RenderElement` — ortogonais à função do decorator, animáveis independentemente via `transition`/`@keyframes` comuns (o RmlUi interpola qualquer propriedade `Unit::COLOUR`/`Unit::NUMBER` independente de qual decorator C++ a consome). `image-tint-mode: none` (default) renderiza pelo caminho **vanilla** `Geometry::Render(translation, texture)` (nenhum shader customizado compilado, `ProgramId::Texture`, custo/comportamento idêntico a um decorator `image()` simples) — a maquinaria nova de (a) só é exercitada quando um autor opta por um modo diferente de `none`.

**(c) Dado por-elemento — uma VAO, shader recompilado a cada chamada de `RenderElement()` (não cacheado).** `ImageTintDecorator::GenerateElementData` constrói, uma vez por elemento (reconstruída só quando o argumento `src` do próprio decorator muda — o ciclo de vida normal de dado-de-decorator do RmlUi), uma VAO/VBO/EBO de quad de 4 vértices/6 índices dimensionada à caixa de preenchimento do elemento, via chamadas gl3w cruas — a primeira VAO feita à mão da glintfx fora do que o próprio RmlUi gerencia. `image-tint-color`/`-mode`/`-threshold` (e o `opacity` do CSS) **não** são embutidos no momento de `GenerateElementData` — como são propriedades comuns, transicionáveis independentemente (não argumentos de decorator), o RmlUi **não** re-roda `GenerateElementData` quando só elas mudam, então embuti-los uma vez ficaria visualmente obsoleto sob `transition:`/`@keyframes`. Em vez disso, `RenderElement` os relê **frescos a cada chamada** (todo frame, pra toda instância visível) e chama `render_manager->CompileShader("glintfx-tint", {...valores frescos..., "vao": <GLuint cacheado>})` imediatamente seguido de `Geometry::Render(translation, texture, shader)` — o `Rml::CompiledShader` local é um `UniqueRenderResource` (RAII, mesmo idioma que `decorator_polygon.cpp` já documenta) e se autolibera ao fim do escopo. Só a VAO/VBO/EBO (independente do estado de tingimento) é de vida longa, liberada em `ReleaseElementData`.

**(d) Espaço de cor — sem correção gama/sRGB; despremultiplicação/repremultiplicação ao redor da matemática de luminância.** `RenderInterface_GL3::GenerateTexture` faz upload em `GL_RGBA8` (não `GL_SRGB8_ALPHA8`) e toda operação de blend/gradiente/multiply já existente neste renderer trata valores de canal como números [0,1] puros sem **nenhuma** linearização em lugar nenhum do pipeline (confirmado por grep — `GL_SRGB8_ALPHA8` só aparece num comentário de *alternativa desabilitada*, nunca ativada). Aplicar correção gama só no shader de tingimento novo faria elementos tingidos parecerem inconsistentemente mais claros/escuros que todo elemento vizinho não-tingido/gradiente/`image-color`ado na tela — **recomendação: não adicionar correção gama**, a menos que o consumidor tenha um motivo específico e testado ligado a como `base.png` foi autorada (sinalizado como pergunta em aberto pro líder/consumidor, não decidido aqui). A correção de espaço de cor que **é** obrigatória (independente de gama) é **despremultiplicar antes da matemática de luminância/saturação e repremultiplicar a saída**: `Gl3RenderInterface::LoadTexture` já premultiplica o RGB de toda textura pelo próprio alpha na CPU (`render_gl3.cpp:116-138`), então o `rgb` de um texel em alpha parcial já vem escalado pra baixo — calcular luminância/saturação direto nesse `rgb` escalado (como o esboço GLSL do consumidor faz) sistematicamente **subpondera** o tingimento em pixels semi-transparentes/anti-aliased (bordas tingiriam menos que o interior sólido, um artefato visível). O shader precisa dividir por `texel.a` (protegido contra 0) antes da matemática de luminance-key, depois multiplicar de volta por `texel.a × image-tint-color.a × opacity-do-elemento` na saída. Ver o plano pro GLSL corrigido.

**(e) Carregamento/cache de textura.** `DecoratorInstancerInterface::GetTexture(src)` (`Include/RmlUi/Core/Decorator.h:93` — resolve caminhos relativos contra o documento que declarou a regra `decorator:`, respeita `set_asset_base_url`) é chamado **uma vez**, no momento do `InstanceDecorator` (espelha `PolygonDecorator::Initialise` embutindo argumentos resolvidos na instância do decorator) — **não** por-elemento. Isso significa que a glintfx nem duplica a lógica stb_image/premultiply de `Gl3RenderInterface::LoadTexture` nem contorna o cache/dedup de textura do próprio RmlUi: todo elemento compartilhando a mesma regra `decorator: image-tint(base.png)` compartilha uma `Rml::Texture`, exatamente como o decorator nativo `image` faz.

## Consequences / Consequências

**EN:**

**Positive:**
- Zero patch to vendored RmlUi source; the FetchContent pin (`2cd28864ae25ed345b70598751703a5433b12356`) stays untouched.
- Reuses established, already-audited glintfx idioms: `PolygonDecorator`'s Initialise/GenerateElementData/ReleaseElementData/RenderElement shape, `Gl3RenderInterface`'s existing subclass-override pattern (previously only `LoadTexture`), the fail-high hardening philosophy (`sides∈[3,1024]` precedent).
- `image-tint-mode: none` (the default) has **zero** runtime cost beyond a plain `image()` decorator — no shader compiled, no custom VAO drawn.
- Colour/mode/threshold stay genuinely orthogonal, RCSS-idiomatic, interpolable properties, exactly as the líder specified — the only deviation is the decorator function's *name*.

**Negative / accepted as cost:**
- **First hand-rolled GLSL program + raw VAO in glintfx.** Every prior "GL3 effect" (glow, blur, backdrop-blur, mask, gradient fill) was either 100% native RmlUi (`filter`/`backdrop-filter`/`mask-image`) or reused RmlUi's *own* shipped shader (`polygon()`'s gradient reuse). This is the first genuinely new fragment shader authored by glintfx, interleaved with RmlUi's own render pass — a real increase in the surface a future maintainer must understand to safely touch `render_gl3.cpp`.
- `render_manager->CompileShader("glintfx-tint", ...)` runs **every frame**, for every tinted (non-`none`-mode) element, not once — a deliberate trade (freshness for animation correctness) with a real, if small, per-frame allocation cost (one `new`/`delete` pair for the wrapper struct). Acceptable at UI-icon counts; would need revisiting if a document ever tints hundreds of simultaneously-visible elements.
- Every `CompileShader`/`RenderShader`/`ReleaseShader` call in the **entire application** (including `polygon()`'s gradients, unrelated to tint) now passes through one extra wrap/unwrap indirection — negligible (compile-time, not per-frame, for non-tint shaders) but universal.

**Risks / points of attention (for the implementer to validate before/while building):**
- **State-restore discipline.** `this->ResetProgram()` must run on *every* exit path of the tint branch of `RenderShader` (including any early-return on a GL error), or the very next native RmlUi draw call in the same frame can silently render with the wrong (tint) program still bound. Needs a test that renders a tinted element immediately followed by a `polygon()`/gradient/plain-text element in the same frame and diff-checks the result.
- **Blend state assumption.** The design assumes RmlUi's ambient `glBlendFunc(GL_ONE, GL_ONE_MINUS_SRC_ALPHA)` (set once per `BeginFrame`, per `render_gl3.cpp`'s own header comment on premultiplied compositing) is already correctly bound when `RenderShader` runs — the tint draw call must **not** touch `glBlendFunc`/`glEnable(GL_BLEND)` itself. Should hold given `RenderShader` only ever runs mid-frame, but worth an explicit assertion/comment, not just an assumption.
- **Scissor/clip state.** An element inside `overflow: hidden` relies on ambient `glScissor`/clip-mask GL state already being active when `RenderElement` runs (RmlUi applies it before calling into the decorator). The tint draw must not disable/reset scissor — should be automatic (we never touch `GL_SCISSOR_TEST`), but needs a test with a tinted image inside a scrollable/clipped container.
- **Gamma correction — open question, not decided here (see (d)).** If the consumer's actual `base.png` was authored/exported with an sRGB-aware tool expecting linear-space compositing, the "no gamma correction" recommendation may need revisiting — confirm visually against the real asset before shipping, don't assume the ADR's recommendation is final without a look at the actual texture.
- **`image-tint-threshold`/`-color`/`-mode` hardening specifics** (líder's explicit ask): `-mode` gated first by RmlUi's own `keyword` parser (invalid keyword never reaches C++ — falls back to the property's default/inherited value automatically), a defense-in-depth `default: → None` switch case kept anyway (matches the codebase's "RCSS parser is a transport, not a gate" philosophy already documented in `decorator_polygon.cpp`); `-threshold` clamped to `[0.0, 0.999]` **after** an `isfinite` guard (mirrors the `sides`/`rotation` NaN/Inf hardening precedent exactly — float-cast-overflow class of bug; the upper bound is strictly below `1.0`, not `1.0` itself — a post-release-v0.7.0-review fix, see the Addendum below, since a clamp landing exactly on `1.0` feeds the shader's `smoothstep(edge0, 1.0, L)` with `edge0==edge1`, GLSL UB by spec); `-color` needs no extra clamp (any resolved `Colourb` is valid); missing/failed texture load (`!texture` from `GetTexture`) → `InstanceDecorator` returns `nullptr` (decorator ignored, fail-high, same as `polygon()`'s `sides<3`/invalid-fill rejection and the `load(nullptr)` precedent).
- **The `image-tint()` naming deviation (b) needs the líder's explicit sign-off** before implementation starts — this ADR's `Status: Proposed` reflects that pending confirmation, not a closed decision.

**PT:**

**Positivas:**
- Zero patch no source vendorizado do RmlUi; o pin do FetchContent (`2cd28864ae25ed345b70598751703a5433b12356`) permanece intocado.
- Reusa idiomas já estabelecidos e auditados da glintfx: a forma Initialise/GenerateElementData/ReleaseElementData/RenderElement do `PolygonDecorator`, o padrão de override de subclasse já existente do `Gl3RenderInterface` (antes só `LoadTexture`), a filosofia de hardening fail-high (precedente `sides∈[3,1024]`).
- `image-tint-mode: none` (o default) tem **zero** custo de runtime além de um decorator `image()` simples — nenhum shader compilado, nenhuma VAO customizada desenhada.
- Cor/modo/threshold permanecem propriedades genuinamente ortogonais, idiomáticas ao RCSS, interpoláveis, exatamente como o líder especificou — o único desvio é o *nome* da função do decorator.

**Negativas / aceitas como custo:**
- **Primeiro programa GLSL feito à mão + VAO crua na glintfx.** Todo "efeito GL3" anterior (glow, blur, backdrop-blur, mask, preenchimento em gradiente) era ou 100% nativo do RmlUi (`filter`/`backdrop-filter`/`mask-image`) ou reusava o shader **próprio** já embarcado do RmlUi (o reuso de gradiente do `polygon()`). Este é o primeiro fragment shader genuinamente novo autorado pela glintfx, intercalado com o próprio render pass do RmlUi — um aumento real da superfície que um futuro mantenedor precisa entender pra mexer com segurança em `render_gl3.cpp`.
- `render_manager->CompileShader("glintfx-tint", ...)` roda **todo frame**, pra todo elemento tingido (modo diferente de `none`), não uma vez só — uma troca deliberada (frescor pela correção da animação) com um custo real, ainda que pequeno, de alocação por-frame (um par `new`/`delete` pra struct wrapper). Aceitável em contagens de ícone-de-UI; precisaria ser revisitado se um documento algum dia tingir centenas de elementos simultaneamente visíveis.
- Toda chamada `CompileShader`/`RenderShader`/`ReleaseShader` na aplicação **inteira** (inclusive os gradientes do `polygon()`, sem relação com tingimento) agora passa por uma indireção extra de embrulhar/desembrulhar — negligível (tempo de compile, não por-frame, pra shaders não-tint) mas universal.

**Riscos / pontos de atenção (pro implementer validar antes/durante a construção):**
- **Disciplina de restauração de estado.** `this->ResetProgram()` precisa rodar em *todo* caminho de saída do ramo de tint de `RenderShader` (inclusive qualquer early-return em erro de GL), senão a próxima chamada de draw nativa do RmlUi no MESMO frame pode silenciosamente renderizar com o programa (de tint) errado ainda vinculado. Precisa de um teste que renderize um elemento tingido imediatamente seguido de um elemento `polygon()`/gradiente/texto simples no mesmo frame e diff-cheque o resultado.
- **Suposição de estado de blend.** O design assume que o `glBlendFunc(GL_ONE, GL_ONE_MINUS_SRC_ALPHA)` ambiente do RmlUi (setado uma vez por `BeginFrame`, per o próprio comentário de cabeçalho de `render_gl3.cpp` sobre composição premultiplicada) já está corretamente vinculado quando `RenderShader` roda — a chamada de draw de tint **não** deve tocar `glBlendFunc`/`glEnable(GL_BLEND)` ela mesma. Deve valer dado que `RenderShader` só roda no meio de um frame, mas vale um assert/comentário explícito, não só uma suposição.
- **Estado de scissor/clip.** Um elemento dentro de `overflow: hidden` depende do estado GL ambiente de `glScissor`/clip-mask já estar ativo quando `RenderElement` roda (o RmlUi o aplica antes de chamar o decorator). O draw de tint não deve desabilitar/resetar o scissor — deve ser automático (nunca tocamos `GL_SCISSOR_TEST`), mas precisa de um teste com uma imagem tingida dentro de um container rolável/clipado.
- **Correção gama — pergunta em aberto, não decidida aqui (ver (d)).** Se o `base.png` real do consumidor foi autorado/exportado com uma ferramenta consciente de sRGB esperando composição em espaço linear, a recomendação de "sem correção gama" pode precisar ser revisitada — confirme visualmente contra o asset real antes de lançar, não assuma que a recomendação do ADR é final sem olhar pra textura de fato.
- **Especificidades de hardening de `image-tint-threshold`/`-color`/`-mode`** (pedido explícito do líder): `-mode` protegido primeiro pelo próprio parser `keyword` do RmlUi (keyword inválida nunca chega no C++ — cai de volta ao default/valor herdado da propriedade automaticamente), um `switch` `default: → None` de defesa-em-profundidade mantido mesmo assim (casa com a filosofia já documentada em `decorator_polygon.cpp` de "o parser RCSS é um transporte, não um portão"); `-threshold` clampado a `[0.0, 0.999]` **após** uma guarda `isfinite` (espelha exatamente o precedente de hardening NaN/Inf de `sides`/`rotation` — a classe de bug de float-cast-overflow; o limite superior é estritamente abaixo de `1.0`, não `1.0` em si — um fix pós-review da v0.7.0, ver o Adendo abaixo, já que um clamp caindo exatamente em `1.0` alimenta o `smoothstep(edge0, 1.0, L)` do shader com `edge0==edge1`, UB do GLSL por spec); `-color` não precisa de clamp extra (qualquer `Colourb` resolvido é válido); textura ausente/falha de load (`!texture` de `GetTexture`) → `InstanceDecorator` retorna `nullptr` (decorator ignorado, fail-high, igual à rejeição `sides<3`/fill-inválido do `polygon()` e ao precedente do `load(nullptr)`).
- **O desvio de nomenclatura `image-tint()` (b) precisa do aval explícito do líder** antes de a implementação começar — o `Status: Proposed` deste ADR reflete essa confirmação pendente, não uma decisão fechada.

## Alternatives considered / Alternativas consideradas

**EN:**
1. **Patch the vendored `RmlUi_Renderer_GL3.cpp` (add a case to its internal `CompileShaderType`/`ProgramId` enums).** Rejected outright — violates the project's zero-core-patch rule (`CLAUDE.md`), and would need re-applying on every future RmlUi version bump.
2. **CPU pre-tint the whole texture once at load time (extend `Gl3RenderInterface::LoadTexture`'s existing premultiply pass).** Rejected — one GL texture object is shared by every consumer of the same source path; the real use case needs **N simultaneously different tints of the same base texture on screen at once** (multiple domain-coloured cards), which a single shared pre-tinted texture cannot represent.
3. **A single bundled shorthand property (`image-tint: url(...) color mode threshold;`) instead of separate `image-tint-*` properties.** Rejected — the líder explicitly wants independently-animatable sub-values (e.g. transition only the colour, leave mode/threshold static); RmlUi shorthand interpolation is coarser-grained and would complicate that without a clear win.
4. **Keep `decorator: image(...)` literally and have `ImageTintDecorator` merely *observe* an `image` decorator already present on the element, post-processing its output.** Rejected as not expressible in RmlUi's decorator model — decorators do not see or compose over each other's render output; each decorator in the `decorator:` list draws independently into the same layer, there's no "wrap the previous decorator's result" primitive.

**PT:**
1. **Patchear o `RmlUi_Renderer_GL3.cpp` vendorizado (adicionar um case aos enums internos `CompileShaderType`/`ProgramId`).** Rejeitada de cara — viola a regra de zero-patch-de-core do projeto (`CLAUDE.md`), e precisaria ser reaplicada a cada bump futuro de versão do RmlUi.
2. **Pré-tingir a textura inteira na CPU uma vez no load (estender o passe de premultiply já existente de `Gl3RenderInterface::LoadTexture`).** Rejeitada — um objeto de textura GL é compartilhado por todo consumidor do mesmo caminho de origem; o caso de uso real precisa de **N tingimentos simultaneamente diferentes da mesma textura base na tela ao mesmo tempo** (múltiplos cards coloridos por domínio), que uma única textura pré-tingida compartilhada não consegue representar.
3. **Uma única propriedade shorthand agrupada (`image-tint: url(...) cor modo threshold;`) em vez de propriedades `image-tint-*` separadas.** Rejeitada — o líder explicitamente quer sub-valores animáveis independentemente (ex.: transicionar só a cor, deixando modo/threshold estáticos); a interpolação de shorthand do RmlUi é mais grossa e complicaria isso sem ganho claro.
4. **Manter `decorator: image(...)` literalmente e ter `ImageTintDecorator` apenas *observando* um decorator `image` já presente no elemento, pós-processando sua saída.** Rejeitada como não-expressável no modelo de decorator do RmlUi — decorators não veem nem compõem sobre a saída de render um do outro; cada decorator na lista `decorator:` desenha independentemente na mesma camada, não há uma primitiva "envolva o resultado do decorator anterior".

## Reversibility / Reversibilidade

**EN:** Two-way door. Purely additive (new decorator name, new global properties, new internal `Gl3RenderInterface` override branches) — removing it later means deleting `decorator_image_tint.{hpp,cpp}`, the `"glintfx-tint"` branches in `CompileShader`/`RenderShader`/`ReleaseShader`, and the property registration, with zero impact on any existing decorator/property. The one non-trivial cost of reversal is documentation/consumer-migration if GusWorld ships RCSS using `image-tint()` before a hypothetical rollback.

**PT:** Two-way door. Puramente aditiva (nome de decorator novo, propriedades globais novas, ramos novos de override interno no `Gl3RenderInterface`) — removê-la depois significa deletar `decorator_image_tint.{hpp,cpp}`, os ramos `"glintfx-tint"` em `CompileShader`/`RenderShader`/`ReleaseShader`, e o registro de propriedade, com zero impacto em qualquer decorator/propriedade existente. O único custo não-trivial de reversão é documentação/migração do consumidor se o GusWorld já tiver RCSS usando `image-tint()` antes de um hipotético rollback.

## Addendum (2026-07-09, post-v0.7.0 review) — approximate fidelity of "preservation" / Adendo (2026-07-09, review pós-v0.7.0) — fidelidade aproximada da "preservação"

**EN:** The v0.7.0 adversarial review re-derived the fixture's own numbers and confirmed a fact that is inherent to the formula in (b)/(d) above, **not a bug**: at the *default* `image-tint-threshold: 0.55`, zone B of the test fixture (`#f0c040`, `(240,192,64)`, luminance `L≈0.75`, saturation `sat≈0.69`) does **not** come out untouched. Its tint weight is `w = smoothstep(0.55, 1.0, 0.75) × (1 − 0.69) ≈ 0.13`, a real ~13–16% shift in G/B toward the tint colour — small, but not zero. This is not a corner case; it is exactly what `w = smoothstep(threshold, 1, L) × (1 − saturation)` computes for *any* texel whose luminance clears the threshold, scaled **linearly** by `(1 − saturation)` rather than gated by a hard saturation cutoff. The design (per the líder's original brief) never specified a hard `saturation > X → fully exempt` rule — the linear `(1 − sat)` factor was always the whole mechanism, and it degrades gracefully rather than switching off.

**Practical consequence for consumers:** "preserves saturated (gold) texels" (as worded informally elsewhere in this ADR and in `docs/effects.md`) means *reduces* the tint on saturated texels *proportionally* to their saturation — not an exact, guaranteed no-op. A highly-saturated-but-not-fully-saturated colour (like this fixture's gold, `sat≈0.69`, not `1.0`) still receives a non-trivial fraction of the tint at the default threshold. Near-total preservation of a specific saturated asset requires **tuning `image-tint-threshold` above that asset's own luminance** (`L≈0.75` for this fixture's gold would need `threshold ⪆ 0.75` to floor its weight near 0 via the `smoothstep` term) — a **trade-off**, not a free lunch: raising the threshold also proportionally suppresses tinting of the *neutral* zone the effect exists to recolour (this fixture's zone A, `L=0.941`, needs `threshold` to stay comfortably below `0.941` to keep tinting it at meaningful strength). In short: **`image-tint-threshold` is a per-asset dial the consumer must tune against their own texture's actual luminance histogram, not a universal default that "just works" for preserving every saturated colour** — 0.55 is a reasonable starting point for assets shaped like this fixture, not a guarantee.

**Not implemented now — seed for a future iteration if a consumer asks:** a non-linear response curve, e.g. `(1 − sat)^k` for some `k > 1` (steepens the falloff near high saturation without a hard cutoff), or a second, independent `image-tint-saturation-threshold` property (an explicit hard gate, orthogonal to the luminance gate) for consumers who need near-guaranteed preservation of a specific saturated brand colour. Neither is implemented in v0.7.0 — this paragraph is a design note for a future ADR/PR, not a commitment. Consumer-facing usage guidance (how to pick a threshold for a real asset) is deferred to the release-prep documentation pass, per the líder's own scoping of this task.

**PT:** O review adversarial da v0.7.0 recalculou os números da própria fixture e confirmou um fato inerente à fórmula em (b)/(d) acima, **não um bug**: no `image-tint-threshold: 0.55` *default*, a zona B da fixture de teste (`#f0c040`, `(240,192,64)`, luminância `L≈0.75`, saturação `sat≈0.69`) **não** sai intocada. Seu peso de tingimento é `w = smoothstep(0.55, 1.0, 0.75) × (1 − 0.69) ≈ 0.13`, um desvio real de ~13-16% em G/B em direção à cor de tingimento — pequeno, mas não zero. Isso não é um caso de borda; é exatamente o que `w = smoothstep(threshold, 1, L) × (1 − saturação)` calcula para *qualquer* texel cuja luminância ultrapassa o threshold, escalado **linearmente** por `(1 − saturação)` em vez de protegido por um corte duro de saturação. O design (segundo o brief original do líder) nunca especificou uma regra dura tipo "`saturação > X → totalmente isento`" — o fator linear `(1 − sat)` sempre foi o mecanismo inteiro, e ele degrada graciosamente em vez de desligar.

**Consequência prática para consumidores:** "preserva texels saturados (ouro)" (como redigido informalmente em outros pontos deste ADR e em `docs/effects.md`) significa *reduz* o tingimento em texels saturados *proporcionalmente* à saturação deles — não um no-op exato e garantido. Uma cor altamente-saturada-mas-não-totalmente-saturada (como o ouro desta fixture, `sat≈0.69`, não `1.0`) ainda recebe uma fração não-trivial do tingimento no threshold default. Preservação quase-total de um asset saturado específico exige **tunar `image-tint-threshold` ACIMA da própria luminância desse asset** (`L≈0.75` pro ouro desta fixture precisaria de `threshold ⪆ 0.75` pra achatar seu peso perto de 0 via o termo `smoothstep`) — uma **troca**, não almoço grátis: subir o threshold também suprime proporcionalmente o tingimento da zona *neutra* que o efeito existe pra recolorir (a zona A desta fixture, `L=0.941`, precisa que `threshold` fique confortavelmente abaixo de `0.941` pra continuar sendo tingida com força significativa). Em resumo: **`image-tint-threshold` é um dial por-asset que o consumidor precisa tunar contra o histograma real de luminância da própria textura, não um default universal que "simplesmente funciona" pra preservar toda cor saturada** — 0.55 é um ponto de partida razoável pra assets no formato desta fixture, não uma garantia.

**Não implementado agora — semente pra uma iteração futura se um consumidor pedir:** uma curva de resposta não-linear, ex. `(1 − sat)^k` pra algum `k > 1` (aprofunda a queda perto de alta saturação sem um corte duro), ou uma segunda propriedade independente `image-tint-saturation-threshold` (um portão duro explícito, ortogonal ao portão de luminância) pra consumidores que precisam de preservação quase-garantida de uma cor de marca saturada específica. Nenhuma das duas está implementada na v0.7.0 — este parágrafo é uma nota de design pra um ADR/PR futuro, não um compromisso. A orientação de uso pro consumidor (como escolher um threshold pra um asset real) fica pro passe de documentação de release-prep, conforme o próprio escopo do líder pra esta tarefa.
