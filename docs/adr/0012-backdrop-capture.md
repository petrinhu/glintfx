# ADR-0012 — Backdrop capture: glintfx reads FBO 0, conditionally (`ripple()` decorator) / Captura de backdrop: a glintfx lê o FBO 0, condicionalmente (decorator `ripple()`)

- **Status:** Accepted (2026-07-16)
- **Deciders:** petrus (líder), software-architect
- **Tags:** architecture, api, additive, two-way-door, rendering, embed-mode
- **Relates to:** [[0008-embed-guest-mode]] (narrows the compose-only guarantee, clause (d), for one opt-in case), [[0010-image-tint-luminance-key]] (same `Gl3RenderInterface` override shape, same zero-core-patch constraint). `TODO.md` items `L1.22-CAPTURE` (capture) and `L1.22-WAVE` (decorator).

## Context / Contexto

**EN:** [ADR-0008](0008-embed-guest-mode.md) established the embed/guest mode's central guarantee: `UiLayer::render()` composes the RmlUi-drawn UI **on top of** the host's own FBO 0 -- it never reads it back as a texture, never clears it, never swaps it. `docs/effects.md`'s own header states a second, independent principle: every glintfx effect is **RCSS-declarative**; there is no imperative effect API in C++.

GusWorld (the first embed-mode consumer) asked, on 2026-07-15, for a screen-space "time distortion ripple" that refracts the HOST's OWN scene (the battle scene behind the cockpit UI), radiating out from a resolved action. At the time, `docs/embed-integration.md` section 19 analysed this as **structurally impossible inside glintfx**: compose-only mode never reads the host's framebuffer back, so there is no texture inside glintfx for a ripple to sample from; and an imperative `emit_vfx()` API would both violate the no-imperative-effect principle and still not solve the missing-texture problem by itself. glintfx shipped only host-side hooks instead (event-callback timing, `get_element_box` epicentre, a reference GLSL shader for the host's own post-processing pass) -- confirmed sufficient by the GusWorld dev on 2026-07-15, who retracted the `emit_vfx()` request.

`TODO.md`'s `L1.22-WAVE` entry logged a **dormant seed** at the same time ("Opção B"): capture FBO 0 into an in-library texture and refract it through a `ripple` decorator, using the same `RenderShader`-override pattern [ADR-0010](0010-image-tint-luminance-key.md) already established for `image-tint()`. That entry explicitly scoped the seed as refracting **only glintfx's own composed content** (cockpit panels), *not* the host's underlying scene -- "puxar só se o líder quiser um shimmer sobre a UI glintfx" ("pull only if the líder wants a shimmer over the glintfx UI").

**What actually got built (`feat/backdrop-ripple`, commit `647350f`) exceeds that original scope.** `Gl3RenderInterface::CaptureBackdrop` runs at the very TOP of `RenderGl3::begin_frame_compose` -- BEFORE `SetViewport()`/`BeginFrame()` run, i.e. while FBO 0 still holds whatever the HOST already drew this frame (confirmed by reading the pinned `RmlUi_Renderer_GL3.cpp`: `BeginFrame()` only ever touches RmlUi's own private render-layer FBOs, never FBO 0; FBO 0 is only rebound again inside `EndFrame()`, for the final blit). The captured texture is therefore **the host's own underlying scene**, not "glintfx's own content" as Option B assumed -- exactly the capability section 19's original analysis said was structurally impossible. This ADR exists because that gap between the scoped seed and the actual result is significant enough to need an explicit decision record, not because the feature was unplanned.

**PT:** O [ADR-0008](0008-embed-guest-mode.md) estabeleceu a garantia central do embed/guest mode: `UiLayer::render()` compõe a UI desenhada pelo RmlUi **por cima** do próprio FBO 0 do host -- nunca o lê de volta como textura, nunca o limpa, nunca o troca. O próprio cabeçalho de `docs/effects.md` declara um segundo princípio independente: todo efeito da glintfx é **RCSS-declarativo**; não há API imperativa de efeito em C++.

O GusWorld (primeiro consumidor de embed mode) pediu, em 2026-07-15, uma "onda de distorção de tempo" screen-space que refratasse a própria CENA do host (a cena de batalha atrás da UI do cockpit), irradiando a partir de uma ação resolvida. Na época, a seção 19 de `docs/embed-integration.md` analisou isso como **estruturalmente impossível dentro da glintfx**: o modo compose-only nunca lê de volta o framebuffer do host, então não há textura nenhuma dentro da glintfx pra um ripple amostrar; e uma API imperativa `emit_vfx()` violaria o princípio de sem-efeito-imperativo E ainda não resolveria sozinha o problema da textura ausente. A glintfx distribuiu só ganchos host-side em vez disso (timing via callback de evento, epicentro via `get_element_box`, um shader GLSL de referência pro próprio passe de pós-processamento do host) -- confirmado suficiente pelo dev do GusWorld em 2026-07-15, que retirou o pedido de `emit_vfx()`.

A entrada `L1.22-WAVE` do `TODO.md` registrou uma **semente dormante** ao mesmo tempo ("Opção B"): capturar o FBO 0 numa textura in-library e refratá-la através de um decorator `ripple`, usando o mesmo padrão de override de `RenderShader` que o [ADR-0010](0010-image-tint-luminance-key.md) já estabeleceu pro `image-tint()`. Aquela entrada escopou explicitamente a semente como refratando **só o próprio conteúdo composto da glintfx** (painéis do cockpit), *não* a cena subjacente do host -- "puxar só se o líder quiser um shimmer sobre a UI glintfx".

**O que de fato foi construído (`feat/backdrop-ripple`, commit `647350f`) excede esse escopo original.** `Gl3RenderInterface::CaptureBackdrop` roda bem no TOPO de `RenderGl3::begin_frame_compose` -- ANTES de `SetViewport()`/`BeginFrame()` rodarem, isto é, enquanto o FBO 0 ainda tem o que o HOST já desenhou neste frame (confirmado lendo o `RmlUi_Renderer_GL3.cpp` pinado: o `BeginFrame()` só toca os próprios FBOs privados de render-layer do RmlUi, nunca o FBO 0; o FBO 0 só é vinculado de novo dentro do `EndFrame()`, pro blit final). A textura capturada é, portanto, **a própria cena subjacente do host**, não "o próprio conteúdo da glintfx" como a Opção B assumia -- exatamente a capacidade que a análise original da seção 19 dizia ser estruturalmente impossível. Este ADR existe porque essa distância entre a semente escopada e o resultado real é grande o bastante pra precisar de um registro de decisão explícito, não porque a feature não era planejada.

## Decision / Decisão

**EN:**

**(a) Conditional backdrop read, gated to a real zero cost when unused.** `Gl3RenderInterface::CaptureBackdrop` runs at the top of `RenderGl3::begin_frame_compose`, reading FBO 0's sub-region (`offset_x, offset_y, w, h` -- the exact viewport rectangle `set_viewport` configures) via `glCopyTexSubImage2D` into a glintfx-owned `GL_RGBA8`/`GL_LINEAR`/`GL_CLAMP_TO_EDGE` texture, reallocated only when the captured size changes. The very first thing it does is check `*ripple_active_counter_ > 0` -- an active-instance counter owned by `RippleDecoratorInstancer`, incremented in `GenerateElementData`'s success path and decremented in `ReleaseElementData` (shared across every `ripple()` rule in the document, since the question `CaptureBackdrop` needs answered is "is ANY ripple element alive right now", not per-rule). When the counter is zero, `CaptureBackdrop` returns with **zero** GL calls -- no allocation, no `glCopyTexSubImage2D`, nothing. A host that never loads RCSS using `ripple()` pays exactly the same per-frame cost it paid before this feature existed.

**(b) Same non-invasive override shape as `image-tint()` (ADR-0010), zero core patch.** `CaptureBackdrop` and the new `"glintfx-ripple"`-tagged `CompileShader`/`RenderShader`/`ReleaseShader` branches live inside `Gl3RenderInterface` (glintfx's own subclass of `RenderInterface_GL3`), reached exclusively through RmlUi's public API surface -- the vendored RmlUi source (`examples/RmlUi`, `FetchContent`-pinned) is untouched, same non-negotiable constraint ADR-0010 established. `RippleDecorator` carries **no** `Rml::Texture` at all (unlike `image-tint()`): the sampled "texture" is the captured backdrop, bound directly inside `RenderShader`'s ripple branch. `glReadBuffer`/`glBindFramebuffer(GL_READ_FRAMEBUFFER, ...)` inside `CaptureBackdrop` are ordinary public GL calls, not RmlUi API at all -- no new access to RmlUi internals was needed for the capture itself, only for wiring the counter (`RippleDecoratorInstancer::active_instance_count_ptr()`, connected by `bootstrap.cpp` right after the instancer's construction+registration, same lifetime rule as `polygon_instancer`/`tint_instancer`).

**(c) `GlStateGuard` restores the read-framebuffer state, not `CaptureBackdrop` itself.** `CaptureBackdrop` rebinds `GL_READ_FRAMEBUFFER` to `0` and calls `glReadBuffer(GL_BACK)` but does **not** restore either locally -- a local restore would be pointless, since RmlUi's own `EndFrame()` unconditionally rebinds FBO 0 for both read and draw targets partway through `end_frame_compose` regardless (`glBindFramebuffer(GL_FRAMEBUFFER, 0)`, confirmed in the pinned `RmlUi_Renderer_GL3.cpp`), clobbering any local save/restore moments later. `GlStateGuard`'s constructor runs BEFORE `begin_frame_compose`/`CaptureBackdrop`, and its destructor runs AFTER `end_frame_compose`/`EndFrame()` -- the only scope wide enough to recover the host's TRUE original read-framebuffer binding. `gl_state.hpp` now also snapshots/restores `GL_READ_FRAMEBUFFER_BINDING` and `GL_READ_BUFFER` (in addition to the state it already tracked -- see section 2's table in `docs/embed-integration.md`).

**(d) The compose-only invariant is narrowed to a documented, opt-in exception -- not dropped.** ADR-0008's guarantee ("`UiLayer` never reads the host's own framebuffer back") holds **unconditionally** for any host that never uses `ripple()` -- for those, behaviour, cost, and the invariant's literal wording are all unchanged (the gate in (a) is a real zero, not merely "small"). A host that opts into `decorator: ripple(...)` on any element explicitly accepts that glintfx will read a sub-region of FBO 0 once per frame while that decorator is alive, exactly as documented in `docs/embed-integration.md` section 20. This is a **per-element RCSS opt-in**, not a global mode switch, and is the reason this ADR amends rather than supersedes ADR-0008.

**PT:**

**(a) Leitura condicional de backdrop, com gate a custo real zero quando não usada.** `Gl3RenderInterface::CaptureBackdrop` roda no topo de `RenderGl3::begin_frame_compose`, lendo a sub-região do FBO 0 (`offset_x, offset_y, w, h` -- o retângulo de viewport exato que `set_viewport` configura) via `glCopyTexSubImage2D` numa textura própria da glintfx `GL_RGBA8`/`GL_LINEAR`/`GL_CLAMP_TO_EDGE`, realocada só quando o tamanho capturado muda. A primeiríssima coisa que faz é checar `*ripple_active_counter_ > 0` -- um contador de instância ativa, dono do `RippleDecoratorInstancer`, incrementado no caminho de sucesso de `GenerateElementData` e decrementado em `ReleaseElementData` (compartilhado entre toda regra `ripple()` no documento, já que a pergunta que `CaptureBackdrop` precisa responder é "existe QUALQUER elemento ripple vivo agora", não por-regra). Quando o contador é zero, `CaptureBackdrop` retorna com **zero** chamadas GL -- nenhuma alocação, nenhum `glCopyTexSubImage2D`, nada. Um host que nunca carrega RCSS usando `ripple()` paga exatamente o mesmo custo por-frame que pagava antes desta feature existir.

**(b) Mesma forma de override não-invasiva do `image-tint()` (ADR-0010), zero patch de core.** `CaptureBackdrop` e os novos ramos de `CompileShader`/`RenderShader`/`ReleaseShader` marcados `"glintfx-ripple"` vivem dentro do `Gl3RenderInterface` (subclasse própria da glintfx de `RenderInterface_GL3`), alcançados exclusivamente pela superfície pública de API do RmlUi -- o source vendorizado do RmlUi (`examples/RmlUi`, pinado via `FetchContent`) fica intocado, mesma restrição inegociável que o ADR-0010 estabeleceu. `RippleDecorator` não carrega **nenhuma** `Rml::Texture` (diferente do `image-tint()`): a "textura" amostrada é o backdrop capturado, vinculado diretamente dentro do ramo de ripple de `RenderShader`. `glReadBuffer`/`glBindFramebuffer(GL_READ_FRAMEBUFFER, ...)` dentro de `CaptureBackdrop` são chamadas GL públicas comuns, não API do RmlUi nenhuma -- nenhum acesso novo a internals do RmlUi foi necessário pra captura em si, só pra conectar o contador (`RippleDecoratorInstancer::active_instance_count_ptr()`, conectado por `bootstrap.cpp` logo após construção+registro do instancer, mesma regra de lifetime de `polygon_instancer`/`tint_instancer`).

**(c) O `GlStateGuard` restaura o estado de read-framebuffer, não o próprio `CaptureBackdrop`.** `CaptureBackdrop` revincula `GL_READ_FRAMEBUFFER` a `0` e chama `glReadBuffer(GL_BACK)` mas **não** restaura nenhum dos dois localmente -- uma restauração local seria inútil, já que o próprio `EndFrame()` do RmlUi revincula o FBO 0 incondicionalmente pros alvos de read e draw no meio do `end_frame_compose` de qualquer jeito (`glBindFramebuffer(GL_FRAMEBUFFER, 0)`, confirmado no `RmlUi_Renderer_GL3.cpp` pinado), sobrepondo qualquer save/restore local momentos depois. O construtor do `GlStateGuard` roda ANTES de `begin_frame_compose`/`CaptureBackdrop`, e o destrutor dele roda DEPOIS de `end_frame_compose`/`EndFrame()` -- o único escopo largo o bastante pra recuperar o binding de read-framebuffer ORIGINAL VERDADEIRO do host. `gl_state.hpp` agora também tira snapshot/restaura `GL_READ_FRAMEBUFFER_BINDING` e `GL_READ_BUFFER` (além do estado que já rastreava -- ver a tabela da seção 2 em `docs/embed-integration.md`).

**(d) O invariante compose-only é estreitado para uma exceção documentada, opt-in -- não abandonado.** A garantia do ADR-0008 ("`UiLayer` nunca lê de volta o próprio framebuffer do host") vale **incondicionalmente** pra todo host que nunca usa `ripple()` -- pra esses, comportamento, custo e a própria redação literal do invariante ficam inalterados (o gate de (a) é um zero real, não só "pequeno"). Um host que opta por `decorator: ripple(...)` em qualquer elemento aceita explicitamente que a glintfx vai ler uma sub-região do FBO 0 uma vez por frame enquanto aquele decorator estiver vivo, exatamente como documentado na seção 20 de `docs/embed-integration.md`. Isto é um **opt-in RCSS por-elemento**, não uma troca de modo global, e é o motivo deste ADR emendar em vez de suceder o ADR-0008.

## Consequences / Consequências

**EN:**

**Positive:**
- Delivers the actual consumer ask (refract the host's live scene) declaratively (RCSS decorator + properties), no `emit_vfx()`, preserving the "no imperative effect API" principle stated in `docs/effects.md`.
- Cost-zero-when-inactive gate means every existing host (GusWorld's current cockpit-only integration, any future UI-only `App` user) pays nothing extra unless it opts in.
- Reuses the audited `image-tint()`/ADR-0010 override shape end-to-end (same `GenerateElementData`/`RenderElement` discipline, same tagged-wrapper `CompileShader`/`RenderShader`/`ReleaseShader` dispatch, same `GlStateGuard`-extension precedent this project already set for prior FBO-touching work).
- Fixes a GLSL-UB class of bug proactively during implementation ("auditoria dominó" against `image-tint-threshold`'s precedent, ADR-0010): the naive `smoothstep(width, 0.0, ...)` ring formula has `edge0 > edge1`, which is GLSL UB; rewritten as `1.0 - smoothstep(0.0, width, ...)`, plus a division-by-zero guard replacing `normalize()` at the ring's exact origin.

**Negative / accepted as cost:**
- ADR-0008's compose-only guarantee is no longer an absolute for the library as a whole -- it is conditional on RCSS content. Any future audit or consumer relying on the literal, unqualified wording "`UiLayer` never reads the host's framebuffer back" (sections 0/2 of `docs/embed-integration.md`, pre-this-ADR) needs the section 20 update.
- Frame ordering (host draws its scene into FBO 0 **before** `render()`) was already mandatory for the composited UI to look correct; with `ripple()` active, violating it now feeds the ripple shader stale/undefined FBO 0 content, not just a wrong-looking overlay -- same root-cause bug (host called `render()` too early), a subtler symptom to diagnose.
- Read-framebuffer state must now be tracked through the ENTIRE `render_compose` scope via `GlStateGuard`, not locally at the capture call site -- a non-obvious invariant (documented explicitly in `gl_state.hpp` and here) a future maintainer extending either file must preserve.
- Section 19 of `docs/embed-integration.md` (the host-side reference-shader pattern) is now partially superseded for the common case -- kept, not deleted, since it remains the right tool for hosts needing a fully custom pipeline, but the document now carries two related-but-different patterns side by side, a documentation-maintenance cost.

**Risks / points of attention (validated during implementation, per the líder's "auditoria-dominó" standard):**
- **Y-flip mismatch, found via manual GPU verification (not in the original task spec).** `fragTexCoord` is RmlUi's own document-space convention (origin top-left, y-down); the captured backdrop texture is populated straight from the OpenGL-native backbuffer (origin bottom-left, y-up) -- the same convention mismatch every pixel-reading test in this repo already compensates for (e.g. `tests/image_tint_sanity.cpp`'s `pixel_at()` `row = h - 1 - content_y`). Sampling with the raw (un-flipped) V read the backdrop vertically mirrored; fixed in the shader's final texture lookup only (`vec2(uv.x, 1.0 - uv.y)`) -- the ring/offset math upstream of that lookup intentionally stays in document-space, since `ripple-origin-y` is an author-facing property that must keep ordinary top-down semantics.
- **`ripple-width` hardening.** Floored to a small positive epsilon (`0.0001f`) on the CPU side (`ResolveRippleState`, `decorator_ripple.cpp`) in addition to the shader-side rewrite in (Positive) above -- belt-and-suspenders, same "clamp, don't reject" philosophy `ResolveTintState` documents for `image-tint-threshold`.
- **Verification performed so far is manual, not an automated test.** Confirmed via Xvfb + a two-colour-halves render + diff scan: capture only occurs when a ripple is active, refraction crosses the colour boundary correctly post-Y-fix, alpha=ring stays localized (~0.17% of pixels in the test scenario), no garbage colour leaks through. **An automated pixel-readback test (mirroring `echo_sanity`'s or `image_tint_sanity`'s own precedent) is still owed** before this feature is considered release-ready -- flagged here so it is not lost, per `TODO.md`'s own `🔍 Pendente verificação` convention for this item.

**PT:**

**Positivas:**
- Entrega o pedido real do consumidor (refratar a cena viva do host) declarativamente (decorator RCSS + propriedades), sem `emit_vfx()`, preservando o princípio "sem API imperativa de efeito" declarado em `docs/effects.md`.
- O gate de custo-zero-quando-inativo significa que todo host existente (integração atual só-cockpit do GusWorld, qualquer futuro usuário `App` só-de-UI) não paga nada extra a menos que opte por usar.
- Reusa a forma de override auditada de `image-tint()`/ADR-0010 de ponta a ponta (mesma disciplina `GenerateElementData`/`RenderElement`, mesmo despacho por wrapper marcado de `CompileShader`/`RenderShader`/`ReleaseShader`, mesmo precedente de extensão do `GlStateGuard` que este projeto já estabeleceu pra trabalho anterior que tocava o FBO).
- Corrige proativamente durante a implementação uma classe de bug de UB do GLSL ("auditoria dominó" contra o precedente do `image-tint-threshold`, ADR-0010): a fórmula ingênua do anel `smoothstep(width, 0.0, ...)` tem `edge0 > edge1`, UB do GLSL; reescrita como `1.0 - smoothstep(0.0, width, ...)`, mais uma guarda de divisão por zero substituindo `normalize()` bem na origem do anel.

**Negativas / aceitas como custo:**
- A garantia compose-only do ADR-0008 deixa de ser um absoluto pra lib como um todo -- passa a ser condicional ao conteúdo RCSS. Qualquer auditoria ou consumidor futuro que confie na redação literal, não-qualificada, "`UiLayer` nunca lê o framebuffer do host de volta" (seções 0/2 de `docs/embed-integration.md`, pré-este-ADR) precisa da atualização da seção 20.
- A ordem de frame (o host desenha a cena no FBO 0 **antes** do `render()`) já era obrigatória pra UI composta parecer correta; com `ripple()` ativo, violá-la agora alimenta o shader de ripple com conteúdo obsoleto/indefinido do FBO 0, não só uma sobreposição com aparência errada -- mesma causa raiz de bug (o host chamou `render()` cedo demais), um sintoma mais sutil de diagnosticar.
- O estado de read-framebuffer agora precisa ser rastreado por TODO o escopo de `render_compose` via `GlStateGuard`, não localmente no call-site da captura -- um invariante não-óbvio (documentado explicitamente em `gl_state.hpp` e aqui) que um futuro mantenedor estendendo qualquer um dos dois arquivos precisa preservar.
- A seção 19 de `docs/embed-integration.md` (o padrão de shader de referência host-side) fica parcialmente superada pro caso comum -- mantida, não deletada, já que continua sendo a ferramenta certa pra hosts que precisam de um pipeline totalmente customizado, mas o documento agora carrega dois padrões relacionados-mas-diferentes lado a lado, um custo de manutenção de documentação.

**Riscos / pontos de atenção (validados durante a implementação, conforme o padrão de "auditoria-dominó" do líder):**
- **Descompasso de flip vertical, achado por verificação manual de GPU (não estava na spec original da tarefa).** `fragTexCoord` é a convenção document-space do próprio RmlUi (origem superior-esquerda, y pra baixo); a textura de backdrop capturada é preenchida direto do backbuffer nativo do OpenGL (origem inferior-esquerda, y pra cima) -- o mesmo descompasso de convenção que todo teste de leitura de pixel neste repo já compensa (ex.: `pixel_at()` de `tests/image_tint_sanity.cpp`, `row = h - 1 - content_y`). Amostrar com o V cru (sem flip) lia o backdrop espelhado verticalmente; corrigido só na busca final de textura do shader (`vec2(uv.x, 1.0 - uv.y)`) -- a matemática de anel/deslocamento anterior a essa busca permanece intencionalmente em document-space, já que `ripple-origin-y` é uma propriedade voltada ao autor e precisa manter a semântica comum de cima-pra-baixo.
- **Hardening de `ripple-width`.** Clampado por baixo a um epsilon positivo pequeno (`0.0001f`) do lado da CPU (`ResolveRippleState`, `decorator_ripple.cpp`) além da reescrita do lado do shader em (Positivas) acima -- cinto-e-suspensório, mesma filosofia "clampa, não rejeita" que `ResolveTintState` documenta pro `image-tint-threshold`.
- **A verificação feita até agora é manual, não um teste automatizado.** Confirmado via Xvfb + render de duas metades coloridas + diff scan: a captura só ocorre quando um ripple está ativo, a refração cruza a borda de cor corretamente pós-fix-de-Y, alpha=ring fica localizado (~0.17% dos pixels no cenário de teste), sem vazamento de cor "garbage". **Um teste automatizado de pixel-readback (espelhando o precedente do próprio `echo_sanity`/`image_tint_sanity`) ainda é devido** antes desta feature ser considerada pronta pra release -- sinalizado aqui pra não se perder, conforme a própria convenção `🔍 Pendente verificação` do `TODO.md` pra este item.

## Options considered / Opções consideradas

**EN:**
1. **Keep host-side only** (status quo before this branch, section 19's originally "confirmed" resolution). Rejected for this feature: the GusWorld dev's 2026-07-15 confirmation was of the *host-side hooks* being sufficient for the scope known AT THAT TIME, not a rejection of an in-library capture ever landing -- `TODO.md` explicitly logged Option B as a dormant seed for exactly this case. Kept as the still-valid alternative for hosts that need a fully custom shader pipeline, a VFX on a render target glintfx never sees, or effect ordering `ripple()`'s single-decorator RCSS model cannot express (section 19, updated).
2. **Option B exactly as scoped in `TODO.md` (capture only glintfx's OWN composed content, not the host's).** Rejected implicitly by *when* the capture actually has to run to reuse `begin_frame_compose`'s existing `(offset_x, offset_y, w, h)` cheaply: capturing before RmlUi draws (i.e. capturing the host's scene) required no extra compose pass and was simpler AND strictly more useful than a second, redundant "capture glintfx's own output" pass would have been -- there was no cost difference that justified deliberately capturing LESS than what the exact same `glCopyTexSubImage2D` call already reads at that point in the frame.
3. **`emit_vfx()` imperative API (the original consumer ask, retracted 2026-07-15).** Rejected per the existing "no imperative effect API" principle (`docs/effects.md`'s header) and because it does not solve the "no texture to sample" problem by itself -- an imperative call still needs SOME captured backdrop to refract. `L1.22-CAPTURE` solves that half of the problem regardless of whether the effect's trigger ends up imperative or declarative; once the capture existed, the declarative `ripple()` decorator was the natural, principle-consistent way to expose it.

**PT:**
1. **Manter só host-side** (status quo antes desta branch, a resolução originalmente "confirmada" da seção 19). Rejeitada pra esta feature: a confirmação do dev do GusWorld em 2026-07-15 foi de que os *ganchos host-side* eram suficientes pro escopo conhecido NAQUELE MOMENTO, não uma rejeição de uma captura in-library algum dia pousar -- o `TODO.md` registrou explicitamente a Opção B como semente dormante exatamente pra este caso. Mantida como a alternativa ainda válida pra hosts que precisam de um pipeline de shader totalmente customizado, um VFX num render target que a glintfx nunca vê, ou uma ordenação de efeito que o modelo RCSS de decorator único do `ripple()` não consegue expressar (seção 19, atualizada).
2. **Opção B exatamente como escopada no `TODO.md` (capturar só o PRÓPRIO conteúdo composto da glintfx, não o do host).** Rejeitada implicitamente por *quando* a captura de fato precisa rodar pra reusar barato o `(offset_x, offset_y, w, h)` já existente de `begin_frame_compose`: capturar antes do RmlUi desenhar (isto é, capturar a cena do host) não exigiu passe de composição extra nenhum e foi mais simples E estritamente mais útil do que um segundo passe redundante de "capturar a própria saída da glintfx" teria sido -- não havia diferença de custo que justificasse capturar deliberadamente MENOS do que a mesmíssima chamada `glCopyTexSubImage2D` já lê naquele ponto do frame.
3. **API imperativa `emit_vfx()` (o pedido original do consumidor, retirado em 2026-07-15).** Rejeitada pelo princípio já existente de "sem API imperativa de efeito" (cabeçalho de `docs/effects.md`) e porque ela não resolve sozinha o problema de "nenhuma textura pra amostrar" -- uma chamada imperativa ainda precisaria de ALGUM backdrop capturado pra refratar. O `L1.22-CAPTURE` resolve essa metade do problema independente do disparo do efeito acabar sendo imperativo ou declarativo; uma vez que a captura existia, o decorator declarativo `ripple()` foi o jeito natural, consistente com o princípio, de expô-la.

## Reversibility / Reversibilidade

**EN:** Two-way door. Purely additive (new decorator name, new global properties, new internal `Gl3RenderInterface` override branches, two new fields on `GlStateGuard`) -- removing it later means deleting `decorator_ripple.{hpp,cpp}`, `CaptureBackdrop` + the `"glintfx-ripple"` branches in `CompileShader`/`RenderShader`/`ReleaseShader`, the `ripple-*` property registrations, and the `GL_READ_FRAMEBUFFER_BINDING`/`GL_READ_BUFFER` fields from `GlStateGuard`, with zero impact on any existing decorator/property/host that never used `ripple()`. The one non-trivial cost of reversal is the same documentation/consumer-migration caveat ADR-0010 already names -- if GusWorld ships RCSS using `ripple()` before a hypothetical rollback -- plus retracting this ADR's amendment to ADR-0008's compose-only guarantee (section 20 of `docs/embed-integration.md` and the section 0/2 addenda would need to revert alongside the code).

**PT:** Two-way door. Puramente aditiva (nome de decorator novo, propriedades globais novas, ramos novos de override interno no `Gl3RenderInterface`, dois campos novos no `GlStateGuard`) -- removê-la depois significa deletar `decorator_ripple.{hpp,cpp}`, `CaptureBackdrop` + os ramos `"glintfx-ripple"` em `CompileShader`/`RenderShader`/`ReleaseShader`, o registro das propriedades `ripple-*`, e os campos `GL_READ_FRAMEBUFFER_BINDING`/`GL_READ_BUFFER` do `GlStateGuard`, com zero impacto em qualquer decorator/propriedade/host que nunca usou `ripple()`. O único custo não-trivial de reversão é a mesma ressalva de documentação/migração de consumidor que o ADR-0010 já nomeia -- se o GusWorld já tiver RCSS usando `ripple()` antes de um hipotético rollback -- mais retirar a emenda deste ADR à garantia compose-only do ADR-0008 (a seção 20 de `docs/embed-integration.md` e os adendos das seções 0/2 precisariam reverter junto do código).
