# ADR-0013 — GL loader symbol boundary: `glx_` prefix on every exported pointer / Fronteira de símbolo do loader GL: prefixo `glx_` em todo ponteiro exportado

- **Status:** Accepted (2026-07-19)
- **Deciders:** petrus (líder), software-architect
- **Tags:** security, hardening, api, additive, two-way-door, embed-mode
- **Relates to:** [[0008-embed-guest-mode]] (the exposure surfaces exactly for embed-mode hosts that link `libglintfx.a` directly), [[0009-internalization-boundary]] (the `glx_` prefix follows the same `libcore.a`/`glx_` naming precedent that ADR-0009 established for the Camada 0 static-link boundary). `TODO.md` item `AUD-L1-GLSYM`.

## Context / Contexto

**EN:** glintfx's own GL 3.3 core-profile function-pointer loader (`gl_loader.h`/`.c`, generated by `tools/gen_glloader.py`, see [ADR pending L1.14-GLLOADER]) declared one pointer variable per GL command, named **identically** to the GL function it stands in for -- e.g. `extern PFNGLCLEARPROC glClear;`. This let existing call sites (`glClear(...)`) keep compiling unchanged once `glx_gl_load()` populated the pointers, which was the loader's whole design goal.

An audit (`AUD-L1-GLSYM`) found the consequence nobody had checked for: those 344 pointer variables are ordinary **global data symbols**, and every one of them ends up in `libglintfx.a`'s symbol table with the exact name of a real GL entry point (`nm libglintfx.a | grep 'B glClearColor'` -> `B glClearColor`, defined in `gl_loader.c.o`). `libglintfx.a` is a **static** archive; when a consuming host's own object files reference `glClear` (a completely ordinary thing for any GL-calling host to do, e.g. GusWorld's SDL3 embed integration under `GLINTFX_BACKEND_GLFW=OFF`) and the link line resolves `libglintfx.a` *before* `libGL.so`, the linker satisfies the host's `glClear` reference against glintfx's **uninitialised BSS pointer slot**, not the driver's real function. The result is a call through a null function pointer -- `SIGSEGV` on the very first GL call the host makes, with a crash signature that looks nothing like its actual cause. This is a boundary leak `tools/check_encapsulation.sh` structurally cannot catch: that script greps `#include` lines for third-party headers leaking into `glintfx/include/glintfx/`; a same-named **symbol** collision at link time is invisible to a text grep of includes.

**PT:** O loader próprio de ponteiros de função OpenGL 3.3 core profile da glintfx (`gl_loader.h`/`.c`, gerado por `tools/gen_glloader.py`, ver ADR pendente L1.14-GLLOADER) declarava uma variável-ponteiro por comando GL, com o nome **idêntico** ao da função GL que substituía -- ex.: `extern PFNGLCLEARPROC glClear;`. Isso deixava os call sites existentes (`glClear(...)`) compilando sem mudança assim que `glx_gl_load()` preenchia os ponteiros, que era o objetivo de design inteiro do loader.

Uma auditoria (`AUD-L1-GLSYM`) achou a consequência que ninguém tinha checado: essas 344 variáveis-ponteiro são símbolos de dado **globais** comuns, e cada uma acaba na tabela de símbolos do `libglintfx.a` com o nome exato de um entry point GL real (`nm libglintfx.a | grep 'B glClearColor'` -> `B glClearColor`, definido em `gl_loader.c.o`). O `libglintfx.a` é um archive **estático**; quando os próprios objetos de um host consumidor referenciam `glClear` (algo completamente comum pra qualquer host que chame GL, ex.: a integração embed SDL3 do GusWorld sob `GLINTFX_BACKEND_GLFW=OFF`) e a linha de link resolve `libglintfx.a` *antes* de `libGL.so`, o linker satisfaz a referência `glClear` do host contra o slot de ponteiro **BSS não-inicializado** da glintfx, não a função real do driver. O resultado é uma chamada através de ponteiro de função nulo -- `SIGSEGV` na primeiríssima chamada GL que o host faz, com uma assinatura de crash que não se parece em nada com a causa real. É um vazamento de fronteira que o `tools/check_encapsulation.sh` estruturalmente não consegue pegar: aquele script faz grep de linhas `#include` de header de terceiro vazando pra `glintfx/include/glintfx/`; uma colisão de **símbolo** de mesmo nome em tempo de link é invisível a um grep de texto de includes.

## Decision / Decisão

**EN:**

**(a) Every exported pointer variable is renamed `glx_<cmd>` (e.g. `glx_glClear`).** `tools/gen_glloader.py` emits `extern PFN<CMD>PROC glx_<cmd>;` in `gl_loader.h` and `PFN<CMD>PROC glx_<cmd> = NULL;` in `gl_loader.c` -- the generator writes the prefixed identifier directly wherever it defines a symbol; it never emits a plain-named data symbol, not even transiently. `nm libglintfx.a` now shows only `glx_gl*` data symbols, zero bare `gl*` ones.

**(b) A second, separate `#define <cmd> glx_<cmd>` block, emitted only after every `extern` declaration (and after the `glcorearb.h` include), keeps glintfx's OWN internal call sites unchanged.** `gl_loader.h` is private (`glintfx/src/`, never installed under `glintfx/include/glintfx/`), so this macro never reaches a consumer's translation unit -- it exists purely so `render_gl3.cpp`'s `glClear(...)` etc. keep reading and compiling exactly as before, without a single call-site edit. The macro renames whole identifier tokens only (C preprocessor semantics); it cannot and does not touch the `PFN<CMD>PROC` typedef tokens (a different identifier entirely) or string literals (the GetProcAddress lookup key in `gl_loader.c`'s symbol table stays the bare `"glClear"`, which is the real name the driver exports).

**(c) The symbol table in `gl_loader.c` keys on the bare GL name, values on the `glx_`-prefixed slot.** `{ "glClear", (void**)&glx_glClear }` -- `glx_gl_load()`'s `GetProcAddress`/`dlsym` resolution logic is otherwise untouched; only the variable it writes into changed name.

**PT:**

**(a) Toda variável-ponteiro exportada é renomeada pra `glx_<cmd>` (ex.: `glx_glClear`).** `tools/gen_glloader.py` emite `extern PFN<CMD>PROC glx_<cmd>;` em `gl_loader.h` e `PFN<CMD>PROC glx_<cmd> = NULL;` em `gl_loader.c` -- o gerador escreve o identificador prefixado diretamente onde quer que defina um símbolo; nunca emite um símbolo de dado com nome cru, nem transitoriamente. `nm libglintfx.a` agora mostra só símbolos de dado `glx_gl*`, zero `gl*` crus.

**(b) Um segundo bloco, separado, `#define <cmd> glx_<cmd>`, emitido só depois de toda declaração `extern` (e depois do include de `glcorearb.h`), mantém os call sites internos, PRÓPRIOS da glintfx, inalterados.** `gl_loader.h` é privado (`glintfx/src/`, nunca instalado sob `glintfx/include/glintfx/`), então esta macro nunca alcança a unidade de tradução de um consumidor -- ela existe só pra que o `glClear(...)` etc. de `render_gl3.cpp` continue lendo e compilando exatamente como antes, sem editar um único call site. A macro renomeia só tokens de identificador inteiros (semântica do pré-processador C); ela não pode e não toca os tokens de typedef `PFN<CMD>PROC` (um identificador completamente diferente) nem literais de string (a chave de busca do GetProcAddress na tabela de símbolos de `gl_loader.c` permanece o `"glClear"` cru, que é o nome real que o driver exporta).

**(c) A tabela de símbolos em `gl_loader.c` é indexada pelo nome GL cru, com valores no slot prefixado com `glx_`.** `{ "glClear", (void**)&glx_glClear }` -- a lógica de resolução `GetProcAddress`/`dlsym` de `glx_gl_load()` fica intocada; só o nome da variável em que ela escreve mudou.

## Consequences / Consequências

**EN:**

**Positive:**
- Kills a real, host-triggerable `SIGSEGV`-on-first-GL-call bug class for any embed consumer that links `libglintfx.a` before `libGL.so` -- confirmed by the exact `nm` reproduction above, not hypothetical.
- Zero call-site edits across the entire codebase (`app.cpp`, `render_gl3.cpp`, `gl_state.hpp`, `decorator_*.cpp`, every test that calls raw GL) -- the `#define` block absorbs the entire rename.
- `nm libglintfx.a | grep -E ' [BbDd] gl[A-Z]'` is now a permanent, cheap CI-able regression check (zero matches expected) for this exact class of symbol leak, the same way `check_encapsulation.sh` already gates the include-leak class.
- Consistent with the `glx_` naming precedent [ADR-0009](0009-internalization-boundary.md) already established for the Camada 0 static-link boundary (`libcore.a`, prefix `glx_`) -- one naming convention for "this identifier crosses a link boundary and must not collide with the host's own symbols" across both layers of the repo.

**Negative / accepted as cost:**
- `tools/gen_glloader.py` is slightly more complex (two identifier-name functions instead of one, an extra emitted block) -- a fixed, one-time cost paid by the generator, not by any hand-maintained file.
- Anyone reading `gl_loader.h` for the first time sees two blocks of near-duplicate content (344 `extern` lines, then 344 `#define` lines) -- mitigated by the bilingual header comment explaining why, but still a larger generated file than the single-block version.

**Risks / points of attention:**
- The fix is a *pointer-variable* rename; it does not (and cannot) prevent a host from independently defining its own COLLIDING `glx_glClear` symbol -- but that would require the host to deliberately reuse glintfx's private internal naming convention, an astronomically unlikely accident compared to a host innocently calling `glClear` (a real GL function every GL host calls).
- The `#define` block's ordering (after `glcorearb.h`'s include, after every `extern` declaration) is load-bearing for readability and for the stated non-collision-with-typedefs reasoning, even though the C preprocessor's whole-token-only substitution semantics mean strict ordering is not, in fact, required for correctness here -- documented in the generator and the header itself so a future edit does not "simplify" the order away and lose the documented reasoning.

**PT:**

**Positivas:**
- Mata uma classe real de bug `SIGSEGV`-na-primeira-chamada-GL, disparável pelo host, pra qualquer consumidor embed que linke `libglintfx.a` antes de `libGL.so` -- confirmado pela reprodução `nm` exata acima, não hipotética.
- Zero edição de call site em todo o código-base (`app.cpp`, `render_gl3.cpp`, `gl_state.hpp`, `decorator_*.cpp`, todo teste que chama GL cru) -- o bloco `#define` absorve a renomeação inteira.
- `nm libglintfx.a | grep -E ' [BbDd] gl[A-Z]'` agora é um check de regressão permanente, barato, gateável em CI (zero matches esperados) pra exatamente esta classe de vazamento de símbolo, do mesmo jeito que `check_encapsulation.sh` já gateia a classe de vazamento de include.
- Consistente com o precedente de nomenclatura `glx_` que o [ADR-0009](0009-internalization-boundary.md) já estabeleceu pra fronteira de link estático da Camada 0 (`libcore.a`, prefixo `glx_`) -- uma convenção de nomenclatura só pra "este identificador cruza uma fronteira de link e não pode colidir com os símbolos do próprio host" nas duas camadas do repo.

**Negativas / aceitas como custo:**
- `tools/gen_glloader.py` fica levemente mais complexo (duas funções de nome de identificador em vez de uma, um bloco extra emitido) -- um custo fixo, pago uma vez, pelo gerador, não por nenhum arquivo mantido à mão.
- Quem lê `gl_loader.h` pela primeira vez vê dois blocos de conteúdo quase-duplicado (344 linhas `extern`, depois 344 linhas `#define`) -- mitigado pelo comentário bilíngue de cabeçalho explicando o porquê, mas ainda um arquivo gerado maior do que a versão de bloco único.

**Riscos / pontos de atenção:**
- O fix é uma renomeação de *variável-ponteiro*; não previne (nem pode prevenir) um host de definir independentemente seu próprio símbolo `glx_glClear` COLIDENTE -- mas isso exigiria que o host reusasse deliberadamente a convenção de nomenclatura interna e privada da glintfx, um acidente astronomicamente improvável comparado a um host inocentemente chamando `glClear` (uma função GL real que todo host GL chama).
- A ordem do bloco `#define` (depois do include de `glcorearb.h`, depois de toda declaração `extern`) é estruturalmente importante pra legibilidade e pro raciocínio declarado de não-colisão-com-typedefs, ainda que a semântica de substituição só-por-token-inteiro do pré-processador C signifique que a ordem estrita não é, de fato, exigida pra correção aqui -- documentado no gerador e no próprio header pra que uma edição futura não "simplifique" a ordem e perca o raciocínio documentado.

## Options considered / Opções consideradas

**EN:**
1. **`static` linkage instead of a name prefix.** Rejected: `gl_loader.c`'s pointer variables must stay externally visible within `libglintfx.a` -- every other translation unit in the library (`app.cpp`, `render_gl3.cpp`, decorators) references them across `.o` boundaries via the `extern` declarations in `gl_loader.h`. `static` would only work if the entire loader collapsed into a single translation unit, a much larger refactor for no benefit over a rename.
2. **Only rename the SYMBOL TABLE's lookup strings, leave the pointer variables bare-named.** Rejected: the lookup strings were never the problem -- they are, and remain, plain string literals invisible to the linker's symbol table. The actual exported data symbols are the pointer *variables*; only renaming those fixes `nm`'s output.
3. **A different prefix (`gfx_`, `gl3_`, no prefix + `__attribute__((visibility("hidden")))`).** The `visibility("hidden")` route was considered and rejected: it suppresses the symbol from the archive's *dynamic*-export view but does **not** prevent the exact static-link name collision this ADR fixes -- `ld` still resolves a `.a`'s hidden-but-external symbols against an external reference at static-link time; hidden visibility only affects what a *shared* library exports. `glx_` (not `gfx_`/`gl3_`) was chosen to match the naming precedent [ADR-0009](0009-internalization-boundary.md) already set for Camada 0's link boundary, rather than invent a second convention for the same kind of problem.

**PT:**
1. **Linkage `static` em vez de prefixo de nome.** Rejeitada: as variáveis-ponteiro de `gl_loader.c` precisam continuar visíveis externamente dentro do `libglintfx.a` -- toda outra unidade de tradução da lib (`app.cpp`, `render_gl3.cpp`, decorators) as referencia através de fronteiras de `.o` via as declarações `extern` em `gl_loader.h`. `static` só funcionaria se o loader inteiro colapsasse numa única unidade de tradução, uma refatoração bem maior sem benefício sobre uma renomeação.
2. **Renomear só as strings de busca da TABELA DE SÍMBOLOS, deixando as variáveis-ponteiro com nome cru.** Rejeitada: as strings de busca nunca foram o problema -- são, e continuam sendo, literais de string comuns, invisíveis à tabela de símbolos do linker. Os símbolos de dado de fato exportados são as *variáveis*-ponteiro; só renomear essas conserta a saída do `nm`.
3. **Um prefixo diferente (`gfx_`, `gl3_`, sem prefixo + `__attribute__((visibility("hidden")))`).** A rota `visibility("hidden")` foi considerada e rejeitada: ela suprime o símbolo da visão de exportação *dinâmica* do archive mas **não** previne a exata colisão de nome em link estático que este ADR conserta -- o `ld` ainda resolve símbolos hidden-mas-externos de um `.a` contra uma referência externa em tempo de link estático; visibilidade hidden só afeta o que uma lib *compartilhada* exporta. `glx_` (e não `gfx_`/`gl3_`) foi escolhido pra casar com o precedente de nomenclatura que o [ADR-0009](0009-internalization-boundary.md) já fixou pra fronteira de link da Camada 0, em vez de inventar uma segunda convenção pro mesmo tipo de problema.

## Reversibility / Reversibilidade

**EN:** Two-way door. Purely a rename inside a generated, do-not-hand-edit pair of files plus the generator that produces them -- reverting means running the generator with the old naming functions restored (or hand-editing back, though that defeats the "generated" contract) and rebuilding; no public API, no RCSS surface, no ADR-0008/ADR-0009 contract changes. The only real cost of reversal is reintroducing the exact symbol-collision bug this ADR exists to fix -- there is no scenario where reverting is desirable absent a stronger alternative fix for the same root cause.

**PT:** Two-way door. Puramente uma renomeação dentro de um par de arquivos gerados, não editar à mão, mais o gerador que os produz -- reverter significa rodar o gerador com as funções de nomenclatura antigas restauradas (ou editar à mão de volta, embora isso quebre o contrato de "gerado") e rebuildar; nenhuma API pública, nenhuma superfície RCSS, nenhuma mudança de contrato do ADR-0008/ADR-0009. O único custo real de reversão é reintroduzir exatamente o bug de colisão de símbolo que este ADR existe para consertar -- não há cenário em que reverter seja desejável na ausência de um fix alternativo mais forte pra mesma causa raiz.
