## Context / Contexto

**EN:** glintfx is repositioning from "an embeddable UI layer for a host that already has a window" into a **batteries-included 2D game framework for Linux x86-64**: the single dependency a developer needs for a simple 2D game (window, frame loop, keyboard/mouse/gamepad input, audio, image decode, text with its own clean-room font engine, time) plus the differentiator nobody in the niche has: **declarative HTML/CSS UI with data-driven GL effects** (glow, gradient, blur, mask, ripple, polygon) and data-binding. Explicit non-goals: not a 3D engine, not an editor, not a full engine (no built-in physics, no mandatory ECS, no asset pipeline, no embedded scripting). The bar is "the 2D functions of SDL/Qt, and more", never Godot/Unity.

The origin story is a real pain, not a market thesis: building GusWorld (consumer number 1, an SDL3 game) required gluing SDL3 (window/input/audio) + Qt + several engines + glintfx itself. That glue is exactly what the indie community reports as its recurring cost (sources in the vision document, section 3). Validation goal: GusWorld running on glintfx alone.

Against that goal the leader imposed one HARD architectural constraint, confirmed explicitly on 2026-07-19: the framework must be **atomized, at subsystem granularity**. The question this ADR answers is what that means precisely, which modules exist, where their boundaries are, and how each atom plugs into the project's existing TDD-plus-audit discipline.

**PT:** A glintfx está se reposicionando de "uma camada de UI embutível pra um host que já tem janela" para um **framework de jogos 2D batteries-included para Linux x86-64**: a única dependência de que um dev precisa pra um jogo 2D simples (janela, loop de frame, input de teclado/mouse/gamepad, áudio, decode de imagem, texto com motor de fonte próprio clean-room, tempo) mais o diferencial que ninguém no nicho tem: **UI declarativa HTML/CSS com efeitos GL data-driven** (glow, gradiente, blur, mask, ripple, polygon) e data-binding. Não-objetivos explícitos: não é engine 3D, não é editor, não é engine completa (sem física embutida, sem ECS obrigatório, sem asset pipeline, sem scripting embarcado). A régua é "as funções 2D do SDL/Qt, e mais", nunca Godot/Unity.

A narrativa de origem é dor real, não tese de mercado: desenvolver o GusWorld (consumidor nº 1, jogo SDL3) exigiu colar SDL3 (janela/input/áudio) + Qt + vários motores + a própria glintfx. Essa cola é exatamente o que a comunidade indie reporta como custo recorrente (fontes no documento de visão, seção 3). Meta de validação: o GusWorld rodar só com glintfx.

Contra essa meta o líder impôs UMA restrição arquitetural dura, confirmada explicitamente em 2026-07-19: o framework deve ser **atomizado, com granularidade por subsistema**. A pergunta que este ADR responde é o que isso significa precisamente, quais módulos existem, onde ficam as fronteiras, e como cada átomo se encaixa na disciplina TDD-mais-auditoria já existente do projeto.

## Decision / Decisão

**EN:**

**(a) Every capability is an independent, opt-in MODULE at subsystem granularity.** One module per capability (window, input, audio, image, text, ui, fx, core, app), never per function or per effect. Each module has a clean public boundary, is buildable, linkable, and testable IN ISOLATION, and is selected by a build flag of the `GLINTFX_MODULE_*` family (e.g. `GLINTFX_MODULE_INPUT`, `GLINTFX_MODULE_AUDIO`). Flag OFF means the module's code does not compile AND its external dependencies do not appear on the link line. The consumer composes a la carte. This is NOT a monolith with feature toggles: SDL3's `SDL_InitSubSystem` still ships one lib where disabling audio does not remove audio from the binary; here OFF removes both code and link deps. The living embryo already proves the pattern in production: `GLINTFX_BACKEND_GLFW=OFF` has produced a GLFW-free embed-only build since v0.2.1, and `GLINTFX_OWN_FONT_ENGINE` flips the font engine.

**(b) The module map.** Canonical list, boundaries, and dependencies (evolves the vision document section 2.5: identical except the **ui/fx split**, ratified by the leader on 2026-07-19 AFTER that revision; where the two diverge, this ADR governs):

| Module (atom) | Flag | Exposes (public boundary) | Depends on | Exists today | Real independence |
|---|---|---|---|---|---|
| **core** | always ON (root) | common types, `version()`, `SystemClock` (time), fail-high error conventions | nothing | yes (`version.hpp`, `system_clock.cpp`) | total |
| **image** | `GLINTFX_MODULE_IMAGE` | PNG/JPG/TGA decode to RGBA pixels (optional premultiply) | core | yes (stb_image vendored), currently glued to ui | high (zero GL, zero window) |
| **text** | `GLINTFX_MODULE_TEXT` | glyph raster to bitmap/atlas + metrics; color emoji (CBDT, later COLR) is a feature INSIDE this subsystem, no flag of its own | core, image (emoji) | yes (`font_engine_own.cpp`), today exposed only via RmlUi's `FontEngineInterface`; the own raster-to-atlas boundary is atomization work | medium (raster isolable; the RmlUi plug becomes a thin adapter owned by ui) |
| **audio** | `GLINTFX_MODULE_AUDIO` | device init, mixer, play/stop/volume, WAV/OGG decode | core ONLY | no (phase A3; miniaudio vendored, leader's decision) | total; the most independent atom in the house: no GL, no window, testable in light CI without Xvfb |
| **window** (includes the loop) | `GLINTFX_MODULE_WINDOW` (alias of today's `GLINTFX_BACKEND_GLFW`) | window + GL context creation, poll, swap, should_close, frame step | core, gl-loader (internal) | yes (`window_glfw.cpp`) | medium; window and event loop are inseparable by nature (poll/swap ARE operations of the window), so they are ONE atom |
| **input** | `GLINTFX_MODULE_INPUT` | `UiEvent`/`Key` (neutral format, exists today) + polling/callbacks for the game; keyboard/mouse backend + gamepad backend (evdev/udev, hotplug) | core; the keyboard/mouse backend depends on **window**; the gamepad backend does NOT (evdev direct) | neutral format yes (`ui_event.hpp`); physical capture NO (verified finding: zero `glfwSet*Callback` in `glintfx/src/`, no `process_event` on `App`) | split and honest: gamepad is window-independent; physical keyboard/mouse cannot be (the events are born in the window). Embed input keeps coming from the host via `UiEvent`, proof the neutral boundary works |
| **ui** | `GLINTFX_MODULE_UI` | LEAN UI: `UiLayer`, RML/RCSS documents, layout + text rendering (base GL3 render substrate), data-binding, DOM r/w, callbacks, and the decorator/effect EXTENSION POINT fx plugs into | core, text, image, gl-loader (internal) | yes (the current product; today fused with fx inside `render_gl3.cpp`, carving is atomization work, see (c)) | high: pulls neither window, nor input, nor audio, NOR fx; a lean UI renders with fx OFF; today's embed-only build is the proof of the outward boundary |
| **fx** (opt-in effects) | `GLINTFX_MODULE_FX` | RCSS effect decorators + GL shaders: glow/box-shadow, drop-shadow, gradient, blur, backdrop-blur, mask, ripple, polygon, image-tint | ui (registers into ui's extension point), core, gl-loader (internal) | yes (`decorator_*.cpp` are already separate TUs; the filter/effect passes live inside `render_gl3.cpp`, carving is atomization work, see (c)) | opt-in atom: OFF = lean UI without the effect shaders in the binary; dependency is one-way fx -> ui, never the reverse |
| **app** (framework composition) | `GLINTFX_MODULE_APP` | `App`: composes window+input+ui(+audio) into "the game inside glintfx" | window, input, ui | partial (`App` without input; A1 closes this) | it is the a-la-carte AGGREGATOR, not a capability atom |

**(c) The ui-to-fx boundary: two atoms, one-way dependency, one honest caveat.** The leader explicitly ratified MAXIMUM atomization here (2026-07-19): **ui and fx are TWO modules, split**. `ui` is the lean UI (text, layout, HTML/CSS, RmlUi core adapter, base GL3 render substrate) and exposes the extension point: decorator-instancer registration plus the renderer's filter/effect hooks. `fx` is the opt-in effect pack that registers into that point. The dependency is declared and one-way, **fx -> ui**; a consumer that wants lean UI does NOT carry the effect shaders. Honest caveat, declared instead of disguised: today the base render path and the effect passes are FUSED inside `render_gl3.cpp` (the decorators are already separate TUs: `decorator_polygon.cpp`, `decorator_ripple.cpp`, `decorator_image_tint.cpp`); carving the effect passes behind the extension point is real atomization work, scheduled as its own phase under the ordering principle of (g). Until that phase lands, the two modules build together under the aggregate target; what is forbidden is pretending the seam exists before it is built.

**window+loop are one atom** (confirmed by the leader, 2026-07-19): poll and swap are operations of the window; a separate "event loop module" would be fiction.

**(d) Dependency rules (the module contract).** (1) A module may only include headers of modules listed in its "depends on" column; the dependency graph is a DAG, cycles are forbidden. (2) No third-party type crosses any module's public boundary (the rule the whole lib already enforces at `glintfx/include/glintfx/`). (3) Flag OFF removes code AND link dependency. (4) Aliases keep every existing consumer unbroken: `GLINTFX_BACKEND_GLFW` aliases `GLINTFX_MODULE_WINDOW`, `GLINTFX_OWN_FONT_ENGINE` remains the text-engine selector inside `GLINTFX_MODULE_TEXT`. (5) CMake mechanics (to be delivered by phase A0): one target per module (`glintfx::input`, `glintfx::audio`, ...) plus the aggregate `glintfx::glintfx` preserving today's consumption; presets are named compositions (embed-only, full, audio-only, gamepad-only), and CI tests presets plus each atom built alone, NOT the 2^N combination matrix.

**(e) Each atom carries its own quality dossier.** The project discipline (TDD red/green/refactor, adversarial review that executes, per-release audit) applies PER MODULE: each module ships with its own test suite that passes with all other optional modules OFF, and each module is auditable alone (its audit chapter cites only its own boundary and its declared dependencies). Concretely for the split of (c): the `ui` suite must pass with fx OFF (lean UI renders), and the `fx` suite exercises the effects on top of `ui`, its declared dependency. The existing include-based encapsulation gate in CI (`tools/check_encapsulation.sh`) extends naturally to inter-module boundaries and is the enforcement mechanism of rules (1)/(2).

**(f) Delivery model: atomization is the FORM of every phase, not a phase of its own.** Each framework-mode phase (A1 input, A2 gamepad, A3 audio, A4 emoji) delivers its capability already inside its own module boundary and flag. The formal flag family and per-module targets arrive with phase A0 (build skeleton); code written before A0 lands must be placed so the future flag split is a build-file move, not a source refactor.

**(g) Execution ordering principle: easiest + best ROI first (leader directive, 2026-07-19).** The gradual absorption of SDL/GLFW/Qt functions picks the cheap, high-value wins first and defers the expensive, low-ROI internalizations. Confirmed sequence: **A0** (modular build skeleton, no new API) -> **A1** (input module in the standalone `App`), then the remaining capability phases ranked by this criterion (gamepad and vendored audio are cheap/high-value; emoji CBDT next). **Audio = vendored miniaudio** at its phase (leader's decision; the long-term internalization clause applies to it like the rest of the pile). **GLFW STAYS for now**: removing it blocks nothing and has low ROI; the own-window stack (X11+Wayland+EGL/GLX) is deferred until the rest of the pile is own. **Linux x86-64 first**; Windows/macOS framework mode deferred. Target scope: Linux x86-64, per the repo's declared platform.

**PT:**

**(a) Toda capacidade é um MÓDULO independente e opt-in, com granularidade por subsistema.** Um módulo por capacidade (window, input, audio, image, text, ui, fx, core, app), nunca por função ou por efeito. Cada módulo tem fronteira pública limpa, é buildável, linkável e testável ISOLADO, e é selecionado por uma flag de build da família `GLINTFX_MODULE_*` (ex.: `GLINTFX_MODULE_INPUT`, `GLINTFX_MODULE_AUDIO`). Flag OFF significa que o código do módulo não compila E as dependências externas dele não aparecem na linha de link. O consumidor compõe à la carte. NÃO é monolito com toggles de feature: o `SDL_InitSubSystem` do SDL3 ainda entrega UMA lib onde desligar áudio não tira o áudio do binário; aqui OFF remove código e deps de link. O embrião vivo já prova o padrão em produção: `GLINTFX_BACKEND_GLFW=OFF` produz build embed-only sem GLFW desde a v0.2.1, e `GLINTFX_OWN_FONT_ENGINE` troca o motor de fonte.

**(b) O mapa de módulos.** Lista canônica, fronteiras e dependências (evolui a seção 2.5 do documento de visão: idêntica exceto o **split ui/fx**, ratificado pelo líder em 2026-07-19 DEPOIS daquela revisão; onde os dois divergirem, este ADR manda):

| Módulo (átomo) | Flag | Expõe (fronteira pública) | Depende de | Existe hoje | Independência real |
|---|---|---|---|---|---|
| **core** | sempre ON (raiz) | tipos comuns, `version()`, `SystemClock` (tempo), convenções de erro fail-high | nada | sim (`version.hpp`, `system_clock.cpp`) | total |
| **image** | `GLINTFX_MODULE_IMAGE` | decode PNG/JPG/TGA para pixels RGBA (premultiply opcional) | core | sim (stb_image vendorizado), hoje colado na ui | alta (zero GL, zero janela) |
| **text** | `GLINTFX_MODULE_TEXT` | raster de glyphs para bitmap/atlas + métricas; emoji colorido (CBDT, depois COLR) é feature DENTRO do subsistema, sem flag própria | core, image (emoji) | sim (`font_engine_own.cpp`), hoje exposto só via `FontEngineInterface` do RmlUi; a fronteira própria raster-para-atlas é trabalho da atomização | média (o raster é isolável; o plug no RmlUi vira adapter fino de posse da ui) |
| **audio** | `GLINTFX_MODULE_AUDIO` | init de device, mixer, play/stop/volume, decode WAV/OGG | SÓ core | não (fase A3; miniaudio vendorizado, decisão do líder) | total; o átomo mais independente da casa: sem GL, sem janela, testável em CI leve sem Xvfb |
| **window** (inclui o loop) | `GLINTFX_MODULE_WINDOW` (alias do atual `GLINTFX_BACKEND_GLFW`) | criação de janela + contexto GL, poll, swap, should_close, frame step | core, gl-loader (interno) | sim (`window_glfw.cpp`) | média; janela e loop de eventos são inseparáveis por natureza (poll/swap SÃO operações da janela), então são UM átomo |
| **input** | `GLINTFX_MODULE_INPUT` | `UiEvent`/`Key` (formato neutro, já existe) + polling/callbacks pro jogo; backend teclado/mouse + backend gamepad (evdev/udev, hotplug) | core; o backend teclado/mouse depende de **window**; o backend gamepad NÃO (evdev direto) | formato neutro sim (`ui_event.hpp`); captura física NÃO (achado verificado: zero `glfwSet*Callback` em `glintfx/src/`, nenhum `process_event` no `App`) | dividida e honesta: gamepad é independente de janela; teclado/mouse físico não tem como ser (os eventos nascem na janela). No embed, o input segue vindo do host via `UiEvent`, prova de que a fronteira neutra funciona |
| **ui** | `GLINTFX_MODULE_UI` | UI ENXUTA: `UiLayer`, documentos RML/RCSS, layout + render de texto (substrato base de render GL3), data-binding, DOM r/w, callbacks, e o PONTO DE EXTENSÃO de decorator/efeito onde o fx pluga | core, text, image, gl-loader (interno) | sim (o produto atual; hoje fundido com o fx dentro do `render_gl3.cpp`, o recorte é trabalho da atomização, ver (c)) | alta: não puxa window, nem input, nem audio, NEM fx; UI enxuta renderiza com fx OFF; o build embed-only de hoje é a prova da fronteira externa |
| **fx** (efeitos opt-in) | `GLINTFX_MODULE_FX` | decorators de efeito RCSS + shaders GL: glow/box-shadow, drop-shadow, degradê, blur, backdrop-blur, mask, ripple, polygon, image-tint | ui (registra-se no ponto de extensão da ui), core, gl-loader (interno) | sim (os `decorator_*.cpp` já são TUs separadas; os passes de filtro/efeito vivem dentro do `render_gl3.cpp`, o recorte é trabalho da atomização, ver (c)) | átomo opt-in: OFF = UI enxuta sem os shaders de efeito no binário; dependência de mão única fx -> ui, nunca o inverso |
| **app** (composição framework) | `GLINTFX_MODULE_APP` | `App`: compõe window+input+ui(+audio) no modo "o jogo dentro da glintfx" | window, input, ui | parcial (`App` sem input; o A1 fecha isso) | é o AGREGADOR à la carte, não um átomo de capacidade |

**(c) A fronteira ui-fx: dois átomos, dependência de mão única, uma ressalva honesta.** O líder ratificou explicitamente a atomização MÁXIMA aqui (2026-07-19): **ui e fx são DOIS módulos, separados**. A `ui` é a UI enxuta (texto, layout, HTML/CSS, adapter do core RmlUi, substrato base de render GL3) e expõe o ponto de extensão: registro de decorator-instancer mais os ganchos de filtro/efeito do renderer. O `fx` é o pacote de efeitos opt-in que se registra nesse ponto. A dependência é declarada e de mão única, **fx -> ui**; um consumidor que quer UI enxuta NÃO carrega os shaders de efeito. Ressalva honesta, declarada em vez de disfarçada: hoje o caminho base de render e os passes de efeito estão FUNDIDOS dentro do `render_gl3.cpp` (os decorators já são TUs separadas: `decorator_polygon.cpp`, `decorator_ripple.cpp`, `decorator_image_tint.cpp`); recortar os passes de efeito atrás do ponto de extensão é trabalho real de atomização, agendado como fase própria sob o princípio de ordenação de (g). Até essa fase aterrissar, os dois módulos buildam juntos sob o alvo agregado; o que fica proibido é fingir que a costura existe antes de ela ser construída.

**window+loop são um átomo só** (confirmado com o líder, 2026-07-19): poll e swap são operações da janela; um "módulo de event loop" separado seria ficção.

**(d) Regras de dependência (o contrato de módulo).** (1) Um módulo só pode incluir headers dos módulos listados na sua coluna "depende de"; o grafo de dependência é um DAG, ciclo é proibido. (2) Nenhum tipo de terceiro cruza a fronteira pública de nenhum módulo (a regra que a lib inteira já aplica em `glintfx/include/glintfx/`). (3) Flag OFF remove código E dependência de link. (4) Aliases mantêm todo consumidor existente intacto: `GLINTFX_BACKEND_GLFW` vira alias de `GLINTFX_MODULE_WINDOW`, `GLINTFX_OWN_FONT_ENGINE` permanece o seletor de motor dentro de `GLINTFX_MODULE_TEXT`. (5) Mecânica CMake (a entregar pela fase A0): um alvo por módulo (`glintfx::input`, `glintfx::audio`, ...) mais o agregado `glintfx::glintfx` preservando o consumo atual; presets são composições nomeadas (embed-only, full, audio-only, gamepad-only), e o CI testa os presets mais cada átomo buildado sozinho, NÃO a matriz combinatória 2^N.

**(e) Cada átomo carrega o próprio dossiê de qualidade.** A disciplina do projeto (TDD red/green/refactor, review adversarial que executa, auditoria por release) aplica POR MÓDULO: cada módulo entrega a própria suíte de testes, que passa com todos os outros módulos opcionais OFF, e cada módulo é auditável sozinho (o capítulo de auditoria dele cita só a própria fronteira e as dependências declaradas). Concretamente pro split de (c): a suíte da `ui` deve passar com fx OFF (UI enxuta renderiza), e a suíte do `fx` exercita os efeitos por cima da `ui`, sua dependência declarada. O gate de encapsulamento include-based já existente no CI (`tools/check_encapsulation.sh`) se estende naturalmente às fronteiras entre módulos e é o mecanismo de enforcement das regras (1)/(2).

**(f) Modelo de entrega: a atomização é a FORMA de cada fase, não uma fase à parte.** Cada fase do framework mode (A1 input, A2 gamepad, A3 áudio, A4 emoji) entrega a capacidade já dentro da própria fronteira de módulo e flag. A família formal de flags e os alvos por módulo chegam com a fase A0 (esqueleto de build); código escrito antes do A0 aterrissar deve ser posicionado de modo que o split futuro de flag seja um movimento de arquivo de build, não um refactor de source.

**(g) Princípio de ordenação da execução: mais fácil + melhor ROI primeiro (diretriz do líder, 2026-07-19).** A absorção gradual das funções do SDL/GLFW/Qt pega os ganhos baratos e de alto valor primeiro e adia as internalizações caras de ROI baixo. Sequência confirmada: **A0** (esqueleto modular de build, sem API nova) -> **A1** (módulo input no `App` standalone), depois as demais fases de capacidade rankeadas por este critério (gamepad e áudio vendorizado são baratos/de alto valor; emoji CBDT em seguida). **Áudio = miniaudio vendorizado** na fase dele (decisão do líder; a cláusula de internalização de longo prazo se aplica a ele como ao resto da pilha). **GLFW FICA por ora**: tirá-lo não bloqueia nada e tem ROI baixo; a pilha de janela própria (X11+Wayland+EGL/GLX) fica adiada até o resto da pilha ser próprio. **Linux x86-64 primeiro**; framework mode Windows/macOS adiado.

## Consequences / Consequências

**EN:**

**Positive:**
- The atomization itself is a declared product differentiator: raylib is deliberately monolithic, SDL3 links as one lib, Qt's core always comes along; the honest precedent (SFML's separate libs) lacks the declarative UI and the effects. "True opt-in modules + HTML/CSS UI + data-driven effects + MPL-2.0 static-link-friendly" exists in none of the four.
- Compositions with standalone value become products for free: embed-only (today's GusWorld), audio-only (an SDL game that only wants the mixer), gamepad-only (a raw X11 host that wants hotplug without SDL), full framework (the endgame GusWorld).
- Zero regression risk to the embed product: `UiLayer` is a preset of this architecture, and the embed (GLFW=OFF) suite is its permanent guard.
- Per-module audit matches the house discipline instead of fighting it; audit scope per release shrinks to the touched atoms.
- The dependency DAG plus the include gate turn "module boundary violated" from a review opinion into a CI failure.

**Negative / accepted as cost:**
- More build-system surface: N flags, N targets, presets, and a CI lane per preset plus per-atom isolated builds. Linear, not combinatorial, but real maintenance.
- The `GLINTFX_MODULE_*` naming commits us before A0 exists; if A0's implementation finds a better mechanic (e.g. CMake components), the flag names still must be honored as aliases.
- Declaring ui+fx a coupled pair means the "every module fully isolated" marketing line needs the honest footnote.

**Risks / points of attention:**
- Granularity creep in BOTH directions: per-function flags (combinatorial explosion) and "merge it all back for convenience" (monolith regression) are both forbidden by this ADR; the module list in (b) changes only by a new ADR.
- The canonical sequence puts phase A0 (modular build skeleton, no new API) BEFORE A1 (vision document, Option A); if the leader fast-tracks a capability slice ahead of A0, its code lands under the existing `GLINTFX_BACKEND_GLFW` gate and rule (f) is the mitigation (positioned so the flag split is a build-file move).
- Gamepad (A2) introduces a new UNTRUSTED input surface (mapping DB file, devices); `.bigtech-porte` trigger (d) stands watch (Narciso/CISO).

**PT:**

**Positivas:**
- A atomização em si é diferencial de produto declarado: raylib é deliberadamente monolítica, SDL3 linka como uma lib só, o núcleo do Qt sempre vem junto; o precedente honesto (libs separadas da SFML) não tem a UI declarativa nem os efeitos. "Módulos opt-in de verdade + UI HTML/CSS + efeitos data-driven + MPL-2.0 amigável a link estático" não existe em nenhum dos quatro.
- Composições com valor próprio viram produto de graça: embed-only (o GusWorld de hoje), audio-only (um jogo SDL que só quer o mixer), gamepad-only (host X11 cru que quer hotplug sem SDL), framework completo (o GusWorld do endgame).
- Zero risco de regressão pro produto embed: o `UiLayer` é um preset desta arquitetura, e a suíte embed (GLFW=OFF) é o guarda permanente dela.
- Auditoria por módulo casa com a disciplina da casa em vez de brigar com ela; o escopo de auditoria por release encolhe pros átomos tocados.
- O DAG de dependência mais o gate de include transformam "fronteira de módulo violada" de opinião de review em falha de CI.

**Negativas / aceitas como custo:**
- Mais superfície de build-system: N flags, N alvos, presets, e uma pista de CI por preset mais builds isolados por átomo. Linear, não combinatório, mas manutenção real.
- A nomenclatura `GLINTFX_MODULE_*` nos compromete antes do A0 existir; se a implementação do A0 achar mecânica melhor (ex.: components do CMake), os nomes de flag ainda precisam ser honrados como aliases.
- Declarar ui+fx par acoplado significa que a linha de marketing "todo módulo totalmente isolado" precisa da nota de rodapé honesta.

**Riscos / pontos de atenção:**
- Deriva de granularidade nas DUAS direções: flag por função (explosão combinatória) e "junta tudo de volta por conveniência" (regressão a monolito) são ambas proibidas por este ADR; a lista de módulos em (b) só muda por novo ADR.
- A sequência canônica põe a fase A0 (esqueleto modular de build, sem API nova) ANTES do A1 (doc de visão, Opção A); se o líder adiantar uma fatia de capacidade à frente do A0, o código dela aterrissa sob o gate existente `GLINTFX_BACKEND_GLFW` e a regra (f) é a mitigação (posicionado pra que o split de flag seja movimento de arquivo de build).
- Gamepad (A2) introduz superfície de input NÃO-CONFIÁVEL nova (arquivo de DB de mapeamento, devices); o gatilho (d) do `.bigtech-porte` fica de vigília (Narciso/CISO).
