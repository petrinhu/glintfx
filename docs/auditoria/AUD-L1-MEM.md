## Review of new GL resources (v0.7 → v0.11) / Revisão dos recursos GL novos (v0.7 → v0.11)

**EN:** `tint_program_` (v0.7), `ripple_program_` + `backdrop_capture_tex_` (v0.11, `L1.22-WAVE`/`L1.22-CAPTURE`) all follow the **same lazy-compile / single-owner / destructor-teardown pattern** already established and already sanitizer-clean:

1. **Creation is idempotent and lazy.** `EnsureTintProgramCompiled`/`EnsureRippleProgramCompiled` (`render_gl3.cpp:965-1075`) each guard on `<name>_program_ || <name>_compile_attempted_` — a compiled program is never recompiled, and a prior hard failure is never retried every frame (avoids a resource-churn or performance-DoS class of self-inflicted issue, not just a correctness nicety). `backdrop_capture_tex_` (`EnsureBackdropCaptured`, `render_gl3.cpp:911-954`) is `glGenTextures`'d exactly once (`if (backdrop_capture_tex_ == 0)`, line 929) and only re-`glTexImage2D`'d (not re-`glGenTextures`'d) on a capture-rectangle size change — no texture-object leak across resizes.
2. **Destructor covers all 3.** `~Gl3RenderInterface()` (`render_gl3.cpp:438-445`) unconditionally checks-then-deletes `tint_program_`, `ripple_program_`, and `backdrop_capture_tex_` — every GL name this class ever allocates is torn down on the same exit path, no early-return or exception path bypasses it (the destructor body has no branches that could skip a `glDelete*` call).
3. **Ownership boundary is honoured.** Both the tint and ripple `RenderShader` branches draw from `wrapper->{tint,ripple}_data.vao` — a per-element resource **owned and released by the decorator's own `ReleaseElementData`** (`decorator_image_tint.cpp`/`decorator_ripple.cpp`), never by `Gl3RenderInterface`. This class only ever owns the 2 GL **programs** and the 1 shared **capture texture**, and the doc comments at every touch-point (destructor, `CompileShader`, `ReleaseShader`) explicitly re-state this boundary — the exact discipline whose violation caused the historical `bind_*` UAF (below).
4. **`EnsureBackdropCaptured`'s un-restored GL state is correctly bounded by `GlStateGuard`.** `EnsureBackdropCaptured` deliberately rebinds `GL_READ_FRAMEBUFFER` to 0 and calls `glReadBuffer(GL_BACK)` without restoring either itself (`render_gl3.cpp:926-927`) — verified this is safe **only** because `GlStateGuard`'s ctor/dtor window (`Engine::render_compose`, `engine.cpp:136`) unconditionally encloses the entire `begin_frame_compose → Context::Render() → end_frame_compose` sequence, and `EnsureBackdropCaptured` is only ever reachable from inside that window (called from `RenderShader`'s ripple branch, itself only reachable from inside `Context::Render()`). Read `gl_state.hpp` in full to confirm the L1.22-CAPTURE addition (`read_fbo_`/`read_buffer_`, ctor lines 113-114, dtor lines 119-127) actually captures+restores `GL_READ_FRAMEBUFFER_BINDING`/`GL_READ_BUFFER` and that the dtor's ordering is correct (rebind the FBO target **before** setting its read-buffer, since `GL_READ_BUFFER` state belongs to whichever FBO is currently bound — the dtor does this in the right order, lines 126-127). No leaked GL binding state reaches the host's own renderer after `render_compose()` returns.
5. **One-capture-per-frame latch (`backdrop_captured_this_frame_`) is correctly scoped** — set `true` **before** any early return inside `EnsureBackdropCaptured` (line 914, an explicit anti-retry-storm discipline: a failed capture with `w<=0`/`h<=0` is not retried on every ripple element that frame) and reset to `false` only by `ArmBackdropCapture` (line 900), called once per `begin_frame_compose`. No use of a stale capture texture across frames was found — `cap_w_`/`cap_h_` gate the re-allocation, and the texture is only ever bound for read (`GL_TEXTURE_2D` sampler), never written outside `EnsureBackdropCaptured`.

No new leak, no new UAF, no new double-free, no new unbounded-growth pattern was found in this surface. The cold-start redesign itself (`L1.22-CAPTURE`, replacing a counter-gated `ripple_active_counter_` design with the on-demand `Arm`/`Ensure` split) was a **correctness** fix (a rendering bug — the opening frame of a ripple animation silently rendering nothing), not a memory-safety one; it does not change this audit's verdict but is noted for completeness since its own doc comment is extensive and worth cross-referencing.

**PT:** `tint_program_` (v0.7), `ripple_program_` + `backdrop_capture_tex_` (v0.11, `L1.22-WAVE`/`L1.22-CAPTURE`) seguem todos o **mesmo padrão de compile-lazy / dono-único / teardown-no-destrutor** já estabelecido e já limpo sob sanitizer:

1. **Criação é idempotente e lazy.** `EnsureTintProgramCompiled`/`EnsureRippleProgramCompiled` (`render_gl3.cpp:965-1075`) cada uma se protege em `<nome>_program_ || <nome>_compile_attempted_` — um programa compilado nunca é recompilado, e uma falha anterior nunca é retentada a cada frame (evita uma classe de churn-de-recurso ou DoS-de-performance auto-infligido, não só uma sutileza de correção). `backdrop_capture_tex_` (`EnsureBackdropCaptured`, `render_gl3.cpp:911-954`) recebe `glGenTextures` exatamente uma vez (`if (backdrop_capture_tex_ == 0)`, linha 929) e só recebe `glTexImage2D` de novo (não `glGenTextures` de novo) numa mudança de tamanho do retângulo de captura — nenhum leak de objeto de textura entre resizes.
2. **Destrutor cobre os 3.** `~Gl3RenderInterface()` (`render_gl3.cpp:438-445`) checa-então-deleta incondicionalmente `tint_program_`, `ripple_program_`, e `backdrop_capture_tex_` — todo nome GL que esta classe algum dia aloca é desmontado no mesmo caminho de saída, nenhum early-return ou caminho de exceção contorna isso (o corpo do destrutor não tem ramos que poderiam pular uma chamada `glDelete*`).
3. **Fronteira de posse é respeitada.** Ambos os ramos de `RenderShader` de tint e ripple desenham a partir de `wrapper->{tint,ripple}_data.vao` — um recurso por-elemento **dono e liberado pelo próprio `ReleaseElementData` do decorator** (`decorator_image_tint.cpp`/`decorator_ripple.cpp`), nunca pelo `Gl3RenderInterface`. Esta classe só é dona dos 2 **programas** GL e da 1 **textura de captura** compartilhada, e os doc comments em todo ponto de toque (destrutor, `CompileShader`, `ReleaseShader`) reafirmam explicitamente essa fronteira — a mesma disciplina cuja violação causou a UAF histórica do `bind_*` (abaixo).
4. **O estado GL não-restaurado de `EnsureBackdropCaptured` é corretamente delimitado pelo `GlStateGuard`.** `EnsureBackdropCaptured` deliberadamente revincula `GL_READ_FRAMEBUFFER` a 0 e chama `glReadBuffer(GL_BACK)` sem restaurar nenhum dos dois por conta própria (`render_gl3.cpp:926-927`) — verificado que isso é seguro **só** porque a janela ctor/dtor do `GlStateGuard` (`Engine::render_compose`, `engine.cpp:136`) envolve incondicionalmente a sequência inteira `begin_frame_compose → Context::Render() → end_frame_compose`, e `EnsureBackdropCaptured` só é alcançável de dentro dessa janela (chamado do ramo ripple de `RenderShader`, ele próprio só alcançável de dentro de `Context::Render()`). Lido o `gl_state.hpp` por inteiro para confirmar que a adição do L1.22-CAPTURE (`read_fbo_`/`read_buffer_`, ctor linhas 113-114, dtor linhas 119-127) de fato captura+restaura `GL_READ_FRAMEBUFFER_BINDING`/`GL_READ_BUFFER` e que a ordem do dtor está correta (revincula o alvo do FBO **antes** de setar o read-buffer dele, já que o estado `GL_READ_BUFFER` pertence a qualquer FBO que esteja atualmente vinculado — o dtor faz isso na ordem certa, linhas 126-127). Nenhum estado de binding GL vazado alcança o próprio renderer do host depois de `render_compose()` retornar.
5. **A trava de uma-captura-por-frame (`backdrop_captured_this_frame_`) está corretamente escopada** — setada `true` **antes** de qualquer retorno antecipado dentro de `EnsureBackdropCaptured` (linha 914, uma disciplina explícita anti-tempestade-de-retentativa: uma captura falha com `w<=0`/`h<=0` não é retentada a cada elemento ripple naquele frame) e resetada a `false` só pelo `ArmBackdropCapture` (linha 900), chamado uma vez por `begin_frame_compose`. Nenhum uso de textura de captura obsoleta entre frames foi encontrado — `cap_w_`/`cap_h_` protegem a realocação, e a textura só é vinculada para leitura (sampler `GL_TEXTURE_2D`), nunca escrita fora de `EnsureBackdropCaptured`.

Nenhum leak novo, nenhuma UAF nova, nenhum double-free novo, nenhum padrão novo de crescimento ilimitado foi encontrado nesta superfície. O próprio redesenho de cold-start (`L1.22-CAPTURE`, substituindo um design protegido por contador `ripple_active_counter_` pela divisão sob-demanda `Arm`/`Ensure`) foi um fix de **correção** (um bug de renderização — o frame de abertura de uma animação de ripple renderizando silenciosamente nada), não de memory-safety; não muda o veredito desta auditoria mas é anotado por completude já que seu próprio doc comment é extenso e vale a referência cruzada.

## Rigor evidence — 3 real bugs found and fixed / Evidência de rigor — 3 bugs reais achados e corrigidos

**EN:** Cited per the task brief as evidence this class of resource has real teeth in its review/test process, not a rubber-stamp:

1. **stb_image leak on exception path → RAII** (`LoadTexture`, v0.2.3 fixup, commit `9e4ea51`) — the raw `stbi_load_from_memory` return was originally freed by a manual `stbi_image_free` call reachable only on the happy path; an exception thrown by the subsequent `GenerateTexture` call (e.g. OOM) would skip it. Fixed by wrapping the decode result in `std::unique_ptr<unsigned char, decltype(&stbi_image_free)>` (current code, `render_gl3.cpp:483-485`, confirmed still in place) — freed on **every** exit path, including exceptions, with no manual free call needed anywhere downstream.
2. **`DataBinder::bind_*` heap-use-after-free on same-key rebind [Important]** (v0.3.1, commit `3b25c06`) — calling `bind_number/string/bound/list` twice with the same key before `load()` overwrote a map slot holding a `std::unique_ptr`, freeing the first cell before RmlUi's own `Bind()` (which silently no-ops on a duplicate key via `map::emplace`) had finished consuming it — leaving RmlUi's variable map pointing at freed memory. Fixed with a check-before-mutate guard (duplicate key rejected, `return false`, before touching the map); reproduced live under ASan pre-fix, confirmed clean post-fix, with a dedicated regression test (`data_model_rebind_uaf`).
3. **`set_click_callback` heap-use-after-free by reentrancy [Critical]** (v0.4.1, commit `70cf07f`, `AUD-TEC-3`) — the internal click listener invoked the registered `std::function` through a pointer into its own storage; a handler that called `set_click_callback(...)` on itself (a plausible "clicked Play → swap in next screen's handler" pattern) destroyed the in-flight functor mid-invocation.
