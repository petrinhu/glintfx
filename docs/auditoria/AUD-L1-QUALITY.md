## Method / Método

**EN:** Read-only, no build (per this pass's own gotcha list). (1) Structural map via `ctags -x --c++-kinds=f` per file (function boundaries + spans). (2) Comment/blank/code line separation per file and per function (`grep -c -E '^\s*(//|$)'` over exact function spans) — the decisive metric, since the repo's own convention (CLAUDE.md "Convenções") *mandates* bilingual doc-comments in the same file. (3) `cppcheck --enable=style,unusedFunction` over `glintfx/src/` (completed clean; findings triaged below, every `unusedFunction` hit cross-checked by grep against `tests/`, `demos/` and the public headers before being accepted or discarded). (4) `lizard` is not installed — cyclomatic complexity estimated by reading the code-dense functions in full plus indentation-depth measurement over exact function spans. (5) Duplication hunted by grep for repeated idioms (the ancestor-id walk, the `GetElementById` prelude) and by `diff` between the two shader-decorator TUs. (6) The `.clang-tidy` config and its documented ~87-finding backlog read and cited by CLASS (per this pass's brief), not re-enumerated. clang-tidy itself was NOT re-run: `glintfx/build/` has no `compile_commands.json` (`CMAKE_EXPORT_COMPILE_COMMANDS` is enabled by the CI lint job, not the local build), and generating one requires a configure step — out of scope for a read-only pass; the CI `lint-and-scan` gate (green at v0.11.1) already runs it with `clang-analyzer-*` as a hard gate.

**PT:** Read-only, sem build (conforme a própria lista de gotchas desta passada). (1) Mapa estrutural via `ctags -x --c++-kinds=f` por arquivo (fronteiras + extensão de funções). (2) Separação comentário/vazio/código por arquivo e por função (`grep -c -E '^\s*(//|$)'` sobre a extensão exata de cada função) — a métrica decisiva, já que a convenção do próprio repo (CLAUDE.md "Convenções") *obriga* doc-comments bilíngues no mesmo arquivo. ## Conclusion / Conclusão

**EN:** The honest answer to "does the glintfx have a god-class problem?" is **no**.
