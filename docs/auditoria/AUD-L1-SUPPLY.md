### `stb_image` v2.30, vendored (`glintfx/third_party/stb/stb_image.h`) / vendorizado

- **`CVE-2026-5186`** (CVSS 4.8, Medium) — use-after-free/double-free in `stbi__load_gif_main`, CWE-119, during multi-frame GIF processing.

Neither vulnerable function is emitted into `stb_image_impl.o`, so neither CVE has an exploitable path through `glintfx::LoadTexture` (the only call site of `stbi_load_from_memory`, which itself never dispatches to the GIF decoder for a non-GIF-only build) — **this is the `STBI_ONLY_*` minimal-surface hardening already in place doing its job**, not a gap this audit needed to close. ## Process — advisory check on pin bump / Processo — checagem de advisory no bump de pin

**EN:** Recommended, lightweight process for the next time a pinned dependency (RmlUi `GIT_TAG`, `stb_image` vendored version) is bumped, to keep this audit's snapshot from silently going stale:

1. **Before bumping**, note the new target commit/version.

## AUD-L1-BUILD fold-in — recommended hardening flags for packagers / dobra do AUD-L1-BUILD — flags de hardening recomendadas pra empacotadores

**EN:** Per the Caetano/CTO decision to fold `AUD-L1-BUILD` into `SUPPLY`+`MEM`+`PARSE` for the minimum cut (see `TODO.md:125`): `glintfx` is distributed **as source** (`FetchContent`/`add_subdirectory`) — this repo's own `CMakeLists.txt` does not, and should not, unilaterally impose hardening compiler/linker flags that a consumer's build system did not opt into (a library forcing `-Wl,-z,now` on a consumer that needs lazy binding for a plugin-loading pattern, for instance, would be a correctness regression, not a security improvement). This is therefore a **recommendation for downstream packagers/consumers**, not a CI-gate change (no CI file was touched): when building `glintfx` (or any C++ consumer embedding it) for distribution, apply:

- **`-D_FORTIFY_SOURCE=3`** — compile-time + runtime buffer-overflow checks on `mem*`/`str*`/`printf`-family calls (requires `-O1`+ and glibc ≥ 2.34 for the `=3` variant; `=2` is the safe fallback on older glibc).
- **`-D_GLIBCXX_ASSERTIONS`** — libstdc++ container bounds-checking (`std::vector::operator[]`, `std::string`, etc.) at low runtime cost; catches exactly the class of OOB-container-access bug this codebase's own `AUD-TEC`/`AUD-L1-PARSE` hardening passes already defend against by hand.
- **`-fstack-protector-strong`** — stack-smashing canary on functions with local arrays/address-taken locals (broader coverage than the default `-fstack-protector`, cheaper than `-fstack-protector-all`).
- **`-Wl,-z,relro,-z,now`** — full RELRO: GOT made read-only after all symbols are resolved at load time (`-z,now` forces eager binding so `-z,relro` has something to protect), closing the classic GOT-overwrite exploitation primitive.

None of these change `glintfx`'s ABI or observable behaviour — they are purely defense-in-depth compile/link-time hardening, safe to layer onto any consumer's own build without coordinating with this repo. `-fsanitize=address,undefined` (already exercised by the nightly gate, `AUD-L1-MEM.md`) remains a **development/CI-time** tool, not a production hardening flag, and is out of scope for this recommendation.

**PT:** Conforme a decisão do Caetano/CTO de dobrar o `AUD-L1-BUILD` em `SUPPLY`+`MEM`+`PARSE` no corte mínimo (ver `TODO.md:125`): a `glintfx` é distribuída **como fonte** (`FetchContent`/`add_subdirectory`) — o próprio `CMakeLists.txt` deste repo não impõe, e não deveria impor, unilateralmente flags de hardening de compilador/linker que o sistema de build de um consumidor não optou por usar (uma lib forçando `-Wl,-z,now` num consumidor que precisa de lazy binding pra um padrão de carregamento de plugin, por exemplo, seria uma regressão de correção, não uma melhoria de segurança). Isto é portanto uma **recomendação pra empacotadores/consumidores a jusante**, não uma mudança de gate de CI (nenhum arquivo de CI foi tocado): ao buildar a `glintfx` (ou qualquer consumidor C++ que a embarque) pra distribuição, aplicar:

- **`-D_FORTIFY_SOURCE=3`** — checagens de overflow de buffer em tempo-de-compilação + tempo-de-execução em chamadas da família `mem*`/`str*`/`printf` (exige `-O1`+ e glibc ≥ 2.34 pra variante `=3`; `=2` é o fallback seguro em glibc mais antiga).
- **`-D_GLIBCXX_ASSERTIONS`** — checagem de bounds de container da libstdc++ (`std::vector::operator[]`, `std::string`, etc.) com baixo custo de runtime; pega exatamente a classe de bug de acesso-fora-de-bounds-de-container que os próprios passes de hardening `AUD-TEC`/`AUD-L1-PARSE` deste codebase já defendem à mão.
- **`-fstack-protector-strong`** — canary anti-stack-smashing em funções com arrays locais/locais com endereço tomado (cobertura mais ampla que o `-fstack-protector` default, mais barato que `-fstack-protector-all`).
- **`-Wl,-z,relro,-z,now`** — RELRO completo: GOT tornado somente-leitura depois que todos os símbolos são resolvidos no load time (`-z,now` força bind ansioso pra que `-z,relro` tenha algo pra proteger), fechando a primitiva clássica de exploração de sobrescrita-de-GOT.

Nenhuma dessas muda a ABI ou o comportamento observável da `glintfx` — são hardening puramente de defesa-em-profundidade em tempo de compilação/link, seguras de acrescentar em cima do build de qualquer consumidor sem coordenar com este repo. `-fsanitize=address,undefined` (já exercitado pelo gate noturno, `AUD-L1-MEM.md`) segue sendo uma ferramenta de **desenvolvimento/tempo-de-CI**, não uma flag de hardening de produção, e está fora do escopo desta recomendação.
