## Method / Método

Then built a **disposable, hosted (non-freestanding) probe harness** outside the repo (`/tmp/.../scratchpad/probe/`, never committed) that compiles the exact same `src/mem.c`, `src/str.c`, `src/conv.c`, `src/alloc.c`, `src/printf.c` sources (unmodified logic, only a tiny `sys_mmap`/`sys_write` stub swapping the raw syscall for the hosted equivalent) under `clang -fsanitize=address,undefined -fno-sanitize-recover=all`, and ran **51 adversarial checks** covering every boundary this audit's scope calls out: `itoa(INT_MIN)`, `atoi`/`atou` saturation at and past `INT_MAX`/`INT_MIN`/`UINT_MAX`, `memmove` overlap in both directions, `memcmp` unsigned byte ordering, all three `strncpy` cases (pad / exact-fit / truncate) plus `n == 0`, `strchr` NUL-match, the `malloc` overflow-guard boundary (`SIZE_MAX-31` accepted, `SIZE_MAX-30` rejected), `malloc(0)`/`realloc(NULL,·)`/`realloc(·,0)`, ~20000 small allocations forcing multiple arena `mmap`s, and `mini_vsnprintf`/`mini_printf` truncation/`cap==0`/`cap==1`/NULL-`%s`/unknown-directive/a `%s` run longer than the internal 256-byte flush buffer. **0 sanitizer reports, 0/51 failures.** Also re-ran the repo's own `make clean && make test` (freestanding build) as a smoke baseline: 15/15 PASS. No product file was modified; the probe harness lives only in the scratchpad and is not part of this deliverable.

## Judgment on scope item 1 — syscall argument validation / Julgamento sobre o item de escopo 1 — validação de argumento de syscall

**EN:** `src/sys_write.c`, `src/sys_read.c`, `src/sys_mmap.c` forward their `buf`/`addr`/`count`/`len` arguments to the raw `syscall3`/`syscall6` instruction verbatim, with no userland pointer/length sanitization. **Judged ACCEPTABLE, not a gap**, for this codebase's actual trust model: (1) this is a single-process, single-privilege-level freestanding runtime with no privilege boundary crossing inside the program — every caller of `sys_write`/`sys_read`/`sys_mmap` is this project's own code, compiled together, never a separate untrusted component; (2) the Linux kernel is the only real enforcement point for pointer validity anyway (a bad `buf` pointer returns `-EFAULT` from the kernel regardless of any userland pre-check we could add — we cannot know if a pointer is dereferenceable without dereferencing it, and Linux gives us that answer authoritatively via the syscall's own return value, already correctly interpreted by `alloc.c`'s `mmap_failed`); (3) adding a userland guard here would be pure redundancy with the kernel's own check, contributing no additional defense-in-depth against a threat that does not exist in this program's actual deployment shape (no network, no multi-tenant process, no privilege-separated caller). Should Layer 0 ever grow a component that accepts raw byte offsets/lengths from an external, adversarial source (a future file-format parser, say) and hands them to `sys_read`/`sys_write`, **that** component — not the thin syscall wrapper itself — would be the correct place to add bounds validation against the source's own semantics; the wrapper's job is to be a faithful, minimal ABI bridge, exactly as documented in `include/sys_mmap.h`'s own header comment ("this helper itself does NOT perform that check — it is the pure, minimal wrapper; the interpretation lives at the call site").

**PT:** `src/sys_write.c`, `src/sys_read.c`, `src/sys_mmap.c` repassam seus argumentos `buf`/`addr`/`count`/`len` pra instrução `syscall3`/`syscall6` crua ipsis litteris, sem nenhuma sanitização userland de ponteiro/tamanho. **Julgado ACEITÁVEL, não uma lacuna**, para o modelo de confiança real deste codebase: (1) esta é uma runtime freestanding de processo único e nível-de-privilégio único, sem cruzamento de fronteira de privilégio dentro do programa — todo chamador de `sys_write`/`sys_read`/`sys_mmap` é código deste próprio projeto, compilado junto, nunca um componente separado não-confiável; (2) o kernel Linux é o único ponto de aplicação real pra validade de ponteiro de qualquer forma (um ponteiro `buf` ruim retorna `-EFAULT` do kernel independente de qualquer pré-checagem userland que pudéssemos acrescentar — não temos como saber se um ponteiro é desreferenciável sem desreferenciar ele, e o Linux nos dá essa resposta de forma autoritativa via o próprio valor de retorno da syscall, já corretamente interpretado pelo `mmap_failed` do `alloc.c`); (3) acrescentar uma guarda userland aqui seria pura redundância com a checagem do próprio kernel, sem contribuir defesa-em-profundidade adicional contra uma ameaça que não existe no formato de deploy real deste programa (sem rede, sem processo multi-tenant, sem chamador com privilégio separado). Se a Camada 0 algum dia crescer um componente que aceite offsets/tamanhos crus de bytes vindos de uma fonte externa e adversarial (um futuro parser de formato de arquivo, digamos) e os repasse a `sys_read`/`sys_write`, **esse** componente — não o wrapper fino de syscall em si — seria o lugar correto pra acrescentar validação de bounds contra a semântica da própria fonte; o trabalho do wrapper é ser uma ponte de ABI fiel e mínima, exatamente como documentado no próprio comentário de cabeçalho de `include/sys_mmap.h` ("este helper em si NÃO faz essa checagem — é o wrapper puro e mínimo; a interpretação mora no ponto de chamada").

## Verified clean, with evidence / Verificado limpo, com evidência

**1. `memmove` overlap safety / Segurança de overlap do `memmove`** (`src/mem.c:74-137`) — Uses `uintptr_t` comparison (not raw pointer relational comparison, which would itself be UB for two independent objects per C23 6.5.9p6 — already correctly called out in the file's own comment) to pick forward-vs-backward copy direction. Probe: forward-overlap (`dst>src`) and backward-overlap (`dst<src`) both reproduce byte-for-byte correct results under ASan; `n==0` on a self-overlapping single byte is a correct no-op.

**2. `strncpy`'s three documented cases / Os três casos documentados do `strncpy`** (`src/str.c:122-144`) — `src_len < n` (pad), `src_len == n` (exact, no NUL), `src_len > n` (truncate, no NUL), and `n == 0` (no-op) all write exactly the documented byte range and never touch `dst[n]` or beyond, confirmed under ASan with a canary byte immediately past `dst[n-1]` in the probe (`dst[3]`/`dst[5]` untouched checks) for the exact-fit and truncate cases.

**3. `memcmp` unsigned byte ordering / Ordenação unsigned de byte do `memcmp`** (`src/mem.c:139-154`) — `0xFF` vs `0x01` correctly compares positive (both operands are `unsigned char` before the subtraction, so the classic signed-`char` `memcmp` bug — where `0xFF` as signed `char` is `-1` and would wrongly compare *less than* `0x01` — cannot occur here). Confirmed by probe.

**4. `itoa(INT_MIN, ..., 10)` / the classic `-INT_MIN` UB trap** (`src/conv.c:170-179`) — The `(unsigned)(-(value + 1)) + 1u` formula (never `-value` directly) avoids the one genuinely UB case in signed-integer negation-for-magnitude on this codebase's entire int↔string surface. Confirmed byte-exact `"-2147483648"` output under UBSan (which would abort on the naive `-value` formulation for this exact input — it does not abort here).

**5. `atoi`/`atou` saturating overflow, both directions / Overflow saturante do `atoi`/`atou`, nas duas direções** (`src/conv.c:184-274`) — The pre-multiply `mag > (limit - digit) / 10` guard correctly saturates at `INT_MAX`/`INT_MIN`/`UINT_MAX` for inputs at, one-past, and grossly past each limit (`"2147483648"`, `"9999999999999999999999"`, `"-2147483649"`, `"4294967296"`, `"99999999999999999999"`), with no signed-integer-overflow UB anywhere in the accumulation (all arithmetic happens in the unsigned domain until the final, UB-safe sign application). `atou("-5") -> 0` (documented gotcha, not a silent unsigned-wraparound of a negative value) confirmed.

**6. `malloc`'s integer-overflow guard, exact boundary / Guarda de overflow-de-inteiro do `malloc`, fronteira exata** (`src/alloc.c:163-171`, flagged in the file's own header comment for this audit — `W12` — to re-verify independently) — The threshold `size > (size_t)-1 - ALLOC_HEADER_SIZE - (ALLOC_ALIGN - 1)` is algebraically `size > SIZE_MAX - 31`. Probe confirmed the **exact** boundary: `malloc(SIZE_MAX - 31)` is accepted (would need to succeed logically, though it will still fail downstream at `mmap` for lack of address space — that failure path is separately covered by `mmap_failed`), while `malloc(SIZE_MAX - 30)` (one single byte past the documented safe threshold) is correctly **rejected** by the guard before any `size_t` arithmetic can wrap. `malloc(SIZE_MAX)` (the degenerate worst case) is also rejected. No wraparound-to-undersized-buffer path exists — this closes out the `W12` self-flagged re-verification cleanly.

**7. Allocator correctness across the full lifecycle / Corretude do alocador ao longo do ciclo de vida completo** (`src/alloc.c`) — `malloc(0) -> NULL`, `realloc(NULL, size) == malloc(size)`, `realloc(ptr, 0)` frees (no-op) and returns `NULL`, `realloc` grow/shrink both preserve exactly `min(old_size, new_size)` bytes and leave the new block fully usable up to its own requested size (no under-allocation), all confirmed under ASan (which would flag any heap-buffer-overflow on the `memset` calls that fill each block to its exact requested size). ~20 000 sequential 64-byte allocations (forcing several 1 MiB arena `mmap` rollovers, per ADR-0004's bump-over-`mmap` strategy) all remained independently readable/writable with no cross-arena corruption.

**8. `mini_vsnprintf`/`mini_printf` truncation contract / Contrato de truncamento do `mini_vsnprintf`/`mini_printf`** (`src/printf.c`) — `cap == 0` never dereferences `buf` (tested with `buf == NULL`); `cap == 1` writes only the NUL terminator; truncation mid-conversion (a `%d` split across the buffer boundary) never writes past `buf[cap-1]` while still reporting the full C99-`vsnprintf`-style logical length (`return_value >= cap` truncation-detection contract holds); `%s` with a `NULL` argument prints the documented `"(null)"` literal instead of dereferencing; the internal 256-byte flush buffer's "run larger than the whole buffer" path (`mini_printf("%s", <999-byte string>)`) was exercised end-to-end through a real `sys_write`-equivalent (pipe) and reproduced the input byte-for-byte with the correct reported write count — no truncation, no corruption, no OOB write into the 256-byte scratch buffer.

| 1 | `src/conv.c:596-620` (`atof`, fractional-digit accumulation loop) | **EN:** `atof`'s own file header claims the DoS-defensive caps (`FCONV_MAX_FRAC_DIGITS`=400, `FCONV_MAX_SCALE_ITER`=400) are "FAR beyond any exponent a legitimate finite/denormal double could need... neither cap ever changes the result for real input — they only bound worst-case WORK". This is **empirically false for a class of legitimate-looking decimal literals**: a string like `"0."` followed by **N nines** (`N` in `[309, 400]`, i.e. well INSIDE the documented "safe, generous" cap, not just at its outer edge) round-trips to values **strictly ≥ 1.0** (mathematically impossible — `0.999...9` is always `< 1.0` for any finite digit count) and, from `N=309` onward, **silently overflows to `+Infinity`** instead of the correct value just under `1.0`.
