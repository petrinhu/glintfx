# glintfx v0.2.3 — data-model (data-binding) — Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** A **v0.2.3** entrega duas features pedidas pelo GusWorld (cockpit), ambas aditivas:
1. **Data-model (data-binding do RmlUi)** por uma API pública limpa em `glintfx::UiLayer` e `glintfx::App` (sem vazar tipos RmlUi), para ligar valores vivos: HP/AP/Mana (números), nome/verbo/alvo (strings), estado (bool) e o **log de combate** (lista de strings). Tarefas T1-T3.
2. **Texturas PNG/JPG** (T4): hoje o `LoadTexture` do backend GL3 do RmlUi só decodifica TGA, então PNG/JPG viram caixa branca (moldura + retrato do cockpit). Override do `LoadTexture` com stb_image + premultiply de alpha.

Tudo funciona nos dois modos (embed + standalone), via `Engine`/`Bootstrap` e o backend GL3 compartilhado.

**Architecture:** O RmlUi liga variáveis por **ponteiro** (`DataModelConstructor::Bind(name, T*)`) e o ponteiro precisa **sobreviver** ao modelo; tudo deve ser registrado **antes** do `LoadDocument` (o documento referencia `data-model="hud"` + `{{var}}` / `data-for`). Como a fachada é key-value (o consumidor passa valores por chave, não ponteiros), o glintfx **possui** o armazenamento estável de cada chave (uma célula por chave, endereço estável) e faz `Bind` uma vez; em `set_*` muta a célula e marca a variável suja (`DataModelHandle::DirtyVariable`), refletida no próximo `update()`. Um `DataBinder` interno encapsula isso, dono pelo `Engine`. Nenhum tipo RmlUi aparece nos headers públicos.

**Tech Stack:** C++ (CMake ≥3.16), RmlUi @`2cd28864ae25ed345b70598751703a5433b12356`, gl3w vendorizado, OpenGL 3.3, FreeType. Testes sob Xvfb.

**Achados de verificação (fonte pinado `glintfx/build/_deps/rmlui-src/Include/RmlUi/Core/`):**
- `Context::CreateDataModel(name, type_register=nullptr, allow_missing_variables=false)` → `DataModelConstructor` (`Context.h:261`); `GetDataModel`/`RemoveDataModel` (`:266,:275`).
- `DataModelConstructor::Bind<T>(name, T* ptr)` liga uma variável a um **ponteiro** (`DataModelHandle.h:41-46`); escalares **arithmetic + String são builtin** (não precisam de `RegisterScalar` — `DataModelHandle.h:119-120`). `bool`/`int`/`double`/`Rml::String` ligam direto.
- Listas: `RegisterArray<Container>()` (`DataModelHandle.h:90-91`) registra `std::vector<T>`; depois `Bind(name, &vector)`. O `value_type` (aqui `Rml::String`) já é builtin.
- Atualização: `DataModelConstructor::GetModelHandle()` → `DataModelHandle`; `DirtyVariable(name)`/`DirtyAllVariables()` (`DataModelHandle.h:18-20`); refletido no próximo `Context::Update()`.
- **Ciclo de vida:** `CreateDataModel` + todos os `Bind` **antes** de `LoadDocument`. `Rml::String` == `std::string`.
- **Texturas (T4):** `LoadTexture(Vector2i& dims, const String& source)` e `GenerateTexture(Span<const byte>, Vector2i)` são `virtual override` em `RenderInterface_GL3` (`Backends/RmlUi_Renderer_GL3.h:40-41`; base `Include/RmlUi/Core/RenderInterface.h:54,59`). Upstream só decodifica TGA → override de `LoadTexture` numa subclasse + stb_image + premultiply, delegando o upload ao `GenerateTexture` da base. stb **não** está vendorizado no glintfx (só gl3w) → vendorizar.

**Referências:** `glintfx/src/{engine,bootstrap,ui_layer,app}.cpp`, headers públicos, `docs/embed-integration.md`, `docs/superpowers/specs/2026-06-30-glintfx-v2-design.md`.

## API escolhida (decisão de porta-de-mão-única)

Superfície pública (neutra), em `UiLayer` e `App`:

```cpp
// --- declaração (chamar ANTES de load) -----------------------------------
bool create_data_model(const char* name);                 // cria o modelo no contexto
bool bind_number(const char* key, double initial = 0.0);  // liga uma célula numérica
bool bind_string(const char* key, const char* initial = "");
bool bind_bool  (const char* key, bool initial = false);
bool bind_list  (const char* key);                        // lista de strings (log)

// --- atualização (chamar DEPOIS de load) ---------------------------------
void set_number(const char* key, double value);
void set_string(const char* key, const char* value);
void set_bool  (const char* key, bool value);
void set_list  (const char* key, const char* const* items, size_t count); // substitui a lista inteira
```

**Trade-offs pesados e por que esta forma:**
- **Key-value (escolhida) vs struct (`RegisterStruct` + Bind de um struct do host):** struct exige o consumidor definir um tipo C++, mantê-lo vivo, e **vazaria** o tipo na borda; key-value mantém o header livre de tipos RmlUi e dá uma superfície uniforme e neutra de linguagem (bom para um futuro binding C). Custo: bookkeeping interno (um registro de células), escondido no `Impl`. **Vale.**
- **Declarar-antes-do-load (escolhida) vs dinâmico (bind a qualquer hora):** o RmlUi **exige** `Bind` antes de `LoadDocument` (as data-views do documento são compiladas no load contra o modelo existente); ligar variável depois do load **não** religa as views já compiladas. Logo o "dinâmico pós-load" briga com o motor e seria frágil. A ergonomia é preservada por uma ordem clara (`create_data_model` → `bind_*` → `load` → `set_*`), validada em runtime (declarar após `load` retorna `false`).
- **Setters tipados explícitos (`set_number/string/bool`) vs `set_data(...)` sobrecarregado:** sobrecarga `set_data(key,double)` vs `set_data(key,bool)` tem **ambiguidade** com literais inteiros (`set_data(key,0)`); nomes explícitos eliminam o footgun e ficam C-ABI-friendly. **Escolhidos.**
- **`set_list` substitui a lista inteira (escolhida) vs push/clear:** substituição é atômica, sem estado parcial; para um log o host mantém seu próprio ring buffer e empurra as N linhas correntes por frame. Menos métodos, mais robusto.
- **`UiLayerConfig.data_model` (rejeitado):** ordenar via campo de config esconde o constraint "antes do load"; uma chamada explícita `create_data_model` torna a ordem óbvia. Config fica mínimo.

```cpp
namespace Rml { class Context; }
#include <cstddef>
namespace glintfx {
class DataBinder {
public:
  DataBinder(); ~DataBinder();
  DataBinder(const DataBinder&) = delete; DataBinder& operator=(const DataBinder&) = delete;
  bool create(Rml::Context* ctx, const char* name);   // CreateDataModel + RegisterArray<vector<String>>
  bool bind_number(const char* key, double initial);
  bool bind_string(const char* key, const char* initial);
  bool bind_bool  (const char* key, bool initial);
  bool bind_list  (const char* key);
  void set_number(const char* key, double v);
  void set_string(const char* key, const char* v);
  void set_bool  (const char* key, bool v);
  void set_list  (const char* key, const char* const* items, std::size_t n);
private:
  struct Impl; Impl* impl_;   // mapas de unique_ptr<double/string/bool/vector<string>> + ctor/handle
};
}
```
`data_binder.cpp` (inclui `<RmlUi/Core.h>`): `Impl` guarda `DataModelConstructor ctor; DataModelHandle handle; bool created=false;` + `std::map<std::string, std::unique_ptr<double>> nums; ...strings(Rml::String); ...bools; ...lists(std::vector<Rml::String>)`.

### Task 4: Texturas PNG/JPG via stb_image (override de `LoadTexture`, com premultiply)

**Files:**
- Create: `glintfx/third_party/stb/stb_image.h` (vendorizado, domínio público), `glintfx/src/stb_image_impl.cpp` (define `STB_IMAGE_IMPLEMENTATION`), `glintfx/tests/texture_png_alpha.cpp`, `glintfx/tests/ui/tex.rml`, `glintfx/tests/ui/circle_rgba.png` (PNG 32-bit com alpha)
- Modify: `glintfx/src/render_gl3.hpp/.cpp` (subclasse com override de `LoadTexture`), `glintfx/CMakeLists.txt` (incluir `third_party/stb` + compilar `stb_image_impl.cpp`), `glintfx/tests/CMakeLists.txt`

**Problema:** o `RenderInterface_GL3::LoadTexture` do RmlUi só decodifica **TGA** (upstream fetchado, NÃO vendorizado — não dá pra patchar). PNG/JPG caem no fallback (caixa branca) no cockpit (moldura + retrato).

**Abordagem cravada (override de método virtual):** `LoadTexture(Vector2i& dims, const String& source)` e `GenerateTexture(Span<const byte>, Vector2i)` são `virtual override` no `RenderInterface_GL3` (`Backends/RmlUi_Renderer_GL3.h:40-41`; base em `Include/RmlUi/Core/RenderInterface.h:54,59`). Criamos uma **subclasse** `Gl3RenderInterface : public RenderInterface_GL3` que **sobrescreve só `LoadTexture`**: decodifica o arquivo com `stb_image` (RGBA), **premultiplica o alpha**, e delega o upload ao **`GenerateTexture` da base** (reusa o caminho GL de criação de textura + mipmaps + filtro do upstream — não reimplementamos GL). Em falha do stb, faz **fallback** para `RenderInterface_GL3::LoadTexture` da base (mantém o TGA do upstream sem regressão). O wrapper `RenderGl3` passa a possuir a subclasse no lugar do tipo upstream; o header público de `render_gl3` **não muda** (segue forward-decl de `Rml::RenderInterface`).

- [ ] **Step 3: Implementar a subclasse `Gl3RenderInterface`**
`render_gl3.cpp` (interno; inclui `<GL/gl3w.h>`, `RmlUi_Renderer_GL3.h`, `stb_image.h`, `<RmlUi/Core/FileInterface.h>`):
```cpp
class Gl3RenderInterface : public RenderInterface_GL3 {
public:
  Rml::TextureHandle LoadTexture(Rml::Vector2i& dims, const Rml::String& source) override {
    // EN: read bytes via RmlUi's FileInterface (respects asset resolution), decode with stb.
    // PT: lê bytes via FileInterface do RmlUi (respeita a resolução de asset), decodifica com stb.
    Rml::FileInterface* fi = Rml::GetFileInterface();
    Rml::FileHandle fh = fi ? fi->Open(source) : 0;
    if (!fh) return RenderInterface_GL3::LoadTexture(dims, source); // EN: fallback (TGA/base).
    fi->Seek(fh, 0, SEEK_END); size_t len = (size_t)fi->Tell(fh); fi->Seek(fh, 0, SEEK_SET);
    std::vector<unsigned char> buf(len); fi->Read(buf.data(), len, fh); fi->Close(fh);
    int w=0,h=0,n=0;
    unsigned char* px = stbi_load_from_memory(buf.data(), (int)len, &w, &h, &n, 4); // force RGBA
    if (!px) return RenderInterface_GL3::LoadTexture(dims, source);                 // EN: fallback.
    // EN: premultiply alpha (GL3 backend composites premultiplied) — avoids edge halo.
    // PT: premultiplica o alpha (backend GL3 compõe premultiplied) — evita halo na borda.
    for (int i = 0; i < w*h; ++i) {
      unsigned a = px[i*4+3];
      px[i*4+0] = (unsigned char)(px[i*4+0]*a/255);
      px[i*4+1] = (unsigned char)(px[i*4+1]*a/255);
      px[i*4+2] = (unsigned char)(px[i*4+2]*a/255);
    }
    dims = Rml::Vector2i(w, h);
    Rml::TextureHandle handle = RenderInterface_GL3::GenerateTexture(
        Rml::Span<const Rml::byte>(reinterpret_cast<const Rml::byte*>(px), (size_t)w*h*4), dims);
    stbi_image_free(px);
    return handle;
  }
};
```
`RenderGl3::Impl` passa a conter `Gl3RenderInterface renderer;` (no lugar de `RenderInterface_GL3 renderer;`). `iface()` segue retornando `&impl_->renderer`. Nada mais muda (begin/end frame idênticos).
> Confirmar no fonte pinado a API do `FileInterface` (`Open/Seek/Tell/Read/Close`, `Include/RmlUi/Core/FileInterface.h`) e a assinatura de `GenerateTexture`/`Span<const byte>`; ajustar se divergir. `Rml::byte` == `unsigned char`.

## Notas de execução

- **Ciclo de vida é o risco central:** `create_data_model` + `bind_*` ANTES de `load`; `set_*` DEPOIS. O `Engine` valida a ordem (retorna false fora de ordem). O constraint vem do RmlUi (views compiladas no load), não é escolha nossa.
- **Ponteiro estável:** as células do `DataBinder` (`unique_ptr` por chave) garantem endereço estável durante a vida do modelo; nunca usar realocação que invalide os ponteiros já passados a `Bind`. `std::map` (não `vector`) preserva endereço dos nós.
- **RmlUi global (N3):** `App` e `UiLayer` não vivos ao mesmo tempo no mesmo processo (relevante no `data_model_smoke`).
- **Zero impacto no default:** sem `create_data_model`, nada de data-model é criado; os 13 testes não tocam o caminho novo.
- **A confirmar no fonte pinado:** sintaxe de `{{var}}` em texto e `data-for="x : list"` (`Source/Core/DataView*`); `CreateDataModel(..., allow_missing_variables=true)` para tolerar `{{var}}` ainda não setada sem erro de log.
- **Texturas (T4):** a mudança é 100% interna ao `render_gl3` (subclasse + override) — o header público não muda; vale para `App` e `UiLayer` (mesmo backend GL3) sem código extra. O **premultiply** é a pegadinha: o backend GL3 compõe premultiplied (ver `docs/embed-integration.md`), o stb devolve alpha straight → premultiplicar antes do upload, senão PNG transparente ganha halo. Ler os bytes via `Rml::GetFileInterface()` (não `stbi_load` por path) para respeitar a resolução de asset do RmlUi. **Fallback** para a base em qualquer falha de decode → TGA do upstream não regride. **Numeração:** a tarefa de texturas é a T4 e a de docs virou T5 (sequência limpa, sem buraco); se preferir a etiqueta literal "T5 texturas" eu troco.
