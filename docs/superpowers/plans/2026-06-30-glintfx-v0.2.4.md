# glintfx v0.2.4 — UA-stylesheet + version() + CI hygiene — Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** A **v0.2.4** entrega três correções pedidas pelo consumidor (GusWorld), todas de correção/DX, zero breaking change:

1. **UA-stylesheet (T1, o item principal):** hoje **nem** `App` **nem** `UiLayer` carregam uma stylesheet User-Agent — o RmlUi tem `display: inline` como default para **todo** elemento (diferente do HTML, onde `div`/`p`/`h1`/`section`/... são block por padrão), então todo consumidor precisa declarar `display: block` manualmente em cada tag estrutural ou o espaçamento vertical (`margin`/`padding`) vira no-op silencioso. O 1º consumidor (GusWorld) apanhou disso no cockpit. glintfx passa a aplicar uma UA-stylesheet mínima embutida a **todo documento carregado, nos DOIS modos** (App + UiLayer), como base de baixa especificidade — o autor sempre pode sobrescrever.
2. **`version()` CMake-driven (T2):** hoje retorna `"0.1.0"` hardcoded (`glintfx/src/app.cpp:25`), defasado desde a v0.1.1. Passa a ler `GLINTFX_VERSION`, gerado pelo CMake a partir de `project(glintfx VERSION ...)` — fonte única que nunca defasa.
3. **CI hygiene (T3):** o comentário dos workflows (`.forgejo/` e `.github/`) diz "10 testes" / "5 testes embed" — hoje são **17** (GLFW=ON) / **7** (GLFW=OFF). Corrige o comentário (não muda comportamento) + documenta a UA-stylesheet em `docs/embed-integration.md` + `CHANGELOG.md`.

**Architecture (UA-stylesheet, T1):** Investigação do fonte pinado do RmlUi (`glintfx/build/_deps/rmlui-src/`) confirma: **não existe** um mecanismo de stylesheet global/default embutido no RmlUi — cada `ElementDocument` só recebe as sheets que o próprio RML declara via `<link>`/`<style>` (`ElementDocument::ProcessHeader`, `Source/Core/ElementDocument.cpp:142`); sem isso, `new_style_sheet` fica nulo e o documento roda só com os defaults por-propriedade do C++ (`display: inline` para todos — `StyleSheetSpecification.cpp:306`). O mecanismo limpo (sem exigir `@import` do consumidor) é aplicar a UA-sheet **depois** de `Context::LoadDocument()` retornar o `doc`, via `StyleSheetContainer::CombineStyleSheetContainer()` — API pública (`Include/RmlUi/Core/StyleSheetContainer.h`), o **mesmo** mecanismo que o próprio RmlUi usa para mesclar múltiplos `<link>` de um documento. `ua_sheet->CombineStyleSheetContainer(doc_sheet_existente_ou_vazia)` produz um container **novo**, não-compilado, com a UA-sheet como base e o `doc`'s próprio sheet mesclado **por cima**; a especificidade de empate é resolvida a favor de quem foi mesclado **por último** (`StyleSheet::MergeStyleSheet`'s acúmulo de `specificity_offset`, `Source/Core/StyleSheet.cpp:39`) — ou seja, o autor **sempre** vence em empate, exatamente como CSS/browser. `doc->SetStyleSheetContainer(combined)` troca o container e marca `DirtyMediaQueries()` (recompila no próximo `Context::Update()`). Aplicado uma única vez, no `Bootstrap::load()` — ponto compartilhado por `Engine::load()`, por sua vez usado tanto por `App::load()` quanto por `UiLayer::load()` — logo **um único ponto de código cobre os dois modos**, sem duplicação. A UA-sheet fica embutida como string C++ (`src/ua_stylesheet.hpp`, header interno, nunca exposto), parseada **uma vez** por lifetime do `Bootstrap` (em `init()`, logo após `Rml::Initialise()`) e nunca compilada diretamente (só usada como base de `CombineStyleSheetContainer`) — assim permanece reutilizável em todo `load()` subsequente sem violar o `RMLUI_ASSERT(!compiled_style_sheet)` de `MergeStyleSheetContainer`. Decisão: **sempre-on**, sem opt-out — é o comportamento esperado de "browser-like" e o autor sempre pode sobrescrever com uma regra de especificidade igual ou maior; não introduz uma opção de config para um caso que não tem motivo real para ser desligado (YAGNI).

**Tech Stack:** C++ (CMake ≥3.16), RmlUi @`2cd28864ae25ed345b70598751703a5433b12356`, gl3w vendorizado, OpenGL 3.3, FreeType. Testes sob Xvfb.

**Achados de verificação (fonte pinado `glintfx/build/_deps/rmlui-src/`):**
- `StyleSheetSpecification.cpp:306`: `RegisterProperty(PropertyId::Display, "display", "inline", ...)` — confirma o default `inline` para TODO elemento (não há mapa por-tag embutido).
- `Source/Core/ElementDocument.cpp:142-210` (`ProcessHeader`): cada `<link>`/`<style>` do RML é mesclado em ordem via `StyleSheetContainer::MergeStyleSheetContainer`; sem nenhum, o documento fica sem `style_sheet_container` (nulo) — não há default global do lado do RmlUi.
- `Include/RmlUi/Core/StyleSheetContainer.h`: `CombineStyleSheetContainer(const StyleSheetContainer&) const` (não-mutante, retorna `SharedPtr` novo) e `MergeStyleSheetContainer(const StyleSheetContainer&)` (mutante). `LoadStyleSheetContainer(Stream*, int begin_line_number = 1)` faz o parse.
- `Source/Core/StyleSheetContainer.cpp:163-177` (`CombineStyleSheetContainer`): clona `this` (cópia rasa dos `media_blocks`) e mescla o argumento **por cima** — ou seja, `base.CombineStyleSheetContainer(overlay)` → `overlay` vence empates. `MergeStyleSheetContainer` tem `RMLUI_ASSERT(!compiled_style_sheet)` — só bloqueia re-merge num container **já compilado**; o clone criado por `CombineStyleSheetContainer` é sempre uma instância nova e não-compilada, então é seguro chamar em todo `load()`.

- `Include/RmlUi/Core/StreamMemory.h`: `StreamMemory(const byte* buffer, size_t buffer_size)` — construtor read-only sobre buffer existente; usado internamente pelo RmlUi para `<style>` inline (`ElementDocument.cpp:180`), mesmo padrão que usaremos para a string embutida.

```cpp
// EN: Scans the framebuffer for pure-red and pure-blue pixels and records their row
//     range. If the two divs are inline (today's bug, no UA-stylesheet), they share the
//     same line box -- their row ranges OVERLAP. If they are block (fixed by T1), they
//     stack -- their row ranges are DISJOINT. Order-agnostic (doesn't assert which one
//     is "on top"): disjointness alone is the proof, robust to either glReadPixels row
//     convention.
// PT: Varre o framebuffer por pixels vermelho/azul puros e registra a faixa de linhas.
//     Se os 2 divs são inline (bug de hoje, sem UA-stylesheet), compartilham a mesma
//     line box -- as faixas de linha SE SOBREPÕEM. Se são block (corrigido por T1),
//     empilham -- as faixas ficam DISJUNTAS. Não assume qual fica "em cima": a
//     disjunção sozinha é a prova, robusta a qualquer convenção de linha do glReadPixels.
struct RowRange { int min_row = -1, max_row = -1; size_t count = 0; };

static bool is_red (const unsigned char* p) { return p[0] > 150 && p[1] < 80  && p[2] < 80;  }
static bool is_blue(const unsigned char* p) { return p[2] > 150 && p[0] < 80  && p[1] < 80;  }

static RowRange scan(const std::vector<unsigned char>& px, int w, int h,
                      bool (*pred)(const unsigned char*)) {
  RowRange r;
  for (int row = 0; row < h; ++row)
    for (int col = 0; col < w; ++col) {
      const unsigned char* p = &px[(size_t)(row * w + col) * 3];
      if (pred(p)) { if (r.min_row < 0) r.min_row = row; r.max_row = row; ++r.count; }
    }
  return r;
}

int main() {
  constexpr int W = 200, H = 160;
  glintfx::WindowGlfw host;
  if (!host.create("ua_stylesheet_sanity_host", W, H)) { /* FAIL 1 */ }
  glintfx::UiLayer ui({ .logical_width = W, .logical_height = H, .load_gl = true });
  if (!ui.ok()) { /* FAIL 2 */ }
  ui.load("ua_stylesheet_scene.rml");
  // ... 2x warm-up update()+render(), glReadPixels FBO0 GL_RGB (ver capture() de dp_ratio_sanity.cpp) ...
  auto red  = scan(px, W, H, is_red);
  auto blue = scan(px, W, H, is_blue);
  if (red.count < 500 || blue.count < 500) { /* FAIL: div não renderizou -- return 3 */ }
  const bool disjoint = (red.max_row < blue.min_row) || (blue.max_row < red.min_row);
  if (!disjoint) { /* FAIL: linhas se sobrepõem -- divs NÃO empilharam -- return 4 */ }
  std::puts("ua_stylesheet_sanity: PASS");
  return 0;
}
```

```cpp
bool Bootstrap::load(const char* rml_path) {
  if (!impl_ || !impl_->ctx) return false;
  Rml::ElementDocument* doc = impl_->ctx->LoadDocument(rml_path);
  if (!doc) return false;
  if (impl_->ua_style_sheet) {
    // EN: Merge UA base (low specificity) under the document's own sheet (merged ON TOP
    //     -- StyleSheet::MergeStyleSheet's specificity_offset accumulation resolves ties
    //     in favour of whoever is merged LAST). Works with or without the document's own
    //     <link>/<style>.
    // PT: Mescla a base UA (baixa especificidade) sob a sheet própria do documento
    //     (mesclada POR CIMA -- o acúmulo de specificity_offset em
    //     StyleSheet::MergeStyleSheet resolve empates a favor de quem é mesclado POR
    //     ÚLTIMO). Funciona com ou sem <link>/<style> próprio do documento.
    const Rml::StyleSheetContainer* existing = doc->GetStyleSheetContainer();
    auto combined = impl_->ua_style_sheet->CombineStyleSheetContainer(
        existing ? *existing : Rml::StyleSheetContainer());
    doc->SetStyleSheetContainer(std::move(combined));
  }
  doc->Show();
  return true;
}
```

```markdown
## 9. UA stylesheet (display: block defaults) / Stylesheet UA (defaults de display: block)

**EN:** RmlUi's built-in `display` default is `inline` for every element (unlike HTML,
where `div`/`p`/`h1`.../`section`/... are block by default). glintfx merges a small
embedded User-Agent stylesheet into every document loaded by `load()` (both `App` and
`UiLayer` -- shared code path in `Bootstrap::load()`):
```
div, p, h1, h2, h3, h4, h5, h6, ul, ol, li,
section, article, header, footer, nav, main { display: block; }
```
It is applied as a **low-specificity base**: any author rule of equal or higher
specificity in your own RCSS overrides it (this is standard CSS cascade behaviour, not
a special case). You do not need to `@import` anything or opt in -- it is always on.
Elements outside this list (e.g. `span`, custom tags) still default to `inline`, matching
RmlUi's own behaviour; add your own `display` rule for those if needed.

**PT:** O padrão embutido do RmlUi para `display` é `inline` para todo elemento
(diferente do HTML, onde `div`/`p`/`h1`.../`section`/... são block por padrão). O
glintfx mescla uma pequena stylesheet User-Agent embutida em todo documento carregado
por `load()` (tanto `App` quanto `UiLayer` -- caminho de código compartilhado em
`Bootstrap::load()`):
```
div, p, h1, h2, h3, h4, h5, h6, ul, ol, li,
section, article, header, footer, nav, main { display: block; }
```
É aplicada como **base de baixa especificidade**: qualquer regra do autor com
especificidade igual ou maior no seu próprio RCSS a sobrepõe (comportamento padrão de
cascata CSS, não um caso especial). Não é necessário `@import` nem opt-in -- está sempre
ativa. Elementos fora desta lista (ex.: `span`, tags customizadas) continuam com default
`inline`, igual ao próprio RmlUi; adicione sua própria regra `display` se precisar.
```

## Notas de execução

- **Mecanismo cravado (T1):** `StyleSheetContainer::CombineStyleSheetContainer()` -- API pública do RmlUi, o MESMO mecanismo que o próprio `ElementDocument::ProcessHeader` usa para mesclar múltiplos `<link>` de um documento. Não é gambiarra: é reuso do caminho oficial de composição de stylesheets, aplicado uma camada acima (fora do parse do RML, depois que `LoadDocument()` retorna). A prova de empate-a-favor-do-último vem de `StyleSheet::MergeStyleSheet`'s `specificity_offset` (nível compilado) -- documentado no `Architecture` acima com linha exata do fonte pinado.
- **Um só ponto de aplicação, dois modos:** `Bootstrap::load()` é chamado por `Engine::load()`, que por sua vez é chamado tanto por `App::load()` quanto por `UiLayer::load()`. Não há necessidade de duplicar a lógica -- e o teste único (`ua_stylesheet_sanity`, via `UiLayer`) já prova o caminho compartilhado; `App` herda o mesmo comportamento por construção (mesmo `Bootstrap`), sem precisar de um segundo teste (mesmo padrão de `dp_ratio_sanity`/`base_url_sanity`, que também só testam via `UiLayer`).
- **Fail-open:** se o parse da UA-sheet embutida falhar (não deveria -- é uma string cravada e testada), `ua_style_sheet` fica nulo e `load()` simplesmente pula o merge -- nunca bloqueia o carregamento de um documento por causa de um problema na sheet interna.
- **Gap conhecido, fora de escopo (T2):** `version()` só existe hoje em builds com `GLINTFX_BACKEND_GLFW=ON` (`app.hpp` só é incluído nesse caso -- ver `glintfx.hpp`). Mover `version()` para um header sempre-presente (ex.: `ui_layer.hpp` ou um novo header "core") tornaria a versão consultável também em builds embed-only, mas é uma mudança de superfície pública (onde a função é declarada) fora do pedido literal desta v0.2.4 ("ler a versão injetada", não "tornar disponível em todo build"). Fica como candidato para uma v0.2.x futura se o líder confirmar que é necessário (é two-way-door: aditivo, sem quebra, fácil de fazer depois).
- **Ordem T1 vs T3:** T3/Step 1 (contagem de testes no CI) assume que T1 já rodou (conta `ua_stylesheet_sanity`). Se as tasks forem executadas fora de ordem, ajustar os números (17/7 sem T1, 18/8 com T1) -- o texto do Step 1 já sinaliza isso.
- **Escopo (YAGNI):** sem opção de opt-out da UA-sheet, sem expandir a lista de tags além do pedido (`ul`/`ol`/`li` cobre "defaults de lista" no sentido de display; margens/paddings de lista tipo HTML NÃO entram -- não foram pedidos e mudariam visualmente documentos existentes sem necessidade real).
