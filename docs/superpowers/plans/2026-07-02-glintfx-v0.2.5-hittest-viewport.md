# glintfx v0.2.5 — click callback + element geometry + viewport origin — Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** A **v0.2.5** entrega três pedidos reais do 1º consumidor (GusWorld, embed mode via `UiLayer`), aprovados pelo líder, que fecham a dívida do cockpit hand-rolled em geometria/hit-test:

1. **F1 -- callback de clique (`set_click_callback`):** hoje um clique injetado via `process_event` chega ao RmlUi (hover/hit-test internos já funcionam) mas **nada volta ao host** -- o GusWorld espelha à mão a geometria dos botões do RCSS num POCO C++ pra saber o que foi clicado, dívida que desalinha silenciosamente a cada mudança de RCSS. `set_click_callback(std::function<void(const char*)>)` em `UiLayer` **e** `App` reporta o `id` do elemento clicado (fase bubble, sobe a árvore até achar o ancestral mais próximo com `id` não-vazio; reporta `""` se nenhum tiver).
2. **F2 -- query de geometria (`get_element_box`):** `get_element_box(const char* id)` retorna x/y/w/h (px físicos, espaço-janela) de um elemento pelo `id`, ou `found=false`. Paridade `UiLayer`+`App`. Nenhum tipo RmlUi vaza no header público (`ElementBox` é struct simples, novo header `element_box.hpp`).
3. **F3 -- `set_viewport` com origem (GAP-2 da INBOX, ativado):** hoje o embed compõe SEMPRE no FBO 0 com origem `(0,0)` (constraint documentada no ADR-0008/F1). O GusWorld precisa de letterbox real: UI numa sub-região da janela. Nova sobrecarga `UiLayer::set_viewport(int x, int y, int w, int h, int target_h)`; a sobrecarga de 2 args existente **continua funcionando** como `(0,0,w,h)`. **UiLayer-only** (ver Architecture -- `App` não ganha este método, por design). É a tarefa arquiteturalmente delicada desta release: exige um contrato de coordenadas explícito para não reintroduzir a MESMA classe de bug que a F1 corrige (desalinhamento silencioso).

**Ordem de execução:** F1 -> F2 -> F3 -> Docs (conforme pedido do líder: F1/F2 são pequenas e destravam a dívida do consumidor primeiro; F3 é a maior e depende conceitualmente de F1 -- o teste de F3 usa o callback de F1 como oráculo de hit-test -- e de F2 -- usa `get_element_box` como oráculo determinístico de geometria). Cada task é independentemente committable; F2 depende de F1 só para reuso de teste, não de código.

## Architecture

### F1 (callback de clique)

Mecanismo cravado: `Rml::Element::AddEventListener(Rml::EventId::Click, listener, /*in_capture_phase=*/false)` (fase bubble) na **raiz do documento** (`doc`), com um `Rml::EventListener` concreto (`ClickEventListener`, interno a `bootstrap.cpp`) que em `ProcessEvent` lê `event.GetTargetElement()` e sobe por `GetParentNode()` até achar um `Element` com `GetId()` não-vazio (inclusive o próprio alvo), reportando `""` se nenhum ancestral (até a raiz do documento) tiver `id`. Lifetime: idioma padrão do próprio RmlUi (confirmado nos samples pinados -- `Samples/basic/{benchmark,animation,harfbuzz,demo}/src/*.cpp`: `new Listener(...)` + `void OnDetach(Element*) override { delete this; }`) -- o listener se autodeleta quando o elemento/documento dono é destruído (via `Rml::Shutdown()` ou um futuro unload), sem gerência manual de lifetime do lado do glintfx. O callback do host (`std::function<void(const char*)>`) fica armazenado em `Bootstrap::Impl::click_cb`; o `ClickEventListener` guarda um `std::function<...>*` que **aponta pra dentro do `Impl`** (não uma cópia) -- lido em `ProcessEvent`, nunca em `OnDetach` -- então `set_click_callback()` pode ser chamado **antes OU depois** de `load()` sem restrição de ordem (diferente da API de data-model): o listener sempre lê o valor **corrente** de `click_cb` no momento do clique, não o valor no momento do attach. `Bootstrap::load()` passa a rastrear `impl_->doc` (o último documento carregado) -- suposição de documento único, consistente com o resto da engine (não há API de multi-documento em lugar nenhum hoje); anexa um `ClickEventListener` novo a cada `load()`.

### F2 (`get_element_box`)

`Rml::Element* el = impl_->doc->GetElementById(id)` (herdado de `Element`, busca na subárvore do documento) + `el->GetAbsoluteOffset(Rml::BoxArea::Border)` (posição, `Rml::Vector2f`, espaço do CONTEÚDO -- topo-esquerda do documento, y para baixo, offset-free) + `el->GetBox().GetSize(Rml::BoxArea::Border)` (tamanho). Forma de retorno: `struct ElementBox { bool found; float x,y,w,h; }` (namespace-scope, plain, sem template/`std::optional`) -- decisão de consistência de estilo: a API pública já usa exclusivamente structs simples com defaults (`AppConfig`, `UiLayerConfig`, `UiEvent`), nunca `std::optional`; introduzir `std::optional<T>` só pra este método quebraria esse padrão sem ganho real (`found` é tão explícito quanto `has_value()`). Novo header público `include/glintfx/element_box.hpp`, incluído por `ui_layer.hpp` e `app.hpp` (mesmo padrão de `ui_event.hpp`), pra que `App` e `UiLayer` compartilhem literalmente o mesmo tipo (paridade real, não duas structs coincidentes).

### F3 (viewport com origem) -- o contrato de coordenadas

**Achado-chave (lido no source pinado, não documentado antes no ADR-0008):** `RenderInterface_GL3::SetViewport` **já** tem uma sobrecarga de 4 argumentos no backend upstream: `void SetViewport(int width, int height, int offset_x = 0, int offset_y = 0)` (`RmlUi_Renderer_GL3.h:24`). O offset é armazenado (`viewport_offset_x/y`) e usado **apenas** no blit final de `EndFrame()`: `glBindFramebuffer(GL_FRAMEBUFFER, 0); glViewport(viewport_offset_x, viewport_offset_y, viewport_width, viewport_height);` (`RmlUi_Renderer_GL3.cpp:909-910`), imediatamente antes do quad fullscreen que desenha o resultado pós-processado no backbuffer da janela. Os passes internos (`BeginFrame()`'s `glViewport(0,0,viewport_width,viewport_height)` em `:857`, e a projeção ortográfica `ProjectOrtho(0,w,h,0,-10000,10000)` em `:810`) permanecem **sempre** ancorados em `(0,0)` -- o CONTEÚDO da UI (layout, hit-test, `Context::SetDimensions`) nunca sabe que existe um offset; só o blit final pro backbuffer da janela é deslocado. **Consequência arquitetural direta: o glintfx não precisa reimplementar nada no shader/blit -- só precisa parar de hardcodar `(0,0)` e repassar o offset do host pra essa sobrecarga já existente.**

**O problema real (por que esta é a tarefa delicada):** `glViewport`/`SetViewport`'s `offset_y` é OpenGL nativo -- origem **inferior-esquerda** (bottom-up). Mas TODO o resto do contrato público do glintfx (`UiEvent.x/y` pra mouse, a leitura de `get_element_box`) é **espaço-janela topo-esquerda, y pra baixo** (convenção padrão de windowing/GLFW/SDL, confirmada em `RmlGLFW::ProcessCursorPosCallback` e em `Context::ProcessMouseMove(int x, int y, ...)`, que trata `x,y` cru como o MESMO espaço de `Context::SetDimensions` -- topo-esquerda). Misturar as duas convenções sem conversão explícita reintroduziria a MESMA classe de bug que a F1 resolve (desalinhamento silencioso de coordenadas) -- desta vez no próprio glintfx, não no host.

**Decisão de design (resolve a pergunta do pedido -- "host injeta coords da janela ou já relativas ao viewport?"):**

- **`UiLayer::set_viewport(int x, int y, int w, int h, int target_h)`** (nova sobrecarga): `(x, y)` são espaço-janela **topo-esquerda, y-pra-baixo** -- a MESMA convenção de `UiEvent`. `(w, h)` continuam as dimensões lógicas/de conteúdo da sub-viewport (dimensão do contexto RmlUi), como sempre foram. `target_h` é a altura TOTAL do render-target do host (a janela inteira) em px físicos -- o ÚNICO parâmetro que existe puramente pra deixar o glintfx fazer, internamente, a conversão topo-para-baixo → OpenGL-baixo-para-cima que o `glViewport` exige (`gl_offset_y = target_h - y - h`; `gl_offset_x = x`, sem flip -- o eixo X é idêntico nas duas convenções). **Por que não pedir `target_w` também:** o eixo X não tem flip nenhum entre as duas convenções -- só Y muda de sentido -- então um `target_w` seria parâmetro morto (YAGNI).
- **Todo o resto do contrato público (mouse via `UiEvent`, retorno de `get_element_box`) continua espaço-janela topo-esquerda, y-pra-baixo, SEM EXCEÇÃO.** O host nunca precisa pensar em coordenadas OpenGL pra input ou pra ler geometria -- só uma vez, no único parâmetro `target_h` de `set_viewport`, que é um número que todo host que já possui a própria janela tem trivialmente à mão (não é um cálculo, é `window_height`).
- **`UiLayer` é a ÚNICA camada que conhece esse offset.** `Engine`/`Bootstrap`/`RenderGl3` continuam 100% espaço-de-conteúdo (offset-free) ou GL-nativo puro (`RenderGl3::begin_frame_compose` só repassa `offset_x, offset_y` já convertidos, sem saber de onde vieram). A tradução acontece nos DOIS pontos onde uma coordenada cruza a fronteira pública do `UiLayer`: (a) `process_event`'s `case T::MouseMove`, subtrai `(impl_->x, impl_->y)` das coordenadas do evento antes de repassar a `Context::ProcessMouseMove` (espaço-janela → espaço-conteúdo); (b) `get_element_box`, soma `(impl_->x, impl_->y)` ao resultado de `Engine::get_element_box` (espaço-conteúdo → espaço-janela) antes de devolver ao host. `App` nunca tem offset (é sempre `(0,0)` implícito -- dono da própria janela inteira), então `App::get_element_box` repassa sem tradução -- e por isso **`App` não ganha `set_viewport`**: não haveria uso real (não faz sentido letterboxar dentro da própria janela que o `App` já possui inteira; se um consumidor futuro quiser isso, é aditivo/two-way-door, YAGNI hoje).
- **Compat retroativa:** `set_viewport(w, h)` (2 args, já existente) passa a ser exatamente o caso `x=y=0`, `target_h=h` (que satisfaz `gl_offset_y = h - 0 - h = 0` -- idêntico ao comportamento de hoje, byte a byte). Internamente isso é um "modo legado" (`letterbox_mode=false`) vs. "modo letterbox" (`letterbox_mode=true`, guarda `target_h`), porque um `UiEvent::Type::Resize` injetado precisa saber recalcular `gl_offset_y` quando `h` muda -- ver Task 3, Step 2.

**Alcance da F3 (o que NÃO muda):** o alvo de composição continua **sempre FBO 0** (só a ORIGEM do viewport dentro do FBO 0 muda -- pedido explícito do líder, "só a origem muda, não o FBO"). Um FBO custom do host bindado antes de `render()` continua não sendo usado como alvo (limitação que segue aberta, ver `docs/embed-integration.md` seção 8).

### ADR

O achado acima **refina** um dos "F1 implementation constraints" já documentados no ADR-0008 ("Viewport origin is hardcoded to (0,0)") -- não é uma nova decisão arquitetural (embed mode já foi decidido e aceito; isto é resolver um gap que o próprio ADR-0008 já registrava). Decisão de mecânica de doc: **acrescentar uma seção nova, datada, ao ADR-0008** ("F3 implementation update -- configurable viewport origin (v0.2.5)"), no MESMO padrão já usado pelas seções "Optional GLFW build" (v0.2.1) e "F1 implementation constraints" -- ambas foram acrescentadas ao ADR-0008 **depois** de `Accepted`, sem reabrir a decisão original. Alternativa considerada e descartada: ADR-0009 novo -- rejeitada porque não há trade-off de opções concorrentes aqui (não é uma nova bifurcação de arquitetura), é a MESMA decisão (embed mode) completando um gap que ela mesma já catalogava; um ADR novo fragmentaria o contexto sem necessidade. Nota: o índice `docs/adr/README.md` descreve ADRs como "imutáveis uma vez `Accepted`" -- lido como regra sobre a DECISÃO em si (Context/Decision/Consequences), não sobre gaps de implementação subsequentes já documentados como tais no próprio ADR; o precedente das duas seções acrescentadas antes confirma essa leitura.

`glintfx/tests/click_callback_sanity.cpp` (modelo: `ui_layer_events.cpp` pra injeção de evento; captura o callback num `std::vector<std::string>`):
```cpp
// SPDX-License-Identifier: MPL-2.0
// EN: Verifies UiLayer::set_click_callback(): (A) direct id on the clicked element; (B) click
//     on a child WITHOUT an id bubbles up to report the nearest ancestor id ("panel"); (C) click
//     on a subtree with NO id anywhere reports "" (never a crash, never silently dropped).
//     A click is simulated as MouseMove (to hover/set mouse_position) + MouseButtonDown + Up at
//     the same point -- RmlUi's own click detection (down+up on the same element == click).
// PT: Verifica UiLayer::set_click_callback(): (A) id direto no elemento clicado; (B) clique num
//     filho SEM id sobe até reportar o ancestral mais próximo com id ("panel"); (C) clique numa
//     subárvore SEM id em lugar nenhum reporta "" (nunca crash, nunca descartado silenciosamente).
//     Um clique é simulado como MouseMove (hover/mouse_position) + MouseButtonDown + Up no mesmo
//     ponto -- a própria detecção de clique do RmlUi (down+up no mesmo elemento == click).
// Copyright (c) 2026 Petrus Silva Costa
#include "../src/window_glfw.hpp"
#include <glintfx/glintfx.hpp>
#include <cstdio>
#include <string>
#include <vector>

static void click_at(glintfx::UiLayer& ui, float x, float y) {
  ui.process_event({ .type = glintfx::UiEvent::Type::MouseMove, .x = x, .y = y });
  ui.process_event({ .type = glintfx::UiEvent::Type::MouseButton, .button = 0, .pressed = true });
  ui.process_event({ .type = glintfx::UiEvent::Type::MouseButton, .button = 0, .pressed = false });
}

int main() {
  glintfx::WindowGlfw host;
  if (!host.create("click_cb_host", 300, 200)) { std::puts("FAIL: host create"); return 1; }

  glintfx::UiLayer ui({ .logical_width = 300, .logical_height = 200, .load_gl = true });
  if (!ui.ok()) { std::puts("FAIL: ui attach"); return 2; }

  std::vector<std::string> hits;
  ui.set_click_callback([&hits](const char* id) { hits.emplace_back(id); });

  if (!ui.load("click_scene.rml")) { std::puts("FAIL: load"); return 3; }
  ui.update(); ui.render();

  // (A) direct id
  click_at(ui, 50.f, 30.f);
  // (B) child without id -> bubbles to "panel"
  click_at(ui, 50.f, 80.f);
  // (C) subtree without any id -> ""
  click_at(ui, 50.f, 130.f);

  if (hits.size() != 3) {
    std::fprintf(stderr, "FAIL: expected 3 hits, got %zu\n", hits.size());
    return 4;
```

```cpp
//     Self-deletes on OnDetach -- RmlUi's own idiom, confirmed in the pinned samples
//     (Samples/basic/{benchmark,animation,harfbuzz,demo}/src/*.cpp).

class ClickEventListener : public Rml::EventListener {
public:
  explicit ClickEventListener(std::function<void(const char*)>* cb) : cb_(cb) {}

  void ProcessEvent(Rml::Event& event) override {
```

```markdown
update is scoped to origin only, not to a custom host FBO target.

full host-facing contract): `set_viewport`'s `(x, y)` are top-down, window-space (the SAME
```

```markdown
## 10. Click callback, element geometry, and the window-space coordinate contract (v0.2.5) / Callback de clique, geometria de elemento e o contrato de coordenadas espaço-janela (v0.2.5)

**EN:** Three additions close the "host mirrors RCSS geometry by hand" gap:

- **`set_click_callback(std::function<void(const char* element_id)>)`** (`UiLayer` + `App`,
  parity): fires on every "click" (bubble phase, attached to the document root by `load()`).
  Reports the id of the click TARGET, or the nearest ANCESTOR with a non-empty id if the
  target itself has none, or `""` if no element in the ancestor chain has an id. No ordering
  constraint versus `load()` -- can be set before or after.
- **`get_element_box(const char* id) -> ElementBox { bool found; float x, y, w, h; }`**
  (`UiLayer` + `App`, parity): border-box geometry (`Rml::BoxArea::Border`) of an element by
  id, or `found=false`.
- **`UiLayer::set_viewport(int x, int y, int w, int h, int target_h)`** (`UiLayer`-only, F3):
  places the composited UI within a sub-region of the host's window (letterbox). The legacy
  `set_viewport(w, h)` is unchanged and equivalent to `(0, 0, w, h, h)`.

**The coordinate contract (read this before using any of the three):** every coordinate that
crosses the public API boundary -- `UiEvent`'s mouse `x`/`y`, `get_element_box()`'s return
value, and `set_viewport`'s `x`/`y` -- is **window/render-target physical pixels, top-left
origin, y-down**. This is the SAME convention regardless of whether a viewport offset is
active. `UiLayer` performs ALL translation to/from its own offset-free content space
internally: it subtracts `(x, y)` from mouse coordinates before forwarding to RmlUi, and adds
`(x, y)` to `get_element_box()`'s raw content-space result before returning it. **The host
never needs to convert coordinates for input or geometry queries.** The ONE place OpenGL's
native bottom-up viewport convention leaks through is `set_viewport`'s `target_h` parameter
(the host's own window height) -- needed only because `glViewport` is unavoidably
bottom-left-origin at the final composite step; see the ADR-0008 "F3 implementation update"
section for the exact formula. `App` never has an offset (it owns its whole window), so
`App::get_element_box()` needs no translation and `App` does not gain `set_viewport`.

**PT:** Três adições fecham a lacuna "o host espelha geometria do RCSS à mão":

- **`set_click_callback(std::function<void(const char* element_id)>)`** (`UiLayer` + `App`,

  alvo não tiver, ou `""` se nenhum elemento na cadeia de ancestrais tiver id. Sem restrição
  de ordem vs. `load()` -- pode ser setado antes ou depois.
- **`get_element_box(const char* id) -> ElementBox { bool found; float x, y, w, h; }`**
  (`UiLayer` + `App`, paridade): geometria border-box (`Rml::BoxArea::Border`) de um elemento
  por id, ou `found=false`.
- **`UiLayer::set_viewport(int x, int y, int w, int h, int target_h)`** (`UiLayer`-only, F3):
  posiciona a UI composta numa sub-região da janela do host (letterbox). O `set_viewport(w,
  h)` legado é inalterado e equivalente a `(0, 0, w, h, h)`.

**O contrato de coordenadas (leia antes de usar qualquer uma das três):** toda coordenada que
cruza a fronteira pública da API -- `x`/`y` de mouse do `UiEvent`, o retorno de
`get_element_box()`, e `x`/`y` de `set_viewport` -- é **pixels físicos do render-target/
janela, origem superior-esquerda, y pra baixo**. É a MESMA convenção independente de haver
offset de viewport ativo. O `UiLayer` faz TODA a tradução de/para seu próprio espaço de
conteúdo offset-free internamente: subtrai `(x, y)` das coordenadas de mouse antes de
repassar ao RmlUi, e soma `(x, y)` ao resultado bruto (espaço-conteúdo) de

input ou queries de geometria.** O ÚNICO lugar onde a convenção nativa bottom-up de viewport

passo final de composição; ver a seção "F3 implementation update" do ADR-0008 pra fórmula
exata. `App` nunca tem offset (possui a janela inteira), então `App::get_element_box()` não
precisa de tradução e `App` não ganha `set_viewport`.
```
