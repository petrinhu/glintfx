## EN

### 1. Goal & scope

glintfx today exposes two families of observable UI activity to the host: **pointer** (`set_click_callback`/`set_click_info_callback`) and **scroll** (`set_scroll_callback`). Forms are currently **inert**: a `<input>`, `<select>`, `<textarea>` or `<form>` can be styled and driven one-way from the host via the data-model (`bind_*`/`set_*`), but the host has no way to learn that the **user** typed, toggled, picked, or submitted something, short of polling. `L1.15-FORMEV` closes this gap by exposing RmlUi's own `Change`/`Submit`/`Focus`/`Blur` events to the host, through the same "no third-party type crosses the public boundary" discipline already enforced for click/scroll.

The scope is deliberately the same four events named in the `TODO.md` entry — no more (see §11 for what is explicitly cut and why).

### 2. Decision point 1 — Callback surface

Three shapes were evaluated, all keeping the "no `Rml::*` type in the public header" rule and the existing `App`+`UiLayer` parity rule.

**Shape A — one generic callback.** `set_form_event_callback(std::function<void(const char* id, FormEventType type, const char* value)> cb)`, where `FormEventType` is a new public `enum class` (`Change, Submit, Focus, Blur`). One registration point, one doc-comment/WARNING block to maintain, smallest header-surface delta (+1 method × 2 classes + 1 enum). Directly matches the anchor use case in §8 (a single "audio-for-event" dispatch table indexed by `type`). Cost: `value` is meaningless for `Focus`/`Blur` (always `""`/`nullptr` — must be documented per-branch instead of per-method), the host must `switch` on `type` internally (the compiler cannot help catch a missing case), and it is the **first** callback in the family that is not verb-named — it breaks the naming idiom `set_<event>_callback` used by every existing entry point.

**Shape B — four specific callbacks.** `set_change_callback(fn(id, value))`, `set_submit_callback(fn(id))`, `set_focus_callback(fn(id))`, `set_blur_callback(fn(id))`. This is the literal extension of the existing family: `set_click_callback`/`set_scroll_callback` are already exactly this shape (id-only, one verb per method), so this is "more of the same," not new design. Each payload is exactly as wide as its event needs (no dead fields). A host that wants sound only on `submit` registers **only** `set_submit_callback` — the other three never touch its code path, satisfying the "cost of an unused listener is zero" requirement literally (not just "zero CPU cost," but zero code the host has to read/skip). Cost: +4 methods × 2 classes (8 new public methods total) instead of +1×2, and four WARNING/LIFETIME doc-comment blocks to keep in sync (mechanical, but real maintenance surface).

**Shape C — hybrid, collapsed pair.** Same as B for `change`/`submit`, but `focus`/`blur` are folded into **one** method `set_focus_change_callback(fn(const char* id, bool focused))` (`focused=true` on focus, `false` on blur) since they are exactly symmetric (same id-only payload, opposite direction of the same state machine — RmlUi itself always pairs them: a `Blur` on the old focus target is always immediately followed by a `Focus` on the new one, or vice-versa). Cuts the surface to 3 methods × 2 classes (6 new public methods) versus B's 8, at the cost of one naming departure from the strict `set_<event>_callback` idiom (`focus_change` is not a literal RmlUi event name) and one bit of host-side branching (`if (focused)`) that B avoids entirely.

**Recommendation:** **Shape B.** It costs the most methods but zero new design — it is a mechanical, low-risk extension of a pattern already shipped twice (click, scroll), with WARNING/LIFETIME blocks that can be adapted paragraph-by-paragraph from `set_scroll_callback`'s existing doc-comment rather than invented. Shape A is attractive only if the líder weighs "header line count" above "consistency with the shipped family" — a real but, on this codebase's track record (it has taken the verbose-but-consistent path three times running: `ClickInfo`, the scroll WARNING block, the focus/blur `bool` return convention), unlikely trade-off. Shape C is a reasonable compromise if the líder wants to cap the surface growth without going fully generic. Formalized as options A/B/C in §12.

### 3. Naming

Whichever shape is chosen, the four RmlUi event names (`change`, `submit`, `focus`, `blur` — confirmed against `examples/RmlUi/Source/Core/EventSpecification.cpp`, lines 17-18 and 42-43) map 1:1 to the method verbs, matching `click`→`set_click_callback` and `scroll`→`set_scroll_callback` exactly. Under Shape B/C, no new noun is invented (`set_change_callback`, `set_submit_callback`, `set_focus_callback`, `set_blur_callback` — or `set_focus_change_callback` under C). Under Shape A, `set_form_event_callback` is the one new noun ("form event" is not an RmlUi term; it is glintfx's own umbrella name for this batch of four DOM-style events, chosen because three of the four only fire on form controls — `focus`/`blur` are the exception, they fire on any focusable element, not just form controls, which is a mild naming inaccuracy worth flagging if Shape A is chosen).

The richer-payload precedent (`ClickInfo`) named its struct `<Verb>Info`; if any payload here grows a struct instead of scalar args (unlikely — see §4, all four fit in `(const char*, const char*)` at most), the same convention applies: `ChangeInfo`, not `FormChangeEvent` or similar.

### 4. Payload per event

RmlUi's own event parameters were read directly (`examples/RmlUi/Source/Core/Elements/{WidgetTextInput,WidgetSlider,WidgetDropDown,InputTypeCheckbox,InputTypeRadio,ElementForm}.cpp`, `EventSpecification.cpp`) to ground this table in fact rather than assumption:

| Event | RmlUi `EventId` | Bubbles? | RmlUi payload observed | glintfx payload proposed |
|---|---|---|---|---|
| `change` | `Change` | **yes** | a single `"value"` string parameter, populated **uniformly across every control kind**: text input/textarea (`WidgetTextInput.cpp:512`) send the current text; checkbox/radio (`InputTypeCheckbox.cpp:28`, `InputTypeRadio.cpp:39`) send `""` when unchecked/deselected and the control's `value` attribute when checked/selected; `<select>`/dropdown (`WidgetDropDown.cpp:280`) and `<input type=range>` slider (`WidgetSlider.cpp:607`) send the selected/current value string. **RmlUi has already done the "serialize by control kind" work for us** — glintfx does not need a discriminated union or a `ControlKind` tag; `event.GetParameter<Rml::String>("value", "").c_str()` is the same one-liner for all five widget kinds. | `(const char* id, const char* value)` — id via the same ancestor-walk-to-nearest-id already used by click/scroll, `value` is RmlUi's own `"value"` parameter, verbatim. |
| `submit` | `Submit` | yes | RmlUi's `ElementForm::Submit()` (`ElementForm.cpp:12-54`) builds a full `Dictionary` of **every named, enabled, submittable control's** `name → value`, plus a `"submit"` key holding the triggering submit-button's own name/value if it has none of its own name — i.e. the native event already carries the **entire form's data**, not just the id. | `(const char* id)` **only**, proposed for v1 — see the explicit trade-off discussion below. |
| `focus` | `Focus` | **no** (`DefaultActionPhase::Target`, confirmed `EventSpecification.cpp:17`) | none (no parameters observed) | `(const char* id)` |
| `blur` | `Blur` | **no** (same line) | none | `(const char* id)` |

**Submit payload trade-off (flagged, not resolved here).** Forwarding the full name/value dictionary would require translating an `Rml::Dictionary` (forbidden third-party type) into a plain-C shape — realistically an array of `{const char* name; const char* value;}` pairs plus a count, i.e. a **new struct family and an ownership/lifetime story for N strings**, not a one-line addition. Keeping `submit` id-only (mirroring `click`/`scroll`'s "id is enough, ask for detail via a getter if you need it" idiom) is the anti-OE-consistent v1 choice: a host that needs field values on submit already has them via the data-model it bound in the first place (`bind_string`/`set_string` per field), or — once `L1.16-DOMRW`'s read-back (`get_string` etc.) lands — can read each field's current value at the moment `set_submit_callback` fires. Speculatively building the array-of-pairs shape now, with no confirmed consumer for it, is exactly the over-engineering this project's anti-OE stance rejects (§11); it is a natural, low-risk **follow-up** (`L1.15b`-style) if a real form-processing host later needs it.

**A structural asymmetry worth flagging for implementation (P3), found while reading `EventSpecification.cpp`:** `Change`/`Submit`/`Click`/`Scroll` all `bubble`, but `Focus`/`Blur` do **not**. Every existing internal listener (`ClickEventListener`, `ScrollEventListener` — see `bootstrap.cpp:707`, and the equivalent for `Scroll`) is attached **once, at the document root, in the bubble phase** (`AddEventListener(EventId, listener, /*in_capture_phase=*/false)`), relying on the event bubbling up from the target to the document. A non-bubbling event does not reach a bubble-phase listener attached anywhere but the target itself. The fix is not a new pattern, though: RmlUi's **capture phase** always descends root→target regardless of the `bubbles` flag (that flag only controls whether propagation continues target→root afterward), so a document-root listener attached with `in_capture_phase=true` for `Focus`/`Blur` specifically (while `Change`/`Submit` keep the existing bubble-phase pattern) should reach every element. This is a real implementation detail, not a design fork — it does not change the public API surface in §2/§3, only the internal listener wiring for two of the four events — but it is exactly the kind of thing that bites a "should be a one-line copy of `ScrollEventListener`" assumption if unflagged.

### 5. Lifetime

`id` and `value` (whichever shape is chosen) point into RmlUi-owned storage — `id` into the resolved `Rml::Element`'s own id string (identical contract to `set_click_callback`'s `element_id` and `ClickInfo::id`), `value` into the `Rml::Event`'s own parameter dictionary entry for `"value"`. **Both are valid only for the duration of the callback invocation.** A host that needs either afterwards must copy it (e.g. into a `std::string`) before returning — same explicit rule already written for `ClickInfo::id` and implied for `set_click_callback`/`set_scroll_callback`'s `element_id`. No new lifetime model is introduced; this section exists to make the rule explicit for the new payload fields rather than leaving it implicit.

### 6. Ordering & reentrancy

**Firing point relative to `update()`.** None of the four events are driven by `update()` itself (unlike, say, a data-model dirty-flag flush). They fire **synchronously, inside whichever call caused them**:
- `change` fires inside `process_event()` for the input that produced it — a `Key`/`Text` event for text controls, a `MouseButton` event for checkbox/radio/select-option clicks, a `MouseMove` event while dragging a slider thumb.
- `submit` fires inside `process_event(MouseButton)` (a submit-type button's click) or `process_event(Key)` (Enter inside a text field, RmlUi's own default action) — always as part of the **same** click/key event's default-action phase that also fires `click`/`click_info` (see §7 for the resulting same-stack ordering).
- `focus`/`blur` fire inside whichever call **moved** focus: `process_event(MouseButton)` (clicking a focusable element), `process_event(Key)` (Tab-navigation), **or** the programmatic `set_focus()`/`clear_focus()` calls shipped by `L1.17-FOCUS` — synchronously, on the same call stack, before those calls return.

All four therefore fire **before** the frame's `render()` (they are input-time, not draw-time), same guarantee already implied for click/scroll.

**Reentrancy discipline — mandatory, not optional.** `CHANGELOG.md [0.4.1]` documents a **Critical heap-use-after-free** (`AUD-TEC-3`) where `set_click_callback`'s internal listener invoked the registered `std::function` through a pointer into its own storage; a handler that called `set_click_callback(...)` again from inside itself freed the functor still executing. The fix — **copy the functor to a local before invoking it** — is now the load-bearing discipline behind every callback in this family (`set_click_callback`, `set_click_info_callback`, `set_scroll_callback` all do this). **This is a hard requirement for the P3 implementation of all four `L1.15-FORMEV` listeners, not a nice-to-have**: any internal `*EventListener::ProcessEvent` that invokes the registered callback MUST copy it to a local `std::function` first, exactly like `ClickEventListener`/`ScrollEventListener` already do.

**Recursion risk — focus/blur specifically (mirroring the `set_scroll_callback` v0.6.0 WARNING, per the task's explicit instruction).** `L1.17-FOCUS`'s `set_focus()`/`clear_focus()` are, like `scroll_element_into_view()`/`set_element_scroll_top()`, **synchronous** — they dispatch the underlying `Focus`/`Blur` event on the same call stack before returning. Calling either of them from **inside** a `focus`/`blur` callback re-enters that same callback. This is a real, plausible pattern (a "focus trap" — e.g. a modal's blur handler immediately re-focusing itself: `on_blur → set_focus("modal")`) and, unlike scroll's offset-convergence dedup, **RmlUi's own focus model has no equivalent dedup for a no-op refocus** confirmed here — this needs to be verified empirically at implementation time (does `Element::Focus()` on the already-focused element re-dispatch `Focus`, or is it a silent no-op? `clear_focus()`'s own doc-comment already documents an analogous idempotent-no-op case: "TRUE when a document IS loaded but nothing is currently focused — there is nothing to undo"). **Until verified, treat a focus-callback that unconditionally re-focuses on every invocation as an unbounded-recursion risk**, same posture as the scroll WARNING, and document the convergence condition once confirmed. `change`/`submit` carry a lower documented risk here: no glintfx method today writes a form control's value in a way that is confirmed to re-dispatch `Change` (RmlUi's `data-value` MODEL→UI direction updates the element's displayed value via a `DataView`, not via re-dispatching `Change` — that event flows only UI→model, via `DataControllerValue`, confirmed reading `examples/RmlUi/Source/Core/DataControllerDefault.cpp:29-34`) — but a `submit`/`change` handler that calls `set_focus()`/`clear_focus()` (a common UX pattern: focus the first invalid field after a failed submit) **does** fall under the focus/blur recursion risk above, cross-callback.

### 7. Synergy with click/scroll

All callback channels in this family are **parallel and independent** — registering `set_change_callback` does not disable/replace `set_click_callback`, `set_click_info_callback`, or `set_scroll_callback`, exactly as `set_click_info_callback` did not replace `set_click_callback` (AUD-PUB-4). A single user interaction routinely fires **more than one** channel, same call stack, in this order:
- Clicking a `type="submit"` button fires `click_callback`/`click_info_callback` first (native `Click` event, id of the button), then — inside that same click's default-action phase — `ElementForm::Submit()` dispatches `Submit`, firing `submit_callback` (id of the `<form>`) **within the same stack, same frame**, before `process_event()` returns.
- Focus and scroll are **not** coupled automatically: per the existing `UiEvent::Type::MouseWheel` doc-comment, `ProcessMouseWheel` scrolls whatever is under the mouse (`hover`), never the focused element — Tab-navigating to an off-screen list item fires `focus_callback` alone; nothing scrolls it into view unless the host explicitly calls `scroll_element_into_view()` in response (which then fires `scroll_callback` **separately**, its own reentrant call into that listener, per §6 of `set_scroll_callback`'s doc-comment). A host wiring "focus sound" + "scroll-into-view sound" therefore registers both callbacks and expects **two** firings for one logical "selection moved off-screen" gesture, not one combined event.

### 8. Anchor case — the líder's sound hooks

The forcing use case named by the líder: a host wants to pick and trigger a sound effect from inside the callback, and the payload must be **sufficient** to do so — at minimum, **event type + id**. Both Shape B (§2) and Shape A satisfy this literally, by different means: under B, the type is implicit in *which* callback fired (a host registering `set_submit_callback` already knows every invocation is a submit — no `type` field needed, the callback IS the type); under A, `type` is an explicit field the host switches on inside one shared handler. Either shape is enough to build the natural game-audio pattern — a per-event-kind (or, under B, per-id) lookup into a sound table, matching how `set_click_callback`/`set_scroll_callback` are already documented as "the hook a host uses to trigger its own click/scroll sound; glintfx does NOT play audio itself."

**Latency.** All four fire **synchronously**, inside the same call stack as the input that produced them (§6) — there is no queueing, no next-frame delay. A host's sound-trigger call inside the callback runs to completion **before** `process_event()`/`set_focus()`/`clear_focus()` returns to its caller, and therefore before that frame's `update()`/`render()`. This is the same latency contract already documented for click/scroll ("glintfx does NOT play audio itself... this is the hook"), extended here explicitly: **if the host's audio API is non-blocking (the expected case — enqueue-to-mixer, fire-and-forget), the added cost is negligible; if the host's audio call is itself blocking (e.g. synchronous disk decode), it stalls the very frame the event occurred in**, same caveat that already applies to anything a host does inside these synchronous callbacks. Document this plainly at implementation time so a host does not mistake "synchronous callback" for "safe to do arbitrarily slow work inside."

### 9. Parity: App + UiLayer

Identical signatures in both classes, same as every existing callback in this family (`set_click_callback`, `set_click_info_callback`, `set_scroll_callback` all already have this parity). **One simplification relative to `click_info`:** none of the four events carry a coordinate pair, so — unlike `ClickInfo`'s `x`/`y`, which `UiLayer::set_click_info_callback` must translate by the active sub-viewport offset (`box.x = x + impl_->x`, per `set_viewport(x,y,w,h,target_h)`) while `App`'s version needs no translation — **`UiLayer`'s implementation of all four new callbacks needs zero coordinate-space math**. The `id`/`value` strings are identical between `App` and `UiLayer`; the only difference between the two classes' internal wiring is which `Rml::Context`/`Rml::ElementDocument` the listener is attached to (mirrors `set_scroll_callback`'s existing App/UiLayer split exactly). This makes `L1.15-FORMEV` a **lower-risk parity item than `L1.13-VPORIGIN`/`click_info` was** — there is no viewport-offset class of bug available to introduce here.

### 10. Relation to data-binding

RmlUi's own two-way binding (`data-value="modelvar"` in `.rml`) is **implemented via a `Change` listener already** — confirmed reading `examples/RmlUi/Source/Core/DataControllerDefault.cpp:29,34`: `DataControllerValue` attaches its own `EventId::Change` listener directly on the bound element and writes the new value back into the model on every `Change`. This is the **UI→model** direction. glintfx's own `DataBinder` (`bind_number/string/bool/list` + `set_number/string/bool/list`) is, as of today, **write-only in the opposite direction** (host→UI; `L1.16-DOMRW` is the pending backlog item that would add read-back). Two consequences:

1. **No conflict, no interception.** RmlUi supports multiple independent listeners per element/event (the exact mechanism `set_click_callback` and `set_click_info_callback` already both rely on, attached as two separate listener instances on the same `EventId::Click`, per `bootstrap.cpp:707-716`). A document-level `set_change_callback` listener and a per-element `data-value` `DataControllerValue` listener on the same control coexist and both fire independently on the same `Change` — registering one does not disable or shadow the other.
2. **`set_change_callback` is, today, the *only* way for a host to learn "the user changed a bound control's value" without polling.** Since `DataBinder` cannot read back a model variable yet (`L1.16-DOMRW` pending), a host that wants an immediate reaction to a user edit (validation, "unsaved changes" flag, the sound-hook anchor case) has no channel for that **until this feature lands** — `set_change_callback` is not a redundant alternative to data-binding, it fills a gap data-binding does not close on its own. Once `L1.16-DOMRW`'s `get_string`/`get_number`/`get_bool` land, both mechanisms remain complementary rather than duplicative: `data-value` keeps the model in sync (state), `set_change_callback` reports the moment of change (event) — document this explicitly so hosts do not think they must pick one over the other.

### 11. Anti-scope-creep — only these four events

The `TODO.md` entry names exactly `change`/`submit`/`focus`/`blur`; this spec does not expand that list. Explicitly out of scope, and why:

- **`keydown`/`keyup` (raw key events).** Already covered from the opposite direction: `UiEvent::Type::Key` is the host-**injected** channel (host tells glintfx about a keypress). An RmlUi-**observed** keydown/keyup callback would be a second channel carrying overlapping information, at a much higher event rate (every keystroke, not the settled value `change` already reports) with no batching — the wrong default for the sound-hook anchor case (would fire 3 times typing "abc" where `change` fires once when the field loses focus or on each committed edit, depending on control). No consumer need identified.
- **`mouseover`/`mouseout` (hover).** No anchor use case — the líder's named cases so far (click, scroll, focus/blur, sound hooks) are all discrete, user-committed actions; hover is a continuous, high-frequency stream (fires on every pixel crossing an element boundary) that would need its own debounce story before being useful for anything, sound included. Zero real demand today.
- **`drag`/`dragstart`/`dragend`/`drop` (RmlUi's native drag-and-drop events).** No widget in GusWorld's shipped or planned screens (cockpit/HUD, menu, per the v2 spec's own scope) uses drag-and-drop. The payload alone (drag source, drop target, transferred data) is a design project of its own scale, disproportionate to zero confirmed demand.

Rule invoked: the same pull-incremental discipline already adopted for `glintfx` v2 ("backlog puxado por demanda real," decision of 2026-07-01) — build the four events with a named consumer and a named anchor use case (sound hooks), defer the rest until a real host asks.

### 12. Pending decision for the líder

Presented for `AskUserQuestion`, orchestrator-delivered. All three keep the "no third-party type in the public header" rule, `App`+`UiLayer` parity, and the mandatory copy-before-invoke reentrancy discipline (§6) — those are not up for debate, only the **surface shape** is.

**Option A — Generic single callback.** `set_form_event_callback(fn(id, FormEventType type, value))` + 1 new public enum. **Effort: lowest** (1 method × 2 classes, 1 shared doc-comment/WARNING block). **Pros:** smallest header delta; naturally fits a single "sound table indexed by type" host pattern. **Cons:** breaks the `set_<verb>_callback` naming idiom used by every shipped callback so far; `value` is dead for `focus`/`blur` (needs per-branch documentation of when it's meaningless); no compile-time exhaustiveness on the host's `switch`.

**Option B — Four specific callbacks (recommended).** `set_change_callback`, `set_submit_callback`, `set_focus_callback`, `set_blur_callback`, each id-only except `change` (id+value). **Effort: low-medium** (4 methods × 2 classes = 8 new public methods, but each is a mechanical extension of the `set_click_callback`/`set_scroll_callback` pattern already shipped twice — no new design, doc-comment blocks adapted paragraph-by-paragraph from `set_scroll_callback`'s existing WARNING). **Pros:** zero naming departure; a host registers only what it needs, literally zero code touching the events it doesn't care about; each payload is exactly as wide as its event needs. **Cons:** largest header-surface growth of the three; four WARNING/LIFETIME blocks to keep in sync over time (mechanical maintenance, not design risk).

**Option C — Hybrid, collapsed focus/blur pair.** `set_change_callback`, `set_submit_callback`, `set_focus_change_callback(fn(id, bool focused))`. **Effort: low-medium**, between A and B (3 methods × 2 classes = 6 new public methods). **Pros:** cuts B's surface by 2 methods by exploiting the genuine symmetry of focus/blur (RmlUi always pairs them); still typed, still opt-in. **Cons:** one naming departure (`focus_change` is not a literal RmlUi event name); host branches on a `bool` instead of getting a dedicated callback per direction.

**Recommendation: Option B.** It is the choice with the least new design risk — this codebase has, three times running (click, click_info, scroll), chosen "verbose but consistent with the shipped naming idiom" over "compact but novel," and every WARNING/reentrancy block this feature needs already exists almost verbatim in `set_scroll_callback`'s doc-comment to adapt from. Option A is a legitimate alternative if the líder weighs header-line-count savings above naming consistency; Option C is the middle ground if the líder wants to cap the method count without going fully generic.

## PT

### 1. Objetivo e escopo

A glintfx hoje expõe duas famílias de atividade de UI observável para o host: **ponteiro** (`set_click_callback`/`set_click_info_callback`) e **rolagem** (`set_scroll_callback`). Formulários hoje são **inertes**: um `<input>`, `<select>`, `<textarea>` ou `<form>` pode ser estilizado e dirigido numa via só (host→UI) pelo data-model (`bind_*`/`set_*`), mas o host não tem como saber que o **usuário** digitou, marcou, escolheu ou submeteu algo, a não ser fazendo polling. `L1.15-FORMEV` fecha essa lacuna expondo os próprios eventos `Change`/`Submit`/`Focus`/`Blur` do RmlUi ao host, pela mesma disciplina "nenhum tipo de terceiro cruza a fronteira pública" já aplicada a click/scroll.

O escopo é deliberadamente os mesmos quatro eventos nomeados no item do `TODO.md` — nada mais (ver §11 pro que é explicitamente cortado e por quê).

### 2. Ponto de decisão 1 — Superfície do callback

Três formatos foram avaliados, todos mantendo a regra "nenhum tipo `Rml::*` no header público" e a regra de paridade `App`+`UiLayer`.

**Formato A — um callback genérico.** `set_form_event_callback(std::function<void(const char* id, FormEventType type, const char* value)> cb)`, com `FormEventType` sendo um novo `enum class` público (`Change, Submit, Focus, Blur`). Um único ponto de registro, um único bloco de doc-comment/AVISO a manter, menor delta de superfície de header (+1 método × 2 classes + 1 enum). Casa diretamente com o caso âncora do §8 (uma única tabela de despacho "áudio-por-evento" indexada por `type`). Custo: `value` não tem sentido para `Focus`/`Blur` (sempre `""`/`nullptr` — precisa ser documentado por ramo em vez de por método), o host precisa de um `switch` interno em `type` (o compilador não ajuda a pegar um caso faltando), e é o **primeiro** callback da família cujo nome não é um verbo — quebra o idioma de nomenclatura `set_<evento>_callback` usado por todo ponto de entrada já existente.

**Formato B — quatro callbacks específicos.** `set_change_callback(fn(id, value))`, `set_submit_callback(fn(id))`, `set_focus_callback(fn(id))`, `set_blur_callback(fn(id))`. É a extensão literal da família existente: `set_click_callback`/`set_scroll_callback` já são exatamente nesse formato (só-id, um verbo por método), então isso é "mais do mesmo," não um design novo. Cada payload é exatamente do tamanho que seu evento precisa (nenhum campo morto). Um host que quer som só em `submit` registra **só** `set_submit_callback` — os outros três nunca tocam no código dele, satisfazendo a exigência "custo de um listener não usado é zero" literalmente (não só "custo de CPU zero," mas zero código que o host precisa ler/pular). Custo: +4 métodos × 2 classes (8 métodos públicos novos ao todo) em vez de +1×2, e quatro blocos de doc-comment AVISO/LIFETIME pra manter sincronizados (mecânico, mas superfície real de manutenção).

**Formato C — híbrido, par colapsado.** Igual a B para `change`/`submit`, mas `focus`/`blur` são dobrados em **um** método só `set_focus_change_callback(fn(const char* id, bool focused))` (`focused=true` no focus, `false` no blur), já que são exatamente simétricos (mesmo payload só-id, direção oposta da mesma máquina de estado — o próprio RmlUi sempre os empareja: um `Blur` no alvo antigo de foco é sempre seguido imediatamente por um `Focus` no novo, ou vice-versa). Corta a superfície para 3 métodos × 2 classes (6 métodos públicos novos) contra os 8 de B, ao custo de um desvio de nomenclatura (`focus_change` não é um nome literal de evento do RmlUi) e um pouco de branching do lado do host (`if (focused)`) que B evita completamente.

**Recomendação:** **Formato B.** Custa mais métodos, mas zero design novo — é uma extensão mecânica, de baixo risco, de um padrão já shipado duas vezes (click, scroll), com blocos AVISO/LIFETIME que podem ser adaptados parágrafo-a-parágrafo do doc-comment já existente de `set_scroll_callback`, em vez de inventados. O Formato A só é atrativo se o líder pesar "contagem de linhas de header" acima de "consistência com a família já shipada" — um trade-off real mas, no histórico deste repositório (escolheu o caminho verboso-mas-consistente três vezes seguidas: `ClickInfo`, o bloco AVISO de scroll, a convenção de retorno `bool` de focus/blur), improvável. O Formato C é um meio-termo razoável se o líder quiser limitar o crescimento da superfície sem ir totalmente genérico. Formalizado como opções A/B/C no §12.

### 3. Nomenclatura

Qualquer que seja o formato escolhido, os quatro nomes de evento do RmlUi (`change`, `submit`, `focus`, `blur` — confirmados contra `examples/RmlUi/Source/Core/EventSpecification.cpp`, linhas 17-18 e 42-43) mapeiam 1:1 para os verbos dos métodos, casando exatamente com `click`→`set_click_callback` e `scroll`→`set_scroll_callback`. No Formato B/C, nenhum substantivo novo é inventado (`set_change_callback`, `set_submit_callback`, `set_focus_callback`, `set_blur_callback` — ou `set_focus_change_callback` no C). No Formato A, `set_form_event_callback` é o único substantivo novo ("form event" não é termo do RmlUi; é um nome guarda-chuva próprio da glintfx pra este lote de quatro eventos estilo-DOM, escolhido porque três dos quatro só disparam em controles de formulário — `focus`/`blur` são a exceção, disparam em qualquer elemento focável, não só controles de formulário, o que é uma imprecisão leve de nomenclatura a sinalizar se o Formato A for escolhido).

O precedente de payload mais rico (`ClickInfo`) nomeou sua struct `<Verbo>Info`; se algum payload aqui crescer para uma struct em vez de argumentos escalares (improvável — ver §4, os quatro cabem em `(const char*, const char*)` no máximo), a mesma convenção se aplica: `ChangeInfo`, não `FormChangeEvent` ou similar.

### 4. Payload por evento

Os próprios parâmetros de evento do RmlUi foram lidos diretamente (`examples/RmlUi/Source/Core/Elements/{WidgetTextInput,WidgetSlider,WidgetDropDown,InputTypeCheckbox,InputTypeRadio,ElementForm}.cpp`, `EventSpecification.cpp`) pra fundamentar esta tabela em fato, não suposição:

| Evento | `EventId` do RmlUi | Borbulha (bubble)? | Payload observado do RmlUi | Payload proposto pra glintfx |
|---|---|---|---|---|
| `change` | `Change` | **sim** | um único parâmetro string `"value"`, populado **uniformemente entre todo tipo de controle**: text input/textarea (`WidgetTextInput.cpp:512`) mandam o texto atual; checkbox/radio (`InputTypeCheckbox.cpp:28`, `InputTypeRadio.cpp:39`) mandam `""` quando desmarcado/desselecionado e o atributo `value` do controle quando marcado/selecionado; `<select>`/dropdown (`WidgetDropDown.cpp:280`) e slider `<input type=range>` (`WidgetSlider.cpp:607`) mandam a string do valor selecionado/atual. **O RmlUi já fez o trabalho de "serializar por tipo de controle" por nós** — a glintfx não precisa de uma união discriminada nem de uma tag `ControlKind`; `event.GetParameter<Rml::String>("value", "").c_str()` é a mesma linha única pros cinco tipos de widget. | `(const char* id, const char* value)` — id pela mesma subida de ancestral até o id mais próximo já usada por click/scroll, `value` é o próprio parâmetro `"value"` do RmlUi, verbatim. |
| `submit` | `Submit` | sim | o `ElementForm::Submit()` do RmlUi (`ElementForm.cpp:12-54`) monta um `Dictionary` completo de `name → value` de **todo controle nomeado, habilitado e submissível**, mais uma chave `"submit"` com o próprio nome/valor do botão de submissão que disparou, se ele não tiver nome próprio — ou seja, o evento nativo já carrega os **dados do formulário inteiro**, não só o id. | `(const char* id)` **apenas**, proposto pra v1 — ver a discussão de trade-off explícita abaixo. |
| `focus` | `Focus` | **não** (`DefaultActionPhase::Target`, confirmado `EventSpecification.cpp:17`) | nenhum (nenhum parâmetro observado) | `(const char* id)` |
| `blur` | `Blur` | **não** (mesma linha) | nenhum | `(const char* id)` |

**Trade-off do payload de submit (sinalizado, não resolvido aqui).** Encaminhar o dicionário completo de nome/valor exigiria traduzir um `Rml::Dictionary` (tipo de terceiro proibido) para um formato C puro — realisticamente um array de `{const char* name; const char* value;}` mais uma contagem, ou seja, uma **família de struct nova e uma história de posse/lifetime pra N strings**, não uma adição de uma linha. Manter `submit` só-id (espelhando o idioma "o id basta, peça o detalhe via um getter se precisar" de click/scroll) é a escolha v1 consistente com anti-OE: um host que precisa dos valores dos campos no submit já os tem via o data-model que ele mesmo ligou (`bind_string`/`set_string` por campo), ou — assim que a leitura de volta do `L1.16-DOMRW` (`get_string` etc.) pousar — pode ler o valor atual de cada campo no momento em que `set_submit_callback` dispara. Construir especulativamente o formato array-de-pares agora, sem consumidor confirmado, é exatamente o over-engineering que a postura anti-OE deste projeto rejeita (§11); é um **follow-up** natural e de baixo risco (estilo `L1.15b`) se um host real de processamento de formulário precisar depois.

**Uma assimetria estrutural que vale sinalizar pra implementação (P3), achada lendo `EventSpecification.cpp`:** `Change`/`Submit`/`Click`/`Scroll` todos borbulham (`bubble`), mas `Focus`/`Blur` **não**. Todo listener interno já existente (`ClickEventListener`, `ScrollEventListener` — ver `bootstrap.cpp:707`, e o equivalente pra `Scroll`) é anexado **uma vez, na raiz do documento, na fase de bubble** (`AddEventListener(EventId, listener, /*in_capture_phase=*/false)`), contando com o evento borbulhar do alvo até o documento. Um evento que não borbulha não alcança um listener de fase-bubble anexado em qualquer lugar que não seja o próprio alvo. O conserto não é um padrão novo, porém: a **fase de captura** do RmlUi sempre desce raiz→alvo independente da flag `bubbles` (essa flag só controla se a propagação continua alvo→raiz depois) — então um listener na raiz do documento anexado com `in_capture_phase=true` especificamente pra `Focus`/`Blur` (enquanto `Change`/`Submit` mantêm o padrão de fase-bubble já existente) deveria alcançar todo elemento. É um detalhe real de implementação, não uma bifurcação de design — não muda a superfície pública de API do §2/§3, só a fiação interna de listener pra dois dos quatro eventos — mas é exatamente o tipo de coisa que morde uma suposição de "deve ser uma cópia de uma linha só do `ScrollEventListener`" se não sinalizada.

### 5. Lifetime

`id` e `value` (qualquer que seja o formato escolhido) apontam pra dentro de armazenamento possuído pelo RmlUi — `id` pra dentro da própria string de id do `Rml::Element` resolvido (contrato idêntico ao `element_id` de `set_click_callback` e ao `ClickInfo::id`), `value` pra dentro da própria entrada do dicionário de parâmetros do `Rml::Event` para `"value"`. **Ambos são válidos só durante a duração da invocação do callback.** Um host que precise de qualquer um deles depois precisa copiar (ex.: pra um `std::string`) antes de retornar — mesma regra explícita já escrita pro `ClickInfo::id` e implícita pro `element_id` de `set_click_callback`/`set_scroll_callback`. Nenhum modelo de lifetime novo é introduzido; esta seção existe pra tornar a regra explícita pros novos campos de payload em vez de deixá-la implícita.

### 6. Ordem e reentrância

**Ponto de disparo relativo a `update()`.** Nenhum dos quatro eventos é conduzido pelo próprio `update()` (diferente, digamos, de um flush de dirty-flag do data-model). Eles disparam **sincronamente, dentro de qualquer chamada que os causou**:
- `change` dispara dentro de `process_event()` pro input que o produziu — um evento `Key`/`Text` pra controles de texto, um evento `MouseButton` pra cliques em checkbox/radio/opção-de-select, um evento `MouseMove` durante o arraste do thumb de um slider.
- `submit` dispara dentro de `process_event(MouseButton)` (clique num botão tipo submit) ou `process_event(Key)` (Enter dentro de um campo de texto, ação default do próprio RmlUi) — sempre como parte da **mesma** fase de ação-default do evento de clique/tecla que também dispara `click`/`click_info` (ver §7 pra ordenação resultante na mesma pilha).
- `focus`/`blur` disparam dentro de qualquer chamada que **moveu** o foco: `process_event(MouseButton)` (clicar num elemento focável), `process_event(Key)` (navegação por Tab), **ou** as chamadas programáticas `set_focus()`/`clear_focus()` shipadas pelo `L1.17-FOCUS` — sincronamente, na mesma pilha de chamada, antes dessas chamadas retornarem.

Os quatro portanto disparam **antes** do `render()` daquele frame (são de momento-de-input, não de momento-de-desenho), mesma garantia já implícita pra click/scroll.

**Disciplina de reentrância — obrigatória, não opcional.** O `CHANGELOG.md [0.4.1]` documenta um **heap-use-after-free Critical** (`AUD-TEC-3`) onde o listener interno de `set_click_callback` invocava o `std::function` registrado através de um ponteiro pro próprio storage dele; um handler que chamasse `set_click_callback(...)` de novo de dentro de si mesmo liberava o functor ainda em execução. O conserto — **copiar o functor pra uma variável local antes de invocar** — é hoje a disciplina que sustenta todo callback desta família (`set_click_callback`, `set_click_info_callback`, `set_scroll_callback` fazem isso). **Isso é um requisito obrigatório pra implementação P3 dos quatro listeners do `L1.15-FORMEV`, não um "seria bom ter"**: qualquer `*EventListener::ProcessEvent` interno que invoque o callback registrado DEVE copiá-lo pra um `std::function` local primeiro, exatamente como `ClickEventListener`/`ScrollEventListener` já fazem.

**Risco de recursão — focus/blur especificamente (espelhando o AVISO de `set_scroll_callback` da v0.6.0, conforme instrução explícita da tarefa).** `set_focus()`/`clear_focus()` do `L1.17-FOCUS` são, como `scroll_element_into_view()`/`set_element_scroll_top()`, **síncronos** — despacham o evento `Focus`/`Blur` subjacente na mesma pilha de chamada antes de retornar. Chamar qualquer um deles de **dentro** de um callback `focus`/`blur` reentra nesse mesmo callback. É um padrão real e plausível (uma "armadilha de foco" — ex.: o handler de blur de um modal re-focando a si mesmo imediatamente: `on_blur → set_focus("modal")`) e, diferente do dedup por convergência de offset do scroll, **o próprio modelo de foco do RmlUi não tem um dedup confirmado equivalente pra um refoco no-op** — isso precisa ser verificado empiricamente na hora da implementação (`Element::Focus()` num elemento já focado re-despacha `Focus`, ou é um no-op silencioso? O próprio doc-comment de `clear_focus()` já documenta um caso análogo de no-op-idempotente: "TRUE quando um documento ESTÁ carregado mas nada está focado no momento — não há nada para desfazer"). **Até verificado, trate um callback de foco que refoca incondicionalmente a cada invocação como um risco de recursão sem limite**, mesma postura do AVISO de scroll, e documente a condição de convergência assim que confirmada. `change`/`submit` carregam um risco documentado menor aqui: nenhum método da glintfx hoje escreve o valor de um controle de formulário de um jeito confirmado a re-despachar `Change` (a direção MODELO→UI do `data-value` do RmlUi atualiza o valor exibido do elemento via uma `DataView`, não redespachando `Change` — esse evento flui só UI→modelo, via `DataControllerValue`, confirmado lendo `examples/RmlUi/Source/Core/DataControllerDefault.cpp:29-34`) — mas um handler de `submit`/`change` que chama `set_focus()`/`clear_focus()` (um padrão de UX comum: focar o primeiro campo inválido depois de um submit que falhou) **cai** no risco de recursão de focus/blur acima, entre-callbacks.

### 7. Sinergia com click/scroll

Todos os canais de callback desta família são **paralelos e independentes** — registrar `set_change_callback` não desabilita/substitui `set_click_callback`, `set_click_info_callback`, ou `set_scroll_callback`, exatamente como `set_click_info_callback` não substituiu `set_click_callback` (AUD-PUB-4). Uma única interação de usuário rotineiramente dispara **mais de um** canal, na mesma pilha de chamada, nesta ordem:
- Clicar num botão `type="submit"` dispara `click_callback`/`click_info_callback` primeiro (evento `Click` nativo, id do botão), depois — dentro da mesma fase de ação-default daquele clique — `ElementForm::Submit()` despacha `Submit`, disparando `submit_callback` (id do `<form>`) **dentro da mesma pilha, mesmo frame**, antes de `process_event()` retornar.
- Foco e rolagem **não** são acoplados automaticamente: pelo doc-comment já existente de `UiEvent::Type::MouseWheel`, `ProcessMouseWheel` rola o que estiver sob o mouse (`hover`), nunca o elemento focado — navegar por Tab até um item de lista fora da tela dispara `focus_callback` sozinho; nada rola ele pra dentro da vista a menos que o host chame `scroll_element_into_view()` explicitamente em resposta (o que então dispara `scroll_callback` **separadamente**, sua própria chamada reentrante nesse listener, conforme §6 do doc-comment de `set_scroll_callback`). Um host que conecta "som de foco" + "som de rolar-para-a-vista" portanto registra os dois callbacks e espera **duas** disparadas pra um único gesto lógico de "seleção saiu da tela," não um evento combinado.

### 8. Caso âncora — os ganchos de som do líder

O caso de uso forçador nomeado pelo líder: um host quer escolher e disparar um efeito sonoro de dentro do callback, e o payload precisa ser **suficiente** pra isso — no mínimo, **tipo do evento + id**. Tanto o Formato B (§2) quanto o A satisfazem isso literalmente, por meios diferentes: no B, o tipo é implícito em *qual* callback disparou (um host registrando `set_submit_callback` já sabe que toda invocação é um submit — nenhum campo `type` necessário, o callback É o tipo); no A, `type` é um campo explícito que o host trata num único handler compartilhado. Qualquer um dos formatos basta pra construir o padrão natural de áudio-de-jogo — uma busca por-tipo-de-evento (ou, no B, por-id) numa tabela de sons, casando com o jeito que `set_click_callback`/`set_scroll_callback` já são documentados como "o gancho que um host usa para disparar seu próprio som de clique/rolagem; a glintfx NÃO toca áudio."

**Latência.** Os quatro disparam **sincronamente**, na mesma pilha de chamada do input que os produziu (§6) — não há enfileiramento, nenhum atraso pro próximo frame. A chamada do host que dispara o som de dentro do callback roda até completar **antes** de `process_event()`/`set_focus()`/`clear_focus()` retornar pro seu chamador, e portanto antes do `update()`/`render()` daquele frame. É o mesmo contrato de latência já documentado pra click/scroll ("a glintfx NÃO toca áudio... este é o gancho"), estendido aqui explicitamente: **se a API de áudio do host for não-bloqueante (o caso esperado — enfileirar-no-mixer, fire-and-forget), o custo adicionado é desprezível; se a chamada de áudio do host for ela mesma bloqueante (ex.: decode síncrono de disco), ela trava o mesmo frame em que o evento ocorreu**, mesma ressalva que já se aplica a qualquer coisa que um host faça de dentro desses callbacks síncronos. Documentar isso claramente na hora da implementação pra que um host não confunda "callback síncrono" com "seguro pra fazer trabalho arbitrariamente lento dentro."

### 9. Paridade: App + UiLayer

Assinaturas idênticas nas duas classes, igual a todo callback já existente nesta família (`set_click_callback`, `set_click_info_callback`, `set_scroll_callback` já têm essa paridade). **Uma simplificação em relação ao `click_info`:** nenhum dos quatro eventos carrega um par de coordenada, então — diferente do `x`/`y` do `ClickInfo`, que `UiLayer::set_click_info_callback` precisa traduzir pelo offset de sub-viewport ativo (`box.x = x + impl_->x`, conforme `set_viewport(x,y,w,h,target_h)`) enquanto a versão do `App` não precisa de tradução — **a implementação dos quatro novos callbacks no `UiLayer` não precisa de nenhuma matemática de espaço-de-coordenada**. As strings `id`/`value` são idênticas entre `App` e `UiLayer`; a única diferença na fiação interna das duas classes é a qual `Rml::Context`/`Rml::ElementDocument` o listener é anexado (espelha exatamente a divisão App/UiLayer já existente de `set_scroll_callback`). Isso torna `L1.15-FORMEV` um **item de paridade de risco menor do que `L1.13-VPORIGIN`/`click_info` foi** — não há nenhuma classe de bug de offset-de-viewport disponível pra introduzir aqui.

### 10. Relação com data-binding

A própria vinculação bidirecional do RmlUi (`data-value="modelvar"` no `.rml`) é **implementada via um listener de `Change` já existente** — confirmado lendo `examples/RmlUi/Source/Core/DataControllerDefault.cpp:29,34`: `DataControllerValue` anexa seu próprio listener de `EventId::Change` diretamente no elemento ligado e escreve o novo valor de volta no modelo a cada `Change`. Essa é a direção **UI→modelo**. O próprio `DataBinder` da glintfx (`bind_number/string/bool/list` + `set_number/string/bool/list`) é, hoje, **só-escrita na direção oposta** (host→UI; `L1.16-DOMRW` é o item de backlog pendente que adicionaria a leitura de volta). Duas consequências:

1. **Sem conflito, sem interceptação.** O RmlUi suporta múltiplos listeners independentes por elemento/evento (o mecanismo exato do qual `set_click_callback` e `set_click_info_callback` já dependem, anexados como duas instâncias de listener separadas no mesmo `EventId::Click`, conforme `bootstrap.cpp:707-716`). Um listener de `set_change_callback` no nível do documento e um listener `DataControllerValue` de `data-value` por-elemento no mesmo controle coexistem e ambos disparam independentemente no mesmo `Change` — registrar um não desabilita nem esconde o outro.
2. **`set_change_callback` é, hoje, a *única* forma de um host saber "o usuário mudou o valor de um controle ligado" sem fazer polling.** Já que o `DataBinder` ainda não consegue ler de volta uma variável do modelo (`L1.16-DOMRW` pendente), um host que quer uma reação imediata a uma edição do usuário (validação, flag de "mudanças não salvas", o caso âncora do gancho de som) não tem canal pra isso **até esta feature pousar** — `set_change_callback` não é uma alternativa redundante ao data-binding, ele fecha uma lacuna que o data-binding não fecha sozinho. Assim que `get_string`/`get_number`/`get_bool` do `L1.16-DOMRW` pousarem, os dois mecanismos seguem complementares em vez de duplicados: `data-value` mantém o modelo sincronizado (estado), `set_change_callback` reporta o momento da mudança (evento) — documentar isso explicitamente pra que hosts não achem que precisam escolher um ou outro.

### 11. Anti-scope-creep — só estes quatro eventos

O item do `TODO.md` nomeia exatamente `change`/`submit`/`focus`/`blur`; esta spec não expande essa lista. Explicitamente fora de escopo, e por quê:

- **`keydown`/`keyup` (eventos brutos de tecla).** Já coberto do lado oposto: `UiEvent::Type::Key` é o canal **injetado** pelo host (o host informa a glintfx sobre uma tecla pressionada). Um callback de keydown/keyup **observado** pelo RmlUi seria um segundo canal carregando informação sobreposta, a uma taxa de evento muito maior (toda tecla, não o valor assentado que `change` já reporta) sem nenhum batching — o default errado pro caso âncora do gancho de som (dispararia 3 vezes digitando "abc" onde `change` dispara uma vez quando o campo perde foco ou a cada edição confirmada, dependendo do controle). Nenhuma necessidade de consumidor identificada.
- **`mouseover`/`mouseout` (hover).** Nenhum caso âncora — os casos já nomeados pelo líder (click, scroll, focus/blur, ganchos de som) são todos ações discretas, comprometidas pelo usuário; hover é um fluxo contínuo de alta frequência (dispara a cada pixel cruzando a borda de um elemento) que precisaria da própria história de debounce antes de ser útil pra qualquer coisa, som incluso. Zero demanda real hoje.
- **`drag`/`dragstart`/`dragend`/`drop` (eventos nativos de drag-and-drop do RmlUi).** Nenhum widget nas telas shipadas ou planejadas do GusWorld (cockpit/HUD, menu, conforme o próprio escopo da spec v2) usa drag-and-drop. Só o payload (origem do drag, alvo do drop, dado transferido) já é um projeto de design do próprio porte, desproporcional a zero demanda confirmada.

Regra invocada: a mesma disciplina de pull incremental já adotada pra glintfx v2 ("backlog puxado por demanda real," decisão de 2026-07-01) — construir os quatro eventos com um consumidor nomeado e um caso âncora nomeado (ganchos de som), adiar o resto até um host real pedir.

### 12. Decisão pendente do líder

Apresentado pra `AskUserQuestion`, entregue pelo orquestrador. Os três mantêm a regra "nenhum tipo de terceiro no header público", a paridade `App`+`UiLayer`, e a disciplina obrigatória de copiar-antes-de-invocar de reentrância (§6) — isso não está em debate, só o **formato da superfície** está.

**Opção A — Callback único genérico.** `set_form_event_callback(fn(id, FormEventType type, value))` + 1 enum público novo. **Esforço: o menor** (1 método × 2 classes, 1 bloco de doc-comment/AVISO compartilhado). **Prós:** menor delta de header; casa naturalmente com um padrão de host de "tabela de som única indexada por type". **Contras:** quebra o idioma de nomenclatura `set_<verbo>_callback` usado por todo callback já shipado; `value` é morto pra `focus`/`blur` (precisa de documentação por-ramo de quando não tem sentido); nenhuma exaustividade em tempo de compilação no `switch` do host.

**Opção B — Quatro callbacks específicos (recomendada).** `set_change_callback`, `set_submit_callback`, `set_focus_callback`, `set_blur_callback`, cada um só-id exceto `change` (id+value). **Esforço: baixo-médio** (4 métodos × 2 classes = 8 métodos públicos novos, mas cada um é uma extensão mecânica do padrão `set_click_callback`/`set_scroll_callback` já shipado duas vezes — nenhum design novo, blocos de doc-comment adaptados parágrafo-a-parágrafo do AVISO já existente de `set_scroll_callback`). **Prós:** zero desvio de nomenclatura; um host registra só o que precisa, literalmente zero código tocando os eventos que não lhe interessam; cada payload é exatamente do tamanho que seu evento precisa. **Contras:** maior crescimento de superfície de header dos três; quatro blocos AVISO/LIFETIME pra manter sincronizados ao longo do tempo (manutenção mecânica, não risco de design).

**Opção C — Híbrido, par focus/blur colapsado.** `set_change_callback`, `set_submit_callback`, `set_focus_change_callback(fn(id, bool focused))`. **Esforço: baixo-médio**, entre A e B (3 métodos × 2 classes = 6 métodos públicos novos). **Prós:** corta a superfície de B em 2 métodos explorando a simetria genuína de focus/blur (o RmlUi sempre os empareja); ainda tipado, ainda opt-in. **Contras:** um desvio de nomenclatura (`focus_change` não é um nome literal de evento do RmlUi); o host faz branch num `bool` em vez de ganhar um callback dedicado por direção.

**Recomendação: Opção B.** É a escolha com menos risco de design novo — este repositório escolheu, três vezes seguidas (click, click_info, scroll), "verboso mas consistente com o idioma de nomenclatura já shipado" em vez de "compacto mas novo," e todo bloco AVISO/reentrância que esta feature precisa já existe quase palavra-por-palavra no doc-comment de `set_scroll_callback` pra adaptar. A Opção A é uma alternativa legítima se o líder pesar economia de linhas de header acima de consistência de nomenclatura; a Opção C é o meio-termo se o líder quiser limitar a contagem de métodos sem ir totalmente genérico.
