# EN: vptr:ElementDocument — RmlUi (pinned commit 2cd28864, fetched via
#     FetchContent, third-party, NOT glintfx code) reads through an
#     ElementDocument* whose ElementDocument-derived part has already finished
#     destructing, during its own teardown chain:
#       Element::SetOwnerDocument (Source/Core/Element.cpp:2128)
#         owner_document->GetContext()   <- vptr check fires here
#       called from Element::SetParent (Element.cpp:2190)
#       called from ~Element() (Element.cpp:108), itself invoked from
#       ~ElementDocument() while releasing that very document's own children
#       (Context::ReleaseUnloadedDocuments, Context.cpp:1567).
#     i.e. a child element's owner_document pointer aliases the document
#     object that is mid-self-destruction: ~ElementDocument() (derived part)
#     has already run, so at the point Element::SetOwnerDocument dereferences
#     it, the live dynamic type is only the base Element subobject, not
#     ElementDocument — a real, strict-UB, textbook "call through a pointer to
#     the derived type of an object whose derived part is already destroyed"
#     pattern. In practice GetContext() only touches base-Element state that
#     is still alive, so it is not observed to corrupt memory or misbehave —
#     but it is technically UB and correctly flagged.
#     100% reproducible: every embed test that loads then releases a document
#     (8/8 embed tests + all GLFW-backend tests that load a document) hits it
#     exactly once at teardown. Verified NOT reachable through any
#     glintfx-authored Element/ElementDocument subclass (grep confirms glintfx
#     never subclasses Rml::Element / Rml::ElementDocument) — this suppression
#     cannot mask a glintfx bug.
#     Reported upstream candidate / local fix candidate: TESTES.md TST-L1-MEM
#     QA report (this gate's first run, 2026-07-02). NOT patched in-tree:
#     RmlUi is a pinned FetchContent source, patching it here would silently
#     fork the pin without going through the normal
#     bump-the-GIT_TAG-and-review path. Re-verify (and consider raising
#     upstream / bumping the pin) whenever RMLUI GIT_TAG in CMakeLists.txt
#     changes.

#     FetchContent, terceiro, NÃO código glintfx) lê através de um
#     ElementDocument* cuja parte derivada ElementDocument já terminou de ser
#     destruída, durante a própria cadeia de teardown dele:
#       Element::SetOwnerDocument (Source/Core/Element.cpp:2128)
#         owner_document->GetContext()   <- o check de vptr dispara aqui
#       chamado por Element::SetParent (Element.cpp:2190)
#       chamado por ~Element() (Element.cpp:108), por sua vez invocado por
#       ~ElementDocument() ao liberar os próprios filhos daquele documento
#       (Context::ReleaseUnloadedDocuments, Context.cpp:1567).
#     ou seja, o ponteiro owner_document de um elemento-filho aponta para o
#     próprio objeto documento que está no meio da auto-destruição:
#     ~ElementDocument() (parte derivada) já rodou, então no ponto em que
#     Element::SetOwnerDocument o desreferencia, o tipo dinâmico vivo é só o
#     subobjeto base Element, não ElementDocument — um padrão real, de UB

#     não se observa corrupção de memória ou mau comportamento — mas é UB

# EN: vptr:*WidgetScroll.cpp* + vptr:*RemoveEventListener* (TWO entries, both required
#     together — see the matching-mechanism note at the end) — RmlUi (same pinned commit,
#     third-party, NOT glintfx code) hits an analogous teardown-order UB pattern inside its
#     OWN internal scrollbar widget, Rml::WidgetScroll, first exercised by scroll_sanity.cpp
#     (GLINTFX-SCROLL-1, v0.4.0) — it is the first scene/test in the whole suite whose
#     overflow-y:auto content actually overflows enough to instantiate a live scrollbar
#     (30 x 30px items in a 100px-tall container, scroll_scene.rcss).
#     ~WidgetScroll() (Source/Core/WidgetScroll.cpp:44-60) calls RemoveEventListener on up to
#     4 Element* it holds to its own generated child elements (the scrollbar "thumb" `bar`,
#     `track`, and the two `arrows[0]/arrows[1]`) -- by the time the destructor runs, those
#     children's own Element parts may already be mid-destruction in the same recursive
#     teardown cascade (Context::ReleaseUnloadedDocuments -> ~ElementDocument() ->
#     recursive ~Element(), Context.cpp:1567 -- the SAME cascade documented in the
#     vptr:ElementDocument entry above, just touching different internal Element*s). Same
#     root-cause category as that entry: "virtual call / member access through a pointer to
#     an object whose derived part is already destroyed", strict UB, not observed to corrupt
#     memory in practice (RemoveEventListener only touches base-Element listener-list state
#     that is still alive at that point).
#     TWO distinct diagnostic shapes were observed for this SAME root cause, depending on
#     whether the specific RemoveEventListener call got inlined into ~WidgetScroll() by the
#     compiler (a per-call-site, compile-time-fixed decision -- WHICH of the 4
#     bar/track/arrows[0]/arrows[1] pointers is corrupted enough to actually trip the check
#     is runtime-memory-layout-dependent, hence non-deterministic across process runs, which
#     of the two shapes appears first under halt_on_error=1 also varies run-to-run --
#     verified by both direct binary reruns AND `member call` vs `member access` diagnostic
#     wording in the two variants below):
#       (a) INLINED: reported directly at the call site, e.g.
#           "WidgetScroll.cpp:47:27: runtime error: member call on address ... which does
#           not point to an object of type 'Element' / note: object has invalid vptr" --
#           the enclosing function symbol IS ~WidgetScroll() itself (inlining absorbed
#           RemoveEventListener's body), so type-name suppression (vptr:Element) does NOT
#           match (verified empirically: `vptr:Element` alone, and even wildcarded
#           `vptr:*Element*`, left this diagnostic unsuppressed -- "invalid vptr" checks
#           with no recoverable dynamic type apparently do not match on the STATIC
#           TypeDescriptor name the way the vptr:ElementDocument entry above does for its
#           mismatched-but-valid-vptr case). File-scoped matching DOES work here:
#           `vptr:*WidgetScroll.cpp*`.
#       (b) NOT inlined: reported inside the callee body, e.g.
#           "Element.cpp:1254:2: runtime error: member access within address ... which does
#           not point to an object of type 'Element' / note: object has invalid vptr", stack
#           frame #0 = Rml::Element::RemoveEventListener(...), frame #1 =
#           Rml::WidgetScroll::~WidgetScroll() at WidgetScroll.cpp -- a plain
#           `vptr:*WidgetScroll.cpp*` file match does NOT catch this (the diagnostic's OWN
#           reported file is Element.cpp, a large, heavily-shared file across all of RmlUi,
#           too broad to suppress wholesale). Function-scoped matching DOES work here and
#           is precise (Element::RemoveEventListener is not called by any glintfx code --
#           grep confirms): `vptr:*RemoveEventListener*`.
#     Both patterns are needed TOGETHER because which shape appears is non-deterministic
#     (verified: 10/10 clean direct-binary reruns with both entries present combined with
#     the pre-existing LSAN_OPTIONS/ASAN_OPTIONS from this gate; single-pattern attempts
#     each left 1-2 residual failures across repeated runs).
#     Verified NOT reachable through any glintfx-authored Element/WidgetScroll/
#     ElementScroll subclass, and glintfx never calls RemoveEventListener itself (grep
#     confirms both) -- neither suppression can mask a glintfx bug. Newly discovered, not
#     newly introduced: this is a LATENT upstream teardown-order UB that was simply never
#     exercised before scroll_sanity.cpp, because no earlier test/demo scene authored an
#     overflow-y:auto (or overflow-x) container whose content actually overflows its box
#     enough for RmlUi to materialise a live WidgetScroll.
#     Reported upstream candidate / local fix candidate: GLINTFX-SCROLL-1 implementation
#     notes (v0.4.0, 2026-07-07). NOT patched in-tree — same pinned-FetchContent rationale
#     as the entry above; re-verify whenever RMLUI GIT_TAG changes.
